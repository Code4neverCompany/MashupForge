# Scheduled Posting — 401 Unauthorized on cron-fire

**Date:** 2026-04-30
**Triage owner:** Developer
**Severity:** P0 — no scheduled posts firing since ~08:44 UTC today

## Brief premise vs reality

The user/Hermes brief assumed the cron mechanism is **Upstash QStash** with
`UPSTASH_QSTASH_*` env vars. That is incorrect.

Actual mechanism:
- **GitHub Actions** runs `.github/workflows/cron-fire-scheduled-posts.yml`
  on a `*/5 * * * *` schedule, hitting `${APP_URL}/api/social/cron-fire`
  with `Authorization: Bearer ${CRON_SHARED_SECRET}`.
- **Upstash Redis** (NOT QStash) backs the queue (`@upstash/redis`,
  `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN`).
- Workflow IS registered, IS firing every 5 min, and ran 38 times.
  Recent runs at 06:41 UTC succeeded; 08:44 UTC and 09:04 UTC failed.

## Root cause

The cron-fire endpoint is returning **401 Unauthorized**, not 503. That is
diagnostic:

```ts
// app/api/social/cron-fire/route.ts
if (!expected) return 503;        // env unset on server
if (!safeEqual(presented, expected)) return 401;  // bearer mismatch
```

- `vercel env ls` (project `mashup-studio`, scope `4nevercompanys-projects`)
  shows `CRON_SHARED_SECRET` exists in Production, set 3 days ago.
- Probing `https://mashup-studio.vercel.app/api/social/cron-fire` with no
  bearer / dummy bearer / empty bearer all return `401 {"error":"Unauthorized"}`
  → the deployed lambda has a non-empty value.
- GitHub Actions secret `CRON_SHARED_SECRET` was last updated 2026-04-27.
- Last success: 2026-04-30T06:41 UTC. First failure: 2026-04-30T08:44 UTC
  (workflow_dispatch). Window of ~2h where Vercel-side value diverged.

**Conclusion:** the Vercel `CRON_SHARED_SECRET` was changed (rotated/edited
in the dashboard) between 06:41 and 08:44 UTC today, and the GitHub
Actions secret was not updated to match. Bearer comparison fails →
every 5-min run rejects with 401 → no posts fire.

## Adjacent finding (Issue 3 prerequisite)

`/api/social/cron-fire` calls `/api/social/post` with `credentials: {}`,
so scheduled posts depend entirely on `process.env.INSTAGRAM_*` on Vercel:
- `INSTAGRAM_ACCESS_TOKEN` — set in Prod+Preview, 6h ago
- `INSTAGRAM_ACCOUNT_ID` — set in Prod only, 5h ago

`vercel env pull` redacts encrypted/sensitive values to empty strings, so
the pull output cannot be used to verify the actual contents. Once 401 is
fixed, watch the cron summary for `Instagram credentials incomplete`
errors — that would indicate `INSTAGRAM_ACCOUNT_ID` is genuinely empty
on Vercel.

Manual posting works because the browser passes `credentials.instagram`
in the request body, and the route's `??` chain falls through to that.
Scheduled posting cannot fall through because the cron forwarder sends
`{}` for credentials.

## Recommended fix (NOT applied autonomously — touches shared secrets)

Pick one of:

1. **Re-sync from Vercel → GitHub.** Read current Vercel value via
   dashboard (Settings → Environment Variables → CRON_SHARED_SECRET → reveal),
   set the matching value via `gh secret set CRON_SHARED_SECRET -R Code4neverCompany/MashupForge`.
2. **Rotate fresh on both sides.** Generate `openssl rand -hex 32`,
   `vercel env add CRON_SHARED_SECRET production` (replacing existing) +
   redeploy + `gh secret set CRON_SHARED_SECRET`.

Either path requires a Vercel redeploy if the value is changed there.
After fix, smoke-test with:

```
gh workflow run cron-fire-scheduled-posts.yml -R Code4neverCompany/MashupForge
gh run watch -R Code4neverCompany/MashupForge
```

Expect a successful run with JSON summary `{"claimed":N,"posted":...}`.
