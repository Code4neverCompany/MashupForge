## Ref audit — 8 → 1 pairs (spec §4)

### §4.1 — Control signals → AbortControllers in `useState` ✅

```ts
const [runCtrl, setRunCtrl] = useState<AbortController | null>(null);
const [skipCtrl, setSkipCtrl] = useState<AbortController | null>(null);
```

`stopPipeline` and `skipCurrentIdea` reach the current controller via functional setState peek:

```ts
setRunCtrl(c => { c?.abort(); return c; });
```

`c?.abort()` on an already-aborted controller is idempotent (per spec). Safe under React StrictMode's double-invoke of updaters. ✅

### §4.2 — Daemon config reads inside the loop → `peek()` ✅

```ts
const peek = <T,>(setter) => {
  let out: T;
  setter(prev => { out = prev; return prev; });
  return out!;
};
```

Updater is called synchronously by React during `setState` scheduling — `out` is assigned before the next line. This is the documented pattern for reading latest committed state without a ref inside a `useCallback`. ✅

### §4.3 — External props → `latestPropsRef` (documented exception) ✅

ONE consolidated ref + ONE `useEffect` replacing 4 per-field pairs:

```ts
const latestPropsRef = useRef({ ideas, settings, images, savedImages });
useEffect(() => {
  latestPropsRef.current = { ideas, settings, images, savedImages };
}, [deps.ideas, deps.settings, deps.images, deps.savedImages]);
```

Named `latestPropsRef`, documented in code and spec §4.3. No other refs exist in the daemon. ✅

### §4.4 — Checkpoint snapshot refs → local variables ✅

`checkpointImagesRef` is gone. `perIdeaImageIds` in `useIdeaProcessor` is a `let string[]` local — created fresh per `processIdea` call, closed over by `checkpoint(step)`, passed into `writeCheckpointBase`. No ref. ✅

`checkpointSettingsRef` is gone. Settings are snapshotted into `snapshotSettings` at run-start inside `runOuterLoop`'s async callback — a plain local, no ref. ✅

### Composer refs (`userStoppedRef`, `autoStartFiredRef`) ✅

These two refs live in `usePipeline.ts` (the composer), not in the daemon. They are session guards, not state-sync refs:
- `userStoppedRef` — prevents auto-start from re-firing after user-initiated stop.
- `autoStartFiredRef` — makes the auto-start one-shot per mount.

Both are intentionally excluded from `useState` because toggling them must not trigger re-renders (which would re-run the auto-start effect). Correct use of refs. ✅

**Net: 8 sync pairs → 1 (`latestPropsRef`) + 2 AbortControllers in `useState` + 2 composer session guards. Rationale documented per spec.**

## Auto-start (BUG-PIPELINE-002 fix) ✅

The auto-start `useEffect` in `usePipeline` addresses the regression filed as `BUG-PIPELINE-002`:

```ts
useEffect(() => {
  if (autoStartFiredRef.current) return;
  if (!daemon.pipelineEnabled || !daemon.pipelineContinuous) return;
  if (daemon.pipelineRunning) return;
  if (userStoppedRef.current) return;
  if (daemon.pendingResume) return;
  const timer = setTimeout(() => {
    // Double-check inside timer — conditions may have changed
    autoStartFiredRef.current = true;
    void startPipeline();
  }, 2000);
  return () => clearTimeout(timer);
}, [daemon.pipelineEnabled, daemon.pipelineContinuous, daemon.pipelineRunning,
    daemon.pendingResume, startPipeline]);
```

The 2-second delay lets IDB hydration settle before the run begins. The `pendingResume` guard prevents a fresh auto-start from trampling a crash-recovery prompt. The `userStoppedRef` honors an explicit stop within the same session. The `autoStartFiredRef` one-shot prevents re-fires when `pipelineRunning` flips false at natural loop exit. All edge cases handled. ✅

## Observations (non-blocking)

### 1. `runOuterLoop` deps include state values used only for snapshotting

`pipelineDelay`, `pipelineContinuous`, `pipelineInterval`, `pipelineTargetDays` appear in `runOuterLoop`'s `useCallback` dep array. Inside the callback, they are used only in `snapshotSettings` (captured at run-start). The loop then reads live values via `peek()`.

Since the closure captures these values correctly (deps ensure the callback is always fresh), this works. But it means `runOuterLoop` — and therefore `startPipeline` — is recreated whenever any pipeline config changes. With `autoStartFiredRef` guarding the auto-start, this causes no correctness problem. If `startPipeline` identity stability ever matters to consumers, `snapshotSettings` could be built by calling `peek()` instead of closing over the state values, allowing those four deps to be dropped. Not worth changing now.

### 2. `desktopCreds` in `processIdea` deps

`useIdeaProcessor` includes `desktopCreds` (from `useDesktopConfig()`) in `processIdea`'s dep array. If `useDesktopConfig` returns a new object reference on every render (common for context values), `processIdea` and `startPipeline` would be recreated frequently. `autoStartFiredRef` prevents redundant pipeline starts, so correctness is unaffected. Worth profiling if users report unexpected re-renders on desktop. Out of V030-001 scope.

### 3. `autoGenerateIdeas` dep in `runOuterLoop`

`autoGenerateIdeas` has an empty `[]` dep array — it never changes. Its presence in `runOuterLoop`'s dep array is harmless but unnecessary. Lint noise only.
