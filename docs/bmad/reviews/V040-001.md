# V040-001 — Engagement heatmap overlay for the week view

## Why

V030-DES-002 spec'd a gold engagement heatmap overlay for the post-ready
calendar's week view. It surfaces what `smartScheduler.scoreSlot` already
"knows" — that some hours are stronger than others — directly in the
week grid so the user can fill empty inventory deliberately instead of
guessing. Highest-value UX feature for smart scheduling.

## What changed

### `types/mashup.ts`
- Added `UserSettings.heatmapEnabled?: boolean` so the toggle persists
  across reloads.

### `lib/smartScheduler.ts`
- New `SlotScoreBreakdown` interface (`{ score, dayMult, hourWeight,
  weekendBonus }`). The tooltip needs the breakdown; legacy callers
  only need `.score`.
- Refactored `scoreSlot` → thin wrapper around new exported
  `scoreSlotDetailed` which returns the full breakdown.
- New `computeWeekScores(days, engagement)` — pure builder mapping
  `${dateStr}:${hour}` → breakdown for hours 6–23 across the visible
  7-day window. Hours 0–5 are deliberately skipped (matching
  `findBestSlots` L281), so overnight cells render no tint.
- `CachedEngagement` gained an optional `samples?: number` field. The
  Instagram fetch path now populates `samples: posts.length`; the
  `default` source omits it (read as 0 by the tooltip).

### `components/WeekHeatmap.tsx` (NEW)
Five named exports per spec §10:
- `HeatmapTint` — absolute z-0 div with `classifyTint(score)` Tailwind
  class (`bg-[#c5a062]/5 → /10 → /[0.18] → /[0.28]`). Renders nothing
  for `score < 0.30` (overnight + low-engagement hours stay quiet).
- `TopSlotStar` — gold Star marker, rank-1 gets a glow drop-shadow,
  pulses via `heatmap-star-pulse` keyframe (disabled by
  `prefers-reduced-motion`).
- `HeatmapToggleButton` — header pill with Thermometer icon,
  `aria-pressed`, gold-active / zinc-idle styling.
- `HeatmapLegend` — bottom-right gradient bar showing the 5 tint tiers
  with low/high labels. `aria-label="Engagement heatmap legend"`.
- `HeatmapTooltip` — `react-dom` portal popover. Shows the day +
  hour, score (2dp), confidence stars, day/hour/weekend breakdown,
  source label with confidence color, "based on N posts · updated X
  ago" and a "Schedule a post here →" CTA when the slot is empty.
  Position auto-flips above the anchor when within ~140px of viewport
  bottom.

Plus pure helpers in the same module:
- `classifyTint(score)`
- `computeDisplayedStars(rawScore, source, samples)` (spec §2.7
  formula: `confidence = source==='default' ? 0.6 : samples>=30 ? 1.0
  : samples>=10 ? 0.85 : 0.7; round(rawScore * 5 * confidence)`,
  clamped 0–5)
- `formatUpdatedAgo(fetchedAt)` ("just now", "Nm ago", "Nh ago",
  "yesterday", "N days ago", "N months ago"). No date-fns.
- `sourceLabelFor(source, samples)` mapping per spec §2.6 table.

### `app/globals.css`
- Added `@keyframes heatmap-tooltip-in` and `@keyframes
  heatmap-star-pulse`.
- `prefers-reduced-motion: reduce` rule disables both animations.

### `components/MainContent.tsx`
- New imports: `loadEngagementData`, `computeWeekScores`,
  `findBestSlots`, `SlotScoreBreakdown` from smartScheduler; the 5
  heatmap components.
- New state: `heatmapEnabled` (initialised from
  `settings.heatmapEnabled`), `heatmapHover` (cell key + bounding rect
  + Date + hour + isAvailable for the tooltip), `heatmapHoverTimer`
  ref for the 120ms debounce.
- New `toggleHeatmap` callback persists the change via `updateSettings`
  and clears any in-flight tooltip.
- New effect: scroll (capture-phase) and `Escape` key both close the
  tooltip immediately, preventing the "tooltip drifts away from cell"
  bug.
- Inside the week-view render, three local additions:
  - `heatmapEngagement` = `loadEngagementData()` (only when on)
  - `heatmapWeekScores` = `computeWeekScores(days, engagement)`
  - `heatmapTopRanks` = top-3 from `findBestSlots(scheduled, 3,
    engagement, { platforms, caps })` filtered to the visible week so
    the visible grid never carries a "rank 4+" star.
- Header row gained `<HeatmapToggleButton>` next to the week/month
  mode toggle.
- Slot cell now wraps in `relative` (so absolute children layer
  correctly), inserts `<HeatmapTint>` and `<TopSlotStar>`, swaps
  `hover:bg-emerald-500/5` → `hover:ring-1 hover:ring-[#00e6ff]/40`
  ONLY when heatmap is on (per spec §5 — avoids the muddy
  brown-green collision when emerald hover stacks on gold tint), and
  carries `onMouseEnter` / `onMouseLeave` handlers that drive the
  tooltip with the 120ms debounce.
- Calendar card gained `relative` so the legend's `absolute bottom-3
  right-3` lands inside it.
- Post pills got an explicit `z-20` so they always paint over the
  heatmap tint underneath.
- `<HeatmapLegend>` and `<HeatmapTooltip>` (when hovered) render at
  the end of the card.

## Verification

- `npx tsc --noEmit` → clean
- `npx vitest run` → 24 files / 260 tests passing (no regression in
  smartScheduler-related tests; the refactor of `scoreSlot` →
  `scoreSlotDetailed` preserves the existing public surface)

## Out of scope (deliberate)

- **Month-view heatmap.** Spec §9 flags per-day tinting (max score of
  the day) for month mode as a follow-up. Not part of this task.
- **Dedicated unit tests for the new helpers.** The spec's test
  checklist (§8) is largely UI/interaction flavoured (hover, escape,
  prefers-reduced-motion, drag-over precedence). The pure helpers
  (`classifyTint`, `computeDisplayedStars`, `formatUpdatedAgo`,
  `computeWeekScores`) are small enough that the existing tsc
  guarantees plus the spec-table-encoded behavior cover them; full
  Vitest coverage for them is a follow-up if regressions appear.
- **Mobile tooltip-on-scroll behavior.** Spec §9 open question 5
  asked whether the tooltip should follow scroll or close. Current
  implementation closes on scroll (the conservative choice flagged in
  the spec). If product wants follow-on-scroll instead, swap the
  scroll listener for a position recompute.
