# V040-009 — Per-platform aspect crop preview on Post Ready cards

The Post Ready grid was rendering every preview as a hard-coded
`aspect-square` thumbnail. That was honest for Instagram-only posts
but actively misleading for everything else: a Pinterest-bound image
got framed in a square even though Pinterest crops to 2:3, and a
Twitter image showed as a square even though the in-feed crop is
16:9. Users were posting and then *seeing* the crop for the first time
in their actual feed.

This task makes the preview reflect the platforms the user has
selected on the card.

## What changed

### `lib/platform-aspect.ts` (NEW)

A pure module with the per-platform aspect map:

```ts
PLATFORM_ASPECT = {
  instagram: { className: 'aspect-square', ratio: '1:1', note: '...' },
  pinterest: { className: 'aspect-[2/3]', ratio: '2:3', note: '...' },
  twitter:   { className: 'aspect-video', ratio: '16:9', note: '...' },
  discord:   { className: 'aspect-square', ratio: '1:1', note: '...' },
};
```

Plus a `getAspectFor(platform)` helper that gracefully falls back to
`aspect-square` for null/unknown — so an unrecognized future channel
shows *something* rather than throwing.

The note field is wired into the chip's `title=` attribute as a
hover-tooltip explaining why the crop looks the way it does.

### `components/postready/AspectPreview.tsx` (NEW, ~115 lines)

Encapsulates the preview block. Props:
- `src`, `alt`, `onClick` — image basics
- `selectedPlatforms: PostPlatform[]` — drives the active aspect
- `overlay` — slot for the existing badge + checkbox children so the
  call-site keeps full control of those without forcing them into
  prop bag

Behavior:
- Active platform defaults to `selectedPlatforms[0]`
- A `useEffect` resyncs the active platform whenever the selection
  changes — if the user un-toggles the currently-previewed platform,
  the preview snaps to the next selected one (or back to a 1:1 default
  when none remain)
- Container's `aspect-*` class is the active platform's aspect
- A small `1:1` / `2:3` / `16:9` chip in the bottom-left makes the
  crop ratio explicit without a wall of text
- When the user has 2+ platforms selected, a tiny tab strip below the
  image lets them flip between aspects (one tap each). With 0 or 1
  platforms the strip hides — single-tab strips are noise, the chip
  already says it all.

### `components/MainContent.tsx`

Two changes:
- Import `AspectPreview`
- Replace the hard-coded preview block in the **single-image** Post
  Ready grid card (L4029-4068) with `<AspectPreview ... overlay={...}/>`.
  The badge + grouping checkbox slot in via the `overlay` prop, so all
  existing UI lives in the same absolute layer over the image.

## Out of scope (deliberate)

- **Carousel cards.** A carousel already shows N images in a horizontal
  strip with their natural `h-36 w-36` thumbs. Imposing a per-platform
  aspect on each thumb would either force letterboxing (ugly) or pick
  one platform's aspect to apply to all (misleading). Carousels also
  fan out to a fixed Instagram set in practice. Out of scope.
- **Calendar view thumbnails.** The calendar mode already uses tiny
  fixed-aspect cells for layout density; the aspect of the thumbnail
  isn't the message there. Out of scope.
- **Pinterest 1000x1500 portrait alternate**, **IG 4:5 portrait
  alternate**, **Twitter 1:1 alternate.** Each platform has secondary
  ratios users *can* opt into via custom crops. The default preview
  reflects the platform's *primary* feed crop — what the user sees if
  they upload without cropping. Picking secondary aspects belongs in a
  follow-up "custom crop tool" task, not the preview itself.
- **Actually cropping the upload to the previewed aspect.** This task
  is preview-only — the actual upload still ships the original image
  and lets each platform's API do its native crop. Auto-cropping at
  upload is a separate behavioral change that needs its own design
  pass (do we save N cropped variants? Does the user approve the
  crop?). Out of scope.
- **Animated transition between aspects.** The aspect class change
  causes an instant resize when the user clicks a platform tab. A
  `transition-[aspect-ratio]` is tempting but Tailwind's aspect
  utilities don't transition smoothly (the container snaps height
  during reflow). Skipped to avoid jank.
