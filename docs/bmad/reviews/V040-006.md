# V040-006 — 10-second undo for bulk approvals and rejections

## Why

The approval queue's three bulk buttons — `Approve Selected`,
`Reject Selected`, `Approve All (N)` — are one-click mutations over
an arbitrary number of posts. A mis-click on `Approve All` when the
queue has 12 pending items is a 12-item mistake with no recovery
path short of going into IndexedDB by hand.

The pattern for this (Gmail, Slack, VS Code) is a time-bound undo:
show a transient toast with an Undo button for N seconds, dismiss
automatically after. Recovers the mistake; costs nothing if
unused.

## What changed

### `components/UndoToast.tsx` (NEW)

A fixed-positioned (`bottom-6 right-6`) toast with:
- A message the caller supplies ("N posts approved" / "N posts
  rejected").
- An **Undo** button with a live seconds-left counter.
- An `✕` dismiss button that immediately clears the toast without
  triggering undo (accepts the action).
- A bottom progress bar that drains linearly across the duration —
  gives the user a peripheral sense of how much time is left.

Owns its own countdown via `setInterval` (100ms tick, so the
progress bar looks smooth). When the deadline passes the toast
calls `onDismiss`. The parent controls *what* the message says and
*what* Undo does; the toast is pure chrome.

A11y: `role="status"` + `aria-live="polite"` so screen readers
announce the action + undo window. The progress bar is
`aria-hidden` since the Undo button label already carries the
countdown (`Undo 9s`).

### `components/PipelinePanel.tsx`

Three threads added, all in the component that already hosts the
approval queue and holds `settings.scheduledPosts` + `updateSettings`:

1. **`undoState`** — `{ kind: 'approve' | 'reject'; snapshots: ScheduledPost[] } | null`.
   Captured from `settings.scheduledPosts` *before* dispatching the
   bulk mutation, so we have the authoritative pre-action record.
   A new action replaces the previous undo state (standard "one
   toast at a time").

2. **`undoBulkAction`** — merges the snapshots back via
   `updateSettings`:
   - Approve-undo: the posts still exist in the list (flipped to
     `status: 'scheduled'`); replacing them with their snapshot
     restores `status: 'pending_approval'`.
   - Reject-undo: the posts were deleted; append the snapshots.
   - Guard: if a post's current status is already `'posted'` or
     `'failed'` the merge skips it. Within a 10s window the
     auto-poster *can* fire for a single approved post — we don't
     want undo to lie about reversing a live publish.

3. **`onBulkApproveWithAutoContinue` + `onBulkRejectWithUndo`** —
   the handlers passed down to `ApprovalQueue`. Each captures the
   snapshot, dispatches the bulk mutation, and sets `undoState`.
   The approve handler preserves the existing V030-005
   `shouldAutoContinuePipeline` behavior — undo just reverts the
   approval, it does not stop the pipeline if it already started
   as a side effect. (Flagged below.)

   Filter used for approve snapshots: `p.status === 'pending_approval'`
   — the bulk mutation has the same internal guard, so snapshotting
   the same subset keeps the revert symmetric with what was
   actually flipped.

### Wire-up

- Switched `onBulkReject={bulkRejectScheduledPosts}` →
  `onBulkReject={onBulkRejectWithUndo}` on the `<ApprovalQueue>`
  mount (~L656 of PipelinePanel).
- Rendered `<UndoToast>` as a conditional sibling of the queue. The
  `key` is derived from the action kind + snapshot IDs, so a new
  bulk action unmounts + remounts the toast (resets the countdown
  cleanly).

## Out of scope (deliberate)

- **Undo for single approve/reject** (`approveScheduledPost` /
  `rejectScheduledPost` — the per-card `✓` / `✕` buttons). Task says
  *bulk*. Single-item actions are one click with full visual
  context; the 10s banner noise on every single card click would
  outweigh the safety.
- **Undoing `startPipeline()` side effect on approve.** If
  `shouldAutoContinuePipeline` fires, the daemon wakes up. Undo
  reverses the approval (posts flip back to pending_approval) but
  does not stop a mid-flight daemon cycle. That's correct behavior
  — the pipeline is a separate state machine and stopping it would
  be surprising. Users can hit the existing `Stop` button if they
  want the daemon down too.
- **Persisting undo state across page reload.** Intentional — undo
  is a 10s UX affordance, not durable journaling. If the user
  reloads, the window is over.
- **Toast queueing.** Only one toast is shown; a new bulk action
  replaces the previous (via the React `key` trick). Standard
  pattern — queuing toasts buries the latest action and is almost
  never what users want here.

## Known caveats flagged

- **Auto-poster race**: Between `bulkApprove` (flips status to
  `scheduled`) and the 10s undo window, the auto-poster can pick up
  a post whose `date`/`time` is in the past or imminent. The merge
  guard (skip `posted` / `failed`) means undo will correctly leave
  already-published posts alone — but the user might see a partial
  undo ("I clicked Approve All 12 posts, then Undo, but 2 actually
  went out"). Honest and recoverable; still worth watching. If this
  becomes a real problem the fix is to give the auto-poster a
  minimum-age guard on `scheduled` status (e.g., wait 15s before
  considering a post eligible to publish).
