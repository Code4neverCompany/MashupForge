# V050-005 — Inline caption editing in approval queue

**Status:** done
**Classification:** routine
**Why:** Designer + developer consensus — caption is the most-edited
field at approval time, and the current UI forces a round-trip
through Captioning Studio for what should be a one-line tweak.

## What shipped

Tap-to-edit caption directly inside the approval queue, for both
single-post cards and grouped carousel cards. Click the caption,
the static `<p>` swaps to a focused textarea pre-populated with the
current value. Saving rewrites the caption on every targeted
ScheduledPost (and, conditionally, on the matching CarouselGroup) in
a single `updateSettings` pass.

### Keyboard semantics

- **Enter** — save and exit edit mode
- **Shift+Enter** — insert a newline (multi-line captions stay possible)
- **Escape** — cancel without saving (restores the prior value)
- **Blur** — save (same path as Enter)

The Escape path uses a `cancelledRef` flag so the synthetic blur
that follows `textarea.blur()` is treated as a cancel, not as a
second save.

## The carousel-sync rule

Carousel siblings share a caption visually, but the data model has
each post own its own `caption` string AND the `CarouselGroup` itself
optionally owns a `caption` (used by the auto-poster for true
multi-image posts). The naive write would push to all sibling posts
*and* the group every time. That's surprising in two cases:

1. The user edits one of N carousel posts independently — they
   probably did NOT mean to rewrite the group caption.
2. The future Captioning Studio path that lets you intentionally
   diverge a single carousel slide.

The chosen rule, implemented in `applyCaptionEdit`:

> Patch `CarouselGroup.caption` only when **every** post belonging to
> that group is in the edited set.

The `CarouselApprovalCard` always passes the full sibling `postIds`
array on save (because it's a single shared field in the UI), so the
group caption updates whenever the user edits *from the card*.
Single-post cards pass only `[post.id]`, so groups never get touched
from that path.

## Why a separate `lib/caption-edit.ts`

The repo has no jsdom / RTL infrastructure (Vitest is configured
`environment: 'node'` per V050-002 review). Putting the transformation
inline in `MashupContext` would have made it un-unit-testable. Pulling
it into `lib/` lets the test suite cover the rules without dragging
in a React render harness — same shape as the
`lib/pipeline-finalize` extraction that V040 used.

## Out of scope (deliberately not done)

- **Hashtag editing.** Same surface, same pattern, different field.
  Belongs to a follow-up ticket; unbundled because hashtags need a
  pill UI not a textarea.
- **Optimistic UI conflict resolution** for the rare case where a
  pipeline tick rewrites the caption while the user is editing.
  Current behavior: parent change syncs into the draft on next
  render only when not editing — the in-flight edit wins on save.
- **Per-slide caption divergence inside a carousel.** The card
  treats the carousel caption as one field by design (Designer
  brief). If we ever want per-slide divergence at approval time,
  it's a separate UX call.
- **Captioning Studio retirement.** This eliminates the most common
  approval-time round-trip but does not (yet) make Captioning Studio
  redundant — bulk regeneration, hashtag editing, and AI-assisted
  rewrites still live there.
