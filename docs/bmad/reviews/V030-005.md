### AC-3: pipeline auto-continues after bulk approval

New `lib/approval-continue.ts` with `shouldAutoContinuePipeline(input)`.
The rule:

1. pipeline must currently be idle (never fight a running daemon),
2. the action was a bulk action AND at least one post was approved
   (single-click approvals stay passive — the user is clearly reviewing
   one at a time), AND
3. there's still work to do — either the week isn't filled yet OR
   pending ideas remain in the queue.

Wired into `components/PipelinePanel.tsx` via a
`onBulkApproveWithAutoContinue` wrapper around
`bulkApproveScheduledPosts` that reads `pipelineRunning`,
`weekFillStatus.filled`, and the unprocessed-ideas count from context,
then calls `startPipeline()` iff the rule fires. Single-item `onApprove`
remains the raw context function — no implicit run starts.

7 tests cover the decision truth table, including the "1-item bulk"
edge case (Select All on a single-post queue still counts as a
conscious batch action).

### AC-4: visual feedback

`ApprovalQueue.tsx` now shows a transient banner after any approve /
reject action (bulk or single):

- Emerald "N posts moved to schedule" for approvals
- Red "N posts removed from queue" for rejections
- `role="status"` + `aria-live="polite"` for screen readers
- 3-second auto-dismiss via `setTimeout` with cleanup on unmount and on
  re-trigger (prevents overlapping flashes)

Edge case handled: when the final approval empties the queue, the
component previously early-returned null — so the user saw the cards
disappear with no confirmation. Now it renders a banner-only container
for the 3-second window before fully unmounting.

## Scope calls I made

- **No card-exit animation.** The cards already disappear through the
  `filter` — approved posts drop out because they no longer match
  `status === 'pending_approval'`. Adding framer-motion exit animations
  would be nice polish but costs scope for no behavioral gain; the
  banner already confirms the transition.
- **Single-click approvals don't auto-start the pipeline.** The task
  context says "approve batch → pipeline fills schedule automatically"
  and contrasts it with "Not approve → post one → approve again" —
  individual approvals are meant to be manual. The `isBulk` gate in the
  rule keeps that distinction explicit.
- **Reject doesn't trigger continuation.** Same reasoning — rejecting
  content is a negative signal, not a "keep going" signal. Banner still
  fires for feedback.
