## Outcome

### `setPipelineRunning(false)` guaranteed — already done in V030-001

V030-001 rebuilt `runOuterLoop` to delegate its run-state management to
`withPipelineRunning` (`lib/pipeline-runner.ts:9-19`), which has an
explicit `try { await fn(); } finally { setRunning(false); }`. Any
uncaught throw inside the outer loop now lands in that `finally` and
the UI un-pins regardless. AC covered.

### Per-idea 10-min hard timeout (new)

Added to `hooks/usePipelineDaemon.ts`:
- Constant `PER_IDEA_TIMEOUT_MS = 10 * 60 * 1000`.
- Class `IdeaTimeoutError` (typed sentinel, not a magic string).
- Inside the idea loop, `processIdea` is wrapped in `Promise.race` against a
  `setTimeout`-backed rejection. On fire, the skip signal is aborted so
  any in-flight step that checks `isSkipRequested()` unwinds cleanly,
  then the race rejects with `IdeaTimeoutError`.
- Outer catch recognizes `IdeaTimeoutError`, logs `pipeline-timeout` with
  the configured cap, resets the idea status to `'idea'` (next cycle can
  retry), and continues to the next idea.
- `finally` clears the timer so successful completions don't leak handles.

## Notes

Promise.race leaves the losing promise running. The skip-signal abort
inside the timeout handler is the mechanism that tells the in-flight
`processIdea` to wind down — it checks `isSkipRequested()` at every
major step boundary. The orphan promise resolves/rejects to nothing
afterward; no handles leak because the timer is cleared in `finally`.
