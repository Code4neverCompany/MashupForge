# BUG-002 — Auto-updater spawns a second instance, port 19782 conflict

## TL;DR

Frontend awaited `update.downloadAndInstall(...)` and trusted that NSIS would
kill the running app. NSIS doesn't kill the parent — its `/R` flag only
**relaunches**. So after an update on Windows the old `app.exe` was still
running, still holding `DESKTOP_PORT = 19782`, and the freshly installed app
booted onto an ephemeral port. Two `node.exe` sidecars, port conflict, and the
IndexedDB origin pin (STORY-121) silently broke because the origin URL
changed between runs.

**Fix:** add `tauri-plugin-process`, expose its JS `relaunch()` API, and call
`await relaunch()` in the renderer immediately after `downloadAndInstall`
resolves. `relaunch()` triggers a clean Tauri exit → fires
`WindowEvent::CloseRequested` (lib.rs:621) → the existing handler kills the
sidecar `Child` → port 19782 frees → NSIS proceeds → `/R` spawns the new
binary, which binds 19782 cleanly.

## Root cause

`components/UpdateChecker.tsx:117`:

```ts
await update.downloadAndInstall((event) => { /* progress */ });
// On Windows / NSIS the installer closes the app and relaunches itself.
// If we're still alive past this point, fall back to "post-update" hint.
```

That comment is wrong. NSIS installer behavior:

| Flag | Behavior |
|---|---|
| `/S` | Silent install (no UI). |
| `/P` | Passive install (progress UI, no prompts). |
| `/R` | After install, **launch** the application. Does NOT kill anything. |

Tauri's NSIS template uses `/P /R` for the "passive + restart" updater path.
There is no `/K` or "kill parent" option — NSIS isn't a process supervisor.
The installer simply waits for file locks (or fails when files are in use,
depending on the `installMode`), writes new files, and spawns the binary.

Without an explicit exit from the renderer, the old process kept running.
Symptoms reported:

- Two `app.exe` + two `node.exe` after auto-update.
- New instance falls through `resolve_port` (lib.rs:38) into the ephemeral
  fallback because 19782 is taken — so the IndexedDB origin becomes
  `http://127.0.0.1:<random>` rather than `:19782`, severing the persisted
  store from STORY-121.
- "Settings don't persist after update" — same root cause: storage moved.

## Fix — five files

### 1. `src-tauri/Cargo.toml`

Added the Rust crate:

```toml
tauri-plugin-process = "2"
```

Compiles to `tauri-plugin-process v2.3.1`. No transitive surprises (it's a
first-party Tauri plugin).

### 2. `src-tauri/src/lib.rs`

Registered the plugin alongside the other updater-adjacent plugins, with an
explanatory comment so future-me doesn't re-delete it during cleanup:

```rust
.plugin(tauri_plugin_updater::Builder::new().build())
// BUG-002: tauri_plugin_process exposes the JS `relaunch()` API.
// Frontend calls it after `update.downloadAndInstall(...)` so the
// OLD app exits cleanly — firing WindowEvent::CloseRequested
// below, which kills the Node sidecar and frees DESKTOP_PORT
// (19782). Without this the NSIS installer's `/R` flag spawns
// the new app while the old one still holds the stable port,
// forcing the new instance onto an ephemeral port and breaking
// the IndexedDB origin pin (STORY-121). See lib.rs:622+ for
// the sidecar-kill handler this triggers.
.plugin(tauri_plugin_process::init())
```

The `WindowEvent::CloseRequested` handler at lib.rs:621 already does the
right thing — it locks the `SidecarState`, takes the `Child`, calls `.kill()`
and `.wait()`. We're just making sure that handler actually runs during an
update, instead of being skipped because nothing told the window to close.

### 3. `src-tauri/capabilities/default.json`

Granted the renderer permission to call `relaunch`:

```json
"process:default",
"process:allow-restart"
```

Without `process:allow-restart` the JS call would throw "Permission denied"
at runtime — Tauri v2 capabilities are explicit per command.

### 4. `package.json`

Added the matching JS package so the dynamic import resolves:

```json
"@tauri-apps/plugin-process": "^2"
```

Dynamic-import (not static) was the established pattern in this file
(`@tauri-apps/api/app` line 49, `@tauri-apps/plugin-updater` line 85). Same
reason: web build (when `isDesktop !== true`) shouldn't drag the Tauri
plugin shim into the bundle.

### 5. `components/UpdateChecker.tsx`

Replaced the wrong "NSIS will close us" comment with the actual fix:

```ts
await update.downloadAndInstall((event) => { /* progress */ });
// BUG-002: NSIS only RELAUNCHES via `/R`, it does NOT kill the parent.
// Without an explicit exit the old instance keeps holding sidecar port
// 19782 (DESKTOP_PORT in src-tauri/src/lib.rs) and the new instance
// installed by NSIS falls back to an ephemeral port — which breaks
// the IndexedDB origin pin (STORY-121) and orphans settings.
// `relaunch()` from tauri-plugin-process triggers a clean exit, fires
// WindowEvent::CloseRequested in lib.rs, the sidecar Child is killed,
// port 19782 frees, and Tauri spawns the freshly installed binary.
const processMod = await import('@tauri-apps/plugin-process');
await processMod.relaunch();
```

`relaunch()` itself never returns under normal conditions — it tells Tauri's
runtime to spawn the new binary and exit the current one. The `await` is
defensive: if the call itself throws (e.g. permissions wrong, plugin
missing) the existing `catch` block surfaces it as `download-error` so the
UI doesn't hang in `downloading` state forever.

## What did NOT change (and why)

- **The sidecar-kill logic** (`on_window_event(CloseRequested)`, lib.rs:621):
  already correct, no edit. The bug was that we never closed the window.
- **`resolve_port`** (lib.rs:38): the ephemeral fallback is correct
  defensive behavior — we don't want the app to refuse to start when 19782
  is genuinely taken by something else. The fix is to free 19782 *before*
  the new instance starts, which is what `relaunch()` does.
- **NSIS installer config** (tauri.conf.json `bundle.windows.nsis`): no
  edits. Trying to teach NSIS to kill the parent (`!include nsProcess.nsh`,
  `${nsProcess::FindProcess}` etc.) is fragile across antivirus and adds an
  installer dependency. Letting the renderer trigger its own clean exit is
  cleaner and uses logic that already exists in lib.rs.

## How to roll back if needed

The only behavioral change is "renderer calls `relaunch()` after install."
To revert without losing the plugin wiring (in case we want to use it
elsewhere later), revert just `components/UpdateChecker.tsx`. The plugin
costs ~200 KB compiled into the binary; cheap to keep around.

Full revert: `git revert <commit>` undoes all five files atomically.

## Follow-up notes

- **Linux/macOS bundles**: `relaunch()` is cross-platform — works on
  AppImage, dmg, etc. — so this fix is not Windows-specific even though
  the bug only surfaced there.
- **STORY-121 (port pinning)**: still load-bearing. Without the stable
  port, even with `relaunch()` working, the IndexedDB origin would still
  drift any time the app restarted into a contested port.
- **Future**: if we ever migrate off NSIS (e.g. MSIX), `relaunch()` is
  still the right call — installers shouldn't be killing app processes.
