# V040-010 ‚Äî Empty Gallery state with Run Pipeline CTA

The Gallery's empty state was a dead end. It read:

> Save your favorite mashups from the Studio to build your collection.

That's only useful for users who already know how the app works. A
first-time user landing on the Gallery (which is the persistent tab
between sessions for some users) sees this and bounces ‚Äî there's no
button, no path forward, no hint that the pipeline can fill the
gallery automatically.

This task replaces the static message with a CTA-driven empty state
that adapts to where the user is in onboarding.

## What changed

### `components/EmptyGalleryState.tsx` (NEW)

A small standalone component that picks one of three pitches based on
two props (`firstRun` + `ideaCount`):

| State | Trigger | Pitch + CTAs |
|---|---|---|
| **first-run** | `firstRun=true` AND `ideaCount=0` | "Welcome to MashupForge" + two-step CTA: **Add Ideas** (primary) ‚Üí "then" ‚Üí **Run Pipeline** (secondary). Reflects the actual dependency ‚Äî the pipeline can't do anything without ideas to consume. |
| **has-ideas** | `ideaCount > 0` | Acknowledges the pending ideas in the copy ("You have 3 pending ideas ready for the pipeline...") + **Run Pipeline** (primary) + **Manage Ideas** (secondary) |
| **default-empty** | All other "empty gallery" cases | Friendlier copy + **Run Pipeline** + **Open Studio** as escape hatches |

The `firstRun` heuristic is the conjunction of:
- `savedImages.length === 0`
- `images.length === 0` (no in-flight Studio output either)
- `ideas.length === 0` (no pipeline ideas queued)
- `(settings.scheduledPosts ?? []).length === 0` (no posts ever)

That's intentionally strict ‚Äî once the user has *any* signal of
prior activity, we drop the welcome framing because they've clearly
been here before.

### `components/MainContent.tsx`

Two edits:
- Import `EmptyGalleryState`
- Branch the existing combined `view === 'studio' || view === 'gallery'`
  empty block: gallery now renders `<EmptyGalleryState ... />`; studio
  keeps its existing "Click Generate Mashup" message because that tab's
  primary action is already on screen (the Generate button at the top
  of the Studio).

The Studio empty state is left alone ‚Äî it's already self-explanatory
because the Generate Mashup button is the dominant element on that tab.
Only the Gallery needed the rescue.

## Out of scope (deliberate)

- **Empty state for Captioning / Post Ready / Pipeline tabs.** Those
  already have purpose-built empty states that link out (Post Ready ‚Üí
  Gallery / Captioning Studio buttons; Captioning Studio has a similar
  card; Pipeline has its own start button). No regression.
- **First-run modal / coachmarks / tour.** A guided tour is a much
  bigger UX commitment with its own lifecycle (dismiss state, "show
  again" toggle, screen size handling). The acceptance criterion is
  scoped to a CTA ‚Äî coachmarks would be over-build.
- **Animation between empty-state transitions.** Each state mounts /
  unmounts via the existing `motion.div` opacity+scale entry. Adding
  a cross-fade between the three sub-states would require pulling
  AnimatePresence in around the inner branches and isn't visible to
  users (you only see one state per gallery visit).
- **Settings prompt.** A first-run user is also probably missing API
  keys, but the existing platform health dot in the sidebar (V040-007)
  already handles that signal ‚Äî no need to duplicate the prompt here.
