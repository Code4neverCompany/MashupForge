# V040-004 ‚Äî Compact pipeline pill in the header band

## Why

The pipeline daemon runs continuously in the background, but nothing
in the rest of the app surfaces *whether it's alive right now*. If
you're on the Gallery, Captioning, or Post-ready tab, the only way to
check "is the daemon armed? is it sleeping? how many ideas are
queued?" is to click over to the Pipeline tab and read the side
panel. That's friction for the one piece of state the user wants
visible at all times.

Acceptance criteria: status strip in header, visible from every tab,
showing running/idle, next-idea timer, queue count.

## What changed

### `components/PipelineStatusStrip.tsx` (NEW)

A small presentational pill that reads pipeline state from the
existing `useMashup()` context and renders:

- **Status dot** ‚Äî emerald-400 pulsing when `pipelineRunning`,
  amber when `pipelineEnabled && !running` (armed), zinc when idle.
- **Label** ‚Äî `Running` / `Armed` / `Idle`.
- **Queue count** ‚Äî `Q <n>`, cyan when non-zero.
- **Contextual third slot** ‚Äî whichever of these applies:
  - Live countdown `Next in Xm Ys` while the daemon is mid-sleep in
    continuous mode (derived from `pipelineProgress.currentStep`;
    see note below).
  - Current idea title (truncated at 24 chars) while actively
    processing.
  - `Ready` when armed+continuous but not yet running.

Clicking the pill calls `setView('pipeline')`, navigating to the
pipeline tab. `hidden lg:flex` ‚Äî shown at desktop widths only, since
the mobile bottom-nav already exposes the Pipeline tab with its own
icon+label.

**Countdown source (minor implementation note):**
`usePipelineDaemon` does not expose a `pipelineNextRunAt` timestamp
‚Äî its only externally visible signal during sleep is
`pipelineProgress.currentStep` being set to `"Next cycle in N min"`
(L647 in the hook, right before the sleep-slice loop). Rather than
thread a new `nextRunAt` field through the hook ‚Üí context ‚Üí types
stack for a routine UI add, the strip parses that string with
`/Next cycle in (\d+) min/`, locks in a local deadline the first
time the message appears, and runs a 1s `setInterval` to tick the
displayed countdown. When the progress flips to anything else, the
deadline clears. If the daemon ever rewords that string, the
countdown silently falls back to hiding ‚Äî no error, no crash.

### `components/MainContent.tsx`

Two edits:
- Import `PipelineStatusStrip`.
- Mount `<PipelineStatusStrip setView={setView} />` inside the
  existing header's right-hand cluster (L1684, after the tab-row
  scroll-affordance gradient, before the logout button). The header
  already renders unconditionally outside the view switcher, so the
  strip is automatically visible on every tab ‚Äî no per-view
  plumbing needed.

## Out of scope (deliberate)

- **Exposing a dedicated `pipelineNextRunAt` field on the daemon.**
  Would require touching `hooks/usePipelineDaemon.ts` +
  `components/MashupContext.tsx` + `types/mashup.ts`, crossing the
  "routine" classification boundary (3+ files, schema change to
  `MashupContextType`). The string-parse approach is deliberately
  narrow: two files touched, zero API surface change.
- **Mobile visibility of the strip.** The mobile bottom-nav already
  has a Pipeline tab with its own icon; stacking another pipeline
  affordance in the tight mobile header is noise. `hidden lg:flex`
  keeps it desktop-only; if mobile visibility is later wanted, the
  strip is a self-contained component and can be placed anywhere.
- **Error badge.** When `pipelineRunning` is false but the last run
  ended in error, there's no red signal. The Pipeline tab's log
  surfaces errors directly; adding a cross-tab error dot would mean
  threading the last-run outcome through context. Out of scope for
  "status strip" ‚Äî open for a follow-up if Hermes wants it.
