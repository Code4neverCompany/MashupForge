# V050-003 — Typed error recovery for external APIs (Phase 1)

**Status:** done (Phase 1 of 2)
**Date:** 2026-04-19
**Branch:** main
**Commits:** (pending)

## Why

Silent pipeline death is the worst failure mode in the studio.
Leonardo 429s, pi.dev timeouts, and social-post failures all currently
bubble up as `Error`s with a stringly-typed message — call sites
either swallow them with a generic toast ("Animation failed") or
re-derive intent ("does this message contain '429'?") from the error
text. There's no shared notion of "is this retryable" or "did we
already retry it" beyond what each call site reinvents.

Phase 1 introduces the **typed primitives** that the rest of the codebase
will migrate onto: a `Result<T, ApiError>` envelope, a typed `ApiError`
discriminated union, per-API retry budgets that match the brief
(Leonardo 3, pi.dev 2, social 1), and a `Result`-returning
`fetchApi()` that honors the budget. Phase 2 sweeps the actual call
sites onto these primitives and wires per-card error state +
actionable toasts; that sweep is documented below as a separate
ticket because doing both in one pass would touch 21 files
simultaneously without an automated way to verify behavior.

## What changed (Phase 1)

### `lib/result.ts` (new)

- `Result<T, E> = { ok: true; value: T } | { ok: false; error: E }`
- `ok(value)`, `err(error)` constructors.
- `mapResult(r, f)` — transform success branch.
- `unwrap(r)` — throw on err; use sparingly.
- `tryAsync(fn)` — bridge legacy throwing functions onto Result.

Pure functional, zero deps, fully tested (10 cases).

### `lib/api-error.ts` (new)

- `ApiSource = 'leonardo' | 'pi' | 'social' | 'trending'` —
  the four external surfaces the studio talks to.
- `ApiErrorCode = 'http' | 'network' | 'timeout' | 'parse' |
  'rate_limit' | 'auth' | 'budget_exhausted' | 'unknown'`.
- `ApiError { source, code, retryable, status?, message, cause? }`.
- `RETRY_BUDGET: { leonardo: 3, pi: 2, social: 1, trending: 2 }` —
  matches the brief verbatim. Trending is 2 because it's a search-style
  call, not posting.
- `classifyHttpStatus(status)` — canonical "is this retryable, what
  bucket does it fall in" function. Single source of truth so
  `fetchApi` and any future ad-hoc classifier can't disagree.
- Builders: `httpApiError`, `networkApiError`, `budgetExhaustedError`.
- `toastMessageForApiError(error)` — actionable user-facing message
  per error code (auth → "open Settings", rate_limit → "wait a
  moment", etc.). Phase 2 wires this into `showToast` at the call
  sites.

Fully tested (12 cases).

### `lib/fetchWithRetry.ts` (extended, not breaking)

- Existing `fetchWithRetry()` unchanged — every current caller keeps
  working. The throwing behavior is preserved by design because
  fetchWithRetry's contract was "give me a Response or throw" and
  ~21 files depend on that.
- New `fetchApi(source, input, init?, opts?)` — Result-returning
  variant that:
  - Honors `RETRY_BUDGET[source]` by default.
  - Retries on network errors AND retryable HTTP statuses (5xx, 429).
  - **Never** retries 401/403 (auth) — returns the Response in the
    `ok` branch so the caller can read the body and decide.
  - Returns `err(budgetExhaustedError(...))` when retries are
    exhausted, with the last underlying ApiError attached as `cause`.
  - Caller can override `attempts` if they need a tighter budget.

### Tests

- `tests/lib/result.test.ts` — 10 cases covering ok/err narrowing,
  mapResult, unwrap (Error vs string error), tryAsync.
- `tests/lib/api-error.test.ts` — 12 cases covering classifyHttpStatus
  for the canonical codes (429, 401/403, 5xx, 4xx, 2xx), error
  builders, RETRY_BUDGET spec, and toast message rendering.
- `tests/lib/fetchWithRetry.test.ts` — +8 cases for `fetchApi`
  covering the per-API budget table (leonardo 3, pi 2, social 1),
  no-retry-on-auth, retry-on-rate-limit, network-error retry,
  attempts override.

Total: 359/359 tests pass (was 329, +30 new). tsc clean. New files
lint-clean.

## Why this is delivered as Phase 1 + Phase 2 (transparency)

A full sweep — replacing every `try { await fetch(...) } catch { ... }`
across 21 files with `fetchApi` + Result branching, plus wiring
typed toasts into MainContent + Sidebar + UpdateChecker + the
pipeline daemon — is a 500+ LOC change spread over the codebase. We
have:

- **No jsdom/RTL component tests** (same blocker noted in V050-002),
  so behavioral changes to UI surfaces can't be auto-verified.
- **Non-idempotent calls** (`/api/social/post`, `/api/leonardo-video`)
  where a misclassified retryable error could double-post or
  double-charge. The non-idempotent boundary needs careful
  case-by-case review, not a sed-replace.
- **A working pipeline today** that already handles fatal-vs-
  recoverable correctly via control flow. The Phase 2 migration is
  purely about making that handling **typed and inspectable**, not
  about fixing a broken behavior.

Phase 1's primitives are tested in isolation and ship as opt-in:
nothing in the codebase calls them yet, so there is zero behavior
risk. Phase 2 migrations can land incrementally, one integration at a
time, each behind its own PR with its own targeted tests.
