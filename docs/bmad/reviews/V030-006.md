## The underlying issue

`waitForImages` in `hooks/useIdeaProcessor.ts` was polling
`getImages()` â€” which returns `useImageGeneration.images` via the
daemon's `latestPropsRef` â€” for up to **90 Ă— 3s = 4.5 minutes**.
But `generateComparison` (from `useComparison`) writes its outputs to
`comparisonResults`, a *separate* useState store. The two never meet.
Every idea paid the full 4.5-minute cap before `waitForImages` gave up
with an empty array and the pipeline stumbled onward with no images.

The fire-and-forget contract also hid this from tests: the
pipeline-processor's `waitForImages` dep is injected, so unit tests
never exercised the real hook wiring.

## Fix â€” AC-1 & AC-2: observable Promise

`generateComparison` now returns `Promise<GeneratedImage[]>` â€” the array
of successful model outputs collected during its own Leonardo polling.
The side-effectful `setComparisonResults` writes are kept for UI, but
the return value becomes the source of truth for the pipeline.

`useIdeaProcessor` captures that Promise in a closure variable:

```ts
let imageReadyPromise: Promise<GeneratedImage[]> | null = null;
triggerImageGeneration: (prompt, modelIds) => {
  imageReadyPromise = generateComparison(prompt, modelIds, { skipEnhance: false });
  return imageReadyPromise.then(() => undefined);
},
waitForImages: async () => {
  if (!imageReadyPromise) return [];
  const readyImages = await awaitImagesOrSkip(imageReadyPromise, skipSignal);
  for (const img of readyImages) {
    if (!perIdeaImageIds.includes(img.id)) perIdeaImageIds.push(img.id);
  }
  return readyImages;
},
```

`lib/image-readiness.ts::awaitImagesOrSkip(promise, signal)` is a new
pure helper that resolves with the image array when the Promise
settles, or rejects with `SkipIdeaSignal` if the signal aborts first.
Double-settle is guarded, and the abort listener is detached on either
settlement path.

Net effect per idea: **~4.5 min saved** when images arrive cleanly;
user skip now fires within one microtask of the abort; per-model
failures still collapse gracefully (empty array resolves `waitForImages`
and the processor logs the timeout-equivalent `"Timed out waiting"`
branch, which is now a genuine error rather than a silent 4.5-min wait).

The now-unused `getImages` dep was removed from `UseIdeaProcessorDeps`;
`usePipeline` was updated to match. `daemon.getImages` is preserved
for future callers â€” out of scope to rip it out now.

## Fix â€” AC-3: accumulatedPosts reset per cycle

`hooks/usePipelineDaemon.ts:439` previously declared
`const accumulatedPosts: ScheduledPost[] = []` *outside* the
continuous-mode `do/while` loop, then pushed into it from every
processed idea across every cycle. In a long-running continuous daemon
(24+ hours, 100s of ideas) this grew without bound and duplicated data
that was already persisted in `settings.scheduledPosts`.

Moved the declaration inside the `do` block so each cycle starts with
an empty array. The week-fill `countFutureScheduledPosts` check at
line ~605 already unions with `settings.scheduledPosts`, so the reset
is invisible to callers â€” prior-cycle posts are still counted via
settings.
