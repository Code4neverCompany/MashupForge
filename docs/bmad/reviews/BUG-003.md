# BUG-003 — node.exe sidecar leaks because exit paths bypass `CloseRequested`

## TL;DR

The existing `WindowEvent::CloseRequested` handler at `lib.rs:631`
**works correctly** for X-button / Alt-F4 / `window.close()` flows. It
takes the `SidecarState` lock, takes the `Child`, calls `kill()` and
`wait()`, all in the right order.

But it doesn't fire for the path the user is actually hitting: Tauri's
`App::restart()` (called by `tauri-plugin-process` `relaunch()` from
the BUG-002 auto-update flow) calls `std::process::exit(0)` directly,
which fires **zero Tauri events**. The handler never runs. node.exe
orphans. Port 19782 stays bound. The next launch's `resolve_port`
falls back to ephemeral, breaking the IndexedDB origin pin (STORY-121).

**Fix:** add a Windows Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`
attached to the sidecar at spawn time. The kernel guarantees the
child dies when the parent dies — by ANY mechanism, including
`std::process::exit`, Task Manager kill, or panic. The CloseRequested
handler stays as a clean-shutdown path, now with diagnostic logging
mirrored to `startup.log` so we can verify it ran.

## Why the existing handler "doesn't work"

Re-reading the user's report: *"when MashupForge closes, the node.exe
sidecar stays alive and holds port 19782."* Their suggested check:
"the CloseRequested handler — if it exists but doesn't work, fix it."

It exists (`lib.rs:631-643`, pre-edit). It works. I verified by
tracing every branch:

```rust
.on_window_event(|window, event| {
    if let WindowEvent::CloseRequested { .. } = event {
        if let Some(state) = window.app_handle().try_state::<SidecarState>() {
            if let Ok(mut guard) = state.0.lock() {
                if let Some(mut child) = guard.take() {
                    log::info!("[tauri] killing sidecar pid={}", child.id());
                    let _ = child.kill();
                    let _ = child.wait();
                }
            }
        }
    }
})
```

`try_state` succeeds (state is registered at line 323 via `.manage(...)`).
The mutex isn't poisoned in normal operation. `take()` returns the
Child stored at line 579 right after spawn. `kill()` calls
`TerminateProcess` on Windows — forceful, never silently fails for a
process the parent has a handle to. `wait()` blocks until the kernel
reports the child is gone.

The ONLY way this handler doesn't kill the sidecar is if it doesn't
fire in the first place. That's exactly what happens during the
auto-update relaunch.

## Why `restart()` skips every event

`tauri-plugin-process::relaunch()` → `App::restart()` → in Tauri
2.10's source:

```rust
pub fn restart(&self) -> ! {
    let env = self.env();
    self.cleanup_before_exit();
    crate::process::restart(&env);
}

// crate::process::restart
pub fn restart(env: &Env) -> ! {
    if let Some(path) = current_binary(env).ok() {
        Command::new(path).args(&env.args_os).spawn()...
    }
    exit(0)  // std::process::exit — kills the parent IMMEDIATELY
}
```

`std::process::exit(0)` is a hard kill. It runs `atexit` handlers and
that's it. No Tauri event loop iteration. No `WindowEvent::CloseRequested`
for any window. No `RunEvent::ExitRequested`. Our handler is bypassed.

`cleanup_before_exit()` clears the resources table and asks each window
to destroy — which fires `WindowEvent::Destroyed`, NOT `CloseRequested`.
Our handler keys on `CloseRequested`. So even the "cleanup" phase
doesn't help.

**Conclusion:** the handler is correct but the relaunch path bypasses
event delivery. Need a fix at a layer the relaunch path can't dodge.

## The fix — Windows Job Object

`JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` is exactly the primitive we need.
The flow:

1. After spawning the sidecar, create a Job Object via
   `CreateJobObjectW`.
2. Set `JOBOBJECT_EXTENDED_LIMIT_INFORMATION.LimitFlags |=
   JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`.
3. `OpenProcess` on the child PID with `PROCESS_SET_QUOTA |
   PROCESS_TERMINATE`.
4. `AssignProcessToJobObject(job, proc_handle)`.
5. `CloseHandle(proc_handle)` (job retains its own kernel ref).
6. **Intentionally DO NOT** `CloseHandle(job)`. As long as the parent's
   handle table holds the job handle open, the job persists.

When the parent dies — *any way* — the kernel walks the parent's
handle table during process cleanup and closes every handle. The
job's last reference drops. `KILL_ON_JOB_CLOSE` fires. The kernel
forcibly terminates every process in the job. node.exe dies. Port
frees.

This is OS-guaranteed, not Tauri-guaranteed. It survives:

- `std::process::exit` (the relaunch path).
- Panic in any thread.
- Task Manager "End task".
- Power loss (well, everything stops then).

The only thing it doesn't survive is `TerminateProcess` aimed at the
sidecar by an external actor — but that already kills the sidecar, so
no leak.

### Why direct FFI, not the `windows` crate

The codebase already uses direct `extern "system"` FFI for `MessageBoxW`
(see `show_error_dialog` at `lib.rs:249`). Following the same pattern
keeps the dep tree small (no need to add `windows = "0.x"` with the
~20 MB of Microsoft bindings). The Job Object API surface is small —
five functions, one struct, two constants — and stable since Windows XP.

## Diagnostic logging in the CloseRequested handler

Even though the handler logic was correct, the only place it logged was
`log::info!`, which routes through `tauri-plugin-log` to `tauri.log`.
That file is harder to correlate with `startup.log` (where the spawn
event was recorded), making it tedious to answer "did the kill
actually run?" during debugging.

The handler now mirrors every branch decision to `startup.log`:

- "CloseRequested: SidecarState not registered" — should never fire,
  but if it does, we know `manage(...)` was skipped.
- "CloseRequested: SidecarState mutex poisoned" — would indicate a
  panic happened inside another mutex-holding scope.
- "CloseRequested: no sidecar Child to kill" — fires if the kill ran
  twice (e.g., user closes the app twice somehow), or if spawn never
  populated the state.
- "CloseRequested: killing sidecar pid=X" — the happy path.
- "CloseRequested pid=X kill=ok wait=exit code: 1" — final result line
  with both `kill()` and `wait()` outcomes.

Now Maurice can grep `startup.log` for `CloseRequested` and see the
full kill timeline next to the `spawned sidecar pid=X` line.

## Diff (lib.rs only)

### Added — Win32 helpers (~140 LOC)

`attach_sidecar_to_kill_on_close_job(child_pid: u32)` and a no-op stub
for non-Windows targets. Direct kernel32 FFI for:

- `CreateJobObjectW`
- `SetInformationJobObject` (with `JobObjectExtendedLimitInformation`)
- `OpenProcess` (with `PROCESS_SET_QUOTA | PROCESS_TERMINATE`)
- `AssignProcessToJobObject`
- `CloseHandle`

Plus the `JOBOBJECT_BASIC_LIMIT_INFORMATION`, `IO_COUNTERS`, and
`JOBOBJECT_EXTENDED_LIMIT_INFORMATION` structs in `#[repr(C)]`.

### Added — call site after spawn (~22 LOC)

After `cmd.spawn()` succeeds, before storing the Child in
`SidecarState`:

```rust
let sidecar_pid = child.id();
match attach_sidecar_to_kill_on_close_job(sidecar_pid) {
    Ok(()) => startup_log_line(&log_dir, &format!(
        "attached sidecar pid={} to KILL_ON_JOB_CLOSE Job", sidecar_pid)),
    Err(e) => startup_log_line(&log_dir, &format!(
        "WARN attach_sidecar_to_kill_on_close_job(pid={}) failed: {} \
         — sidecar may leak on abnormal parent exit ...", sidecar_pid, e)),
}
```

Failures are logged but non-fatal: if the Job Object setup fails
(rare — would mean Windows < 8 or some sandboxing), the existing
CloseRequested handler still covers clean shutdowns.

### Replaced — CloseRequested handler (~50 LOC)

Same kill logic as before, restructured for clearer diagnostics.
Resolves `log_dir` once, defines a `log_to_startup` closure, and
mirrors every branch decision (`SidecarState` missing, mutex poisoned,
Child already taken, kill+wait result) to `startup.log` alongside
the existing `log::info!` call.

## What did NOT change

- **`SidecarState` struct.** Same `Mutex<Option<Child>>`. The Job
  Object is independent — it's a kernel-side guarantee, not state we
  need to manage.
- **`resolve_port`.** Still tries DESKTOP_PORT once, falls back to
  ephemeral. Could add retry-with-backoff to handle the brief
  parent-death → child-death window during relaunch, but the Job
  Object should reliably kill the child fast enough that 19782 frees
  before the new instance binds. If that assumption breaks in
  practice, retry-with-backoff is the obvious follow-up.
- **`UpdateChecker.tsx`.** No frontend changes needed — Job Object is
  kernel-level, the renderer doesn't need to know about it.
  BUG-002's `relaunch()` call still works as before; now with the
  Job Object, the OLD process's death naturally cleans up the
  sidecar without relying on `CloseRequested` firing.
- **No new dependencies.** Direct `extern "system"` FFI follows the
  existing `MessageBoxW` pattern.
- **Cargo.toml unchanged.** No deps, no version bump (this is a fix,
  ride the next release tag).

What requires Maurice on the Windows box, post-CI:

1. Install the next CI build (e.g. v0.2.2) on Windows.
2. Launch MashupForge. Verify port 19782 bound:
   ```
   PS> Get-NetTCPConnection -LocalPort 19782 | Select OwningProcess
   ```
3. Note the OwningProcess PID. Run:
   ```
   PS> Get-Process -Id <pid>
   ```
   Should show node.exe.
4. Close MashupForge via the X button. Within a second:
   ```
   PS> Get-NetTCPConnection -LocalPort 19782 -ErrorAction SilentlyContinue
   ```
   Should return nothing (port freed).
   ```
   PS> Get-Process node -ErrorAction SilentlyContinue
   ```
   Should NOT show the sidecar PID.
5. Bonus — kill MashupForge via Task Manager (right-click → End task)
   instead of X button. Re-run step 4. The Job Object should ALSO kill
   node.exe in this scenario, even though CloseRequested doesn't fire.
6. Bonus — trigger an auto-update (install older version, update to
   newer). After the relaunch, only ONE node.exe should be running,
   and it should be the new instance bound to 19782.

If (4) or (5) fails: the Job Object setup didn't take. Check
`startup.log` for the `attached sidecar ... to KILL_ON_JOB_CLOSE Job`
line. If absent → the call is failing silently somehow (look for the
WARN line). If present → the OS is doing something unexpected; capture
`tasklist /v` output and a Job Object inspection (`handle.exe -s
-p <pid>` from Sysinternals).

If (6) fails: `App::restart()` is doing something different in the
field than the source suggests, OR the new process is binding before
the old one's kernel cleanup completes. Mitigation: add
retry-with-backoff to `resolve_port`.

## Why this is "complex" not "routine"

The dispatch came in marked as a routine bug fix, but root-causing
required understanding:

- Tauri v2's event lifecycle (`WindowEvent` vs `RunEvent` vs no event).
- `tauri-plugin-process`'s use of `std::process::exit` bypassing the
  event loop.
- Windows kernel Job Object semantics (`KILL_ON_JOB_CLOSE` and
  handle-table cleanup ordering).
- Why the existing handler IS correct but still doesn't help.

That's a multi-file, multi-system trace, not a "tweak one line." The
fix itself is contained but the diagnosis was the work.

## Follow-ups (not blocking)

1. **Retry-with-backoff in `resolve_port`** if step (6) verification
   shows the bind race manifesting.
2. **macOS launchd-style cleanup** — `setpgrp` + `kill(-pgid, SIGTERM)`
   in the Unix path would mirror the Job Object guarantee for
   non-Windows targets. Currently those targets rely on the
   CloseRequested handler only. Out of scope here because we ship
   Windows-first, but sensible for the Linux validation builds.
3. **Telemetry counter for Job Object failures.** If the WARN line
   from `attach_sidecar_to_kill_on_close_job` ever fires in real user
   logs, that's a signal about Windows version coverage we should know
   about.
