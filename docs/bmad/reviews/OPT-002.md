## Problem

Two of the hottest pipeline call sites — `/api/trending` (pi.dev
research) and `/api/leonardo` POST (image generation submit) — used
bare `fetch`. A single network blip or transient 5xx aborted the
whole pipeline cycle for an idea, with the only recovery being
"start the run over". That's bad UX for a 5–10 minute pipeline.

## Fix shape

Single new helper at `lib/fetchWithRetry.ts`:

```ts
export async function fetchWithRetry(input, init?, opts?): Promise<Response> {
  const attempts = opts.attempts ?? 3;
  const baseDelayMs = opts.baseDelayMs ?? 250;
  for (let i = 0; i < attempts; i++) {
    try {
      const res = await fetch(input, init);
      if (res.status >= 500 && i < attempts - 1) {
        await sleep(baseDelayMs * 2 ** i);
        continue;
      }
      return res;
    } catch (e) {
      if (i < attempts - 1) { await sleep(baseDelayMs * 2 ** i); continue; }
      throw e;
    }
  }
}
```

Defaults: 3 attempts, 250ms → 500ms → 1000ms exponential backoff.
Worst-case added latency on a fully-failing call is ~1.75s before
the final throw — small enough that the user perception is "slight
hiccup" instead of "pipeline died".

## Design choices

**Retry only on 5xx and network errors.** 4xx responses are
returned as-is — the server is telling us "your input is wrong, do
not retry". This is the standard HTTP retry semantics every HTTP
library uses; not novel.

**Skip non-idempotent endpoints.** `/api/social/post` is the
obvious candidate to NOT migrate — retrying a partially-succeeded
Instagram post would risk double-posting. Same logic for the
Leonardo status polling loop (`/api/leonardo/[id]` GET) which
already has its own polling cadence; retrying inside the poll would
just stack delays. Only the two truly transient call sites move.

**Don't fold into a global fetch interceptor.** Keeping the helper
opt-in per call site means non-idempotent calls can't accidentally
inherit retry behaviour later. Less magic, less surprise.

**No jitter (yet).** With three attempts and a single client, the
synchronized-retry herd problem doesn't exist. If we ever migrate
the parallel image generation path (currently not retried), revisit.

## Test coverage

7 vitest cases using `vi.useFakeTimers()` + `vi.spyOn(globalThis,
'fetch')`:

1. 200 → returns immediately, 1 fetch call
2. 400 → returns immediately (no retry on 4xx), 1 fetch call
3. 500 then 200 → retries once, 2 fetch calls
4. Persistent 503 → exhausts 3 attempts, returns last 503
5. Network error then 200 → retries once
6. Persistent network error → throws after exhausting attempts
7. `attempts: 1` honoured — no retry

Caught one test bug along the way: `expect(promise).rejects` needs
the assertion to be set up *before* `runAllTimersAsync` to avoid an
unhandled rejection warning. Pattern used:

```ts
const assertion = expect(promise).rejects.toThrow('...');
await vi.runAllTimersAsync();
await assertion;
```

## What this does NOT solve

OPT-002 is "auto-retry on transient failures", not a comprehensive
resilience pass. Out of scope:

- Circuit breaker / open-state after N consecutive failures
- Retry budget across the whole pipeline run
- Migrating non-idempotent calls (would need an idempotency-key
  layer first)
- Telemetry on retry counts (worth doing if maurice wants to know
  *how often* this fires)

Each could be a follow-up if the simple version doesn't cover the
real-world failure modes maurice was hitting.
