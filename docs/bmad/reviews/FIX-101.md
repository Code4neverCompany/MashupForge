## Problem

Both Instagram publish flows (single-image at `route.ts:182` and
carousel at `route.ts:215`) waited a hardcoded 5 seconds between
container creation and `media_publish`. Two failure modes:

- **Wasted latency.** Most container ingestions finish well under 5s,
  so every post paid a full second-stretch tax.
- **Silent flake.** Slow uploads occasionally took longer than 5s, and
  `media_publish` was called against a still-`IN_PROGRESS` container.
  Graph API returns a generic error in that case which was being
  surfaced as a JSON-parse failure (the same pre-FIX-133 hidden by
  `parseJsonOrThrow`, but the underlying retry was still missing).

## Fix

Added `waitForIgContainerReady(hostUrl, containerId, accessToken, ctx)`:

- Polls `GET https://{hostUrl}/v19.0/{containerId}?fields=status_code`
  every 1.5s.
- Resolves on `FINISHED` or `PUBLISHED`.
- Throws on `ERROR` or `EXPIRED` with the terminal state in the
  message.
- Times out at 60s — far longer than the worst real-world ingestion
  Maurice has seen, but short enough to surface a stuck container
  before the request hangs.

Both publish flows replaced their `await new Promise(setTimeout, 5000)`
with `await waitForIgContainerReady(...)`. No other changes.

## Why polling, not webhooks

Graph API does support container-status webhooks, but standing one up
needs a public HTTPS endpoint and a Meta App review pass. Polling is
strictly better than the existing sleep and ships in one file.

## QA checklist

1. Post a single Instagram image. Verify it publishes faster than
   before on a normal-sized upload.
2. Post a 4-image carousel. Verify all four children land in the same
   post (the polling doesn't change carousel semantics, just the wait).
3. Force a slow ingestion (post a high-resolution PNG) and confirm
   the publish succeeds instead of failing on the old 5s timeout.
4. Confirm error path: pass a deliberately bad image_url, observe the
   `ERROR` status_code surface in the API response.
