# V030-007 — Smart Parameter Pre-fill in Studio Tab

**Status:** done
**Date:** 2026-04-18

## Why

The Studio/Compare tab required the user to manually pick models,
aspect ratio, style, image size, and negative prompt before every
generation. That's busy-work the system can do itself: the prompt
contains enough signal to guess sensible defaults, and prior winning
generations are a ground-truth record of which model + settings worked
for similar prompts.

V030-007 adds an explicit "Suggest Parameters" affordance that
proposes all five fields in one shot, with a one-line reason per
field, and lets the user apply-as-is or override any individual
field before generating.

## What changed

### New — `lib/param-suggest.ts`
Pure deterministic suggestion engine. Given:
- the current prompt,
- `LEONARDO_MODELS` + `MODEL_PROMPT_GUIDES`,
- the 19 `LEONARDO_SHARED_STYLES`,
- the full `savedImages` list (prior generations),

…returns a `ParamSuggestion` containing a ranked model shortlist,
aspect ratio, style, image-size tier, optional negative prompt, and a
`reasons` object with human-readable explanations per field.

Design choices:
- **No LLM call** — keyword rules + Jaccard similarity. Instant,
  deterministic, testable without network stubs, and cheap enough to
  run on every prompt change if we ever want live suggestions.
- **Ordering matters** — 9:16 (vertical/reel/tiktok) checked before
  2:3 (portrait/character) so "vertical reel of a dancer" goes to 9:16.
- **Style gating** — a style rule only fires if the target name is
  present in `availableStyles`. Keeps the suggestion honest.
- **Excluded models** — defaults to `['nano-banana']` to match
  `pipeline-processor.ts` which skips nano-banana.
- **Prior-success boost** — winners/approved/post-ready images with
  Jaccard(prompt) > 0.1 add `score × 10` to their model's ranking. The
  `priorMatchCount` is surfaced to the UI so the card can say
  "— 3 prior matches".
- **Negative prompt carry-over** — the closest prior winner with a
  non-empty `negativePrompt` seeds the suggestion.

### New — `tests/lib/param-suggest.test.ts`
18 vitest cases covering: aspect-ratio heuristics (portrait, landscape,
mobile, default), style heuristics (anime, monochrome, no-match skip,
availability gating), image-size tier (detail keywords → 2K),
nano-banana exclusion, default/custom `topN`, prior-winner boost,
negative-prompt carry-over, non-winner ignore, explicit exclusion
lists, and empty-prompt defaults. All 18 pass.

### New — `components/ParamSuggestionCard.tsx`
Two-mode card that appears under the prompt textarea when the user
clicks **Suggest Parameters**.
- **View mode** — summary rows showing each field's value and reason.
- **Edit mode** — inline controls: model toggles (all Leonardo models
  selectable), aspect-ratio + image-size selects, style select
  (gated to `availableStyles`), negative-prompt text input.
- Apply button disabled when `modelIds.length === 0`.
- Styled to match the existing Studio-tab cyan aesthetic
  (`#00e6ff` border + shadow).

### Modified — `types/mashup.ts`
Exported `LEONARDO_SHARED_STYLES` so the suggestion engine and the
card can reference the canonical 19-style list. Const was previously
module-local.

### Modified — `components/MainContent.tsx`
- Imports `suggestParameters`, `ParamSuggestion`, `ParamSuggestionCard`,
  `LEONARDO_SHARED_STYLES`, `MODEL_PROMPT_GUIDES`.
- New state: `paramSuggestion: ParamSuggestion | null`.
- New handlers:
  - `handleSuggestParameters()` — reads the prompt, runs the engine,
    sets the card. Warns via toast if the prompt is empty.
  - `handleApplySuggestion(modelIds, options)` — pushes the selected
    models into `comparisonModels` and merges options into
    `comparisonOptions`, then dismisses the card.
- UI insertion: a small "Suggest Parameters" button sits below the
  prompt textarea and above the existing "AI-Optimized Parameters"
  read-only section. The card appears only after the user requests
  it — no clutter on first render.

## Notes

- Engine is a pure function with no I/O — suitable for a live-update
  mode later (e.g. debounced `onChange` rather than an explicit button)
  without re-architecture.
- The existing per-model `enhancePromptForModel` preview path is
  unchanged; smart pre-fill picks the *shared* params (ratio, size,
  negative) while pi still tunes per-model style/prompt enhancements
  at generation time. No duplication.
- No changes to API routes, schemas, or persisted settings shape.
