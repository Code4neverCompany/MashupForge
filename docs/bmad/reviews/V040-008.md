# V040-008 — Per-platform pipeline approval gating

The pipeline previously hard-coded every produced post to land as
`pending_approval`. That made the approval queue safe but turned the
"auto-post" toggles into a lie: even Discord webhook posts, which are
~zero-risk, sat in the queue until a human clicked through.

This task adds a per-platform auto-approve toggle. Posts whose platforms
are *all* auto-approved skip the queue (`status: 'scheduled'`); posts
that touch any manual-approve platform still gate through approval.

## Why "complex" classification

Per `~/.claude/CLAUDE.md` autonomic-loop rules, this task is `complex`
because it:
- mutates `UserSettings` schema (new optional field)
- changes pipeline behavior shape — a previously-deterministic
  `'pending_approval'` outcome now branches on user config
- spans 4 files (schema, helper, pipeline-processor, settings UI)

Hermes dispatched it directly, so it ran rather than being lifted into
proposals. Documenting the decision so future autonomic-loop turns have
precedent: direct Hermes assignments override the default propose-don't-
run rule for complex work.

## What changed

### `types/mashup.ts`

Added one optional field to `UserSettings`:

```ts
pipelineAutoApprove?: Partial<
  Record<'instagram' | 'pinterest' | 'twitter' | 'discord', boolean>
>;
```

`Partial` + missing keys means: omitted platforms fall through to
hard-coded defaults (see below). Old persisted settings keep working —
no migration needed.

### `lib/pipeline-daemon-utils.ts`

Added two pure helpers:

- `isPlatformAutoApproved(platform, config)` — looks up an explicit
  user override; falls back to a `DEFAULT_AUTO_APPROVE` map.
- `resolvePipelinePostStatus(platforms[], config)` — returns
  `'scheduled'` only if **every** platform on the post is auto-approved;
  otherwise `'pending_approval'`. Empty platforms array also returns
  `'pending_approval'` (defensive).

`DEFAULT_AUTO_APPROVE`:
| Platform   | Default     | Reason |
|------------|-------------|--------|
| instagram  | manual (false) | Graph API is the most failure-prone integration — rate limits, content flags, token rotation. Auto-posting there silently is the most common foot-gun. |
| pinterest  | auto (true) | Pre-V040-008 behavior; webhook-style posting is reliable. |
| twitter    | auto (true) | Pre-V040-008 behavior. |
| discord    | auto (true) | Pre-V040-008 behavior; lowest-stakes channel. |

The Instagram-default-manual choice is the only behavioral break from
prior versions. Users who never open the new setting will see exactly
one new manual-approval gate (Instagram). Everything else continues to
auto-flow as before.

### `lib/pipeline-processor.ts`

Both call sites that constructed `ScheduledPost` objects with
`status: 'pending_approval'` now use the helper:

- L266 — carousel branch: `status: carouselStatus` where
  `carouselStatus = resolvePipelinePostStatus(pipelinePlatforms, settings.pipelineAutoApprove)`
- L368 — single-post branch: same call, inline at the field.

`pipelinePlatforms` was already in scope at both sites (it's the
filtered `availablePlatforms` array the user enabled in the pipeline
config). No new data flow.

### `components/PipelinePanel.tsx`

Added a new "Auto-Approve (per platform)" section in the settings card,
sandwiched between the Platforms picker and the Daily Caps grid. UI
mirrors the daily-caps grid pattern (`grid grid-cols-2 sm:grid-cols-4
gap-2`, same per-platform color palette) so the visual grouping reads
as one unit.

Each tile shows:
- a checkbox bound to `isPlatformAutoApproved(p, autoApprove)` so the
  defaults render correctly even with no user config saved
- the platform name
- a state badge: green "AUTO" or amber "MANUAL"

Persistence is via `updateSettings({ pipelineAutoApprove: { ...autoApprove, [p]: value } })`
— the standard pattern used by the existing daily-caps code right
beneath it.

A footer line spells out the strict-gating rule and calls out
Instagram's manual default so the behavioral break is discoverable
in-app, not just in the changelog.

## Strict multi-platform gating — rationale

A carousel that fans out to Instagram (manual) + Twitter (auto) is one
mistake away from being a misfire. The whole post gates through manual
review — we do **not** split it into "post the Twitter half now, queue
the Instagram half." Reasoning:

- Splitting would mean publishing a multi-platform post asymmetrically,
  which breaks the user's mental model of "one post, multiple channels."
- A user who wants per-channel granularity can already do that by
  scheduling single-platform posts (the pipeline supports both modes).
- The wrong direction to err is "auto-post a half" — the right
  direction is "make the human click once."

This is documented in the helper's JSDoc and surfaced in the settings
footer.

## Out of scope (deliberate)

- **Per-channel post splitting.** Strict whole-post gating is the
  conservative read of the spec; if product wants asymmetric publish
  behavior later, that's a separate task with its own UX surface.
- **Migration of existing pending_approval posts.** Posts already in
  the queue stay there; the gating change only affects newly-pipeline-
  produced posts. There is no schema-side change to `ScheduledPost`,
  so no backfill is needed.
- **Approval-queue UI changes.** The approval queue (`ApprovalQueue.tsx`)
  already handles `pending_approval` posts identically regardless of
  origin. Nothing to update there.
- **Telemetry on auto vs manual approve flow rates.** Useful follow-up,
  but not in the acceptance criteria.
