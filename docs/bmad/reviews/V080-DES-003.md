# Review — V080-DES-003 Settings Tag Sync + AI Prompt Binding

**TLDR:** Tracing showed two of the three pieces were already in place. The Settings modal already exposes full add/remove controls for `agentNiches` / `agentGenres` (`SettingsModal.tsx:829-915`), and the runtime call sites — `useImageGeneration.ts:401-403`, `usePipelineDaemon.ts:247-249`, `useIdeaProcessor.ts:118-120` — already inject `Active Niches: …\nActive Genres: …` into the system prompt at every pi.dev request. The remaining gap was the **"Reset to Default Agent Personality"** button (`SettingsModal.tsx:1041`), which was writing a hardcoded paragraph that named "Marvel, DC, Star Wars, Warhammer 40k" regardless of what the user had selected — so a single click would silently overwrite a user's personalisation. Pulled the prompt builder into `lib/agent-prompt.ts` as `buildDefaultAgentPrompt({ niches, genres })`, wired Reset through it with the user's current selections (or curated defaults when empty), and pinned the contract with 8 tests.

## Files changed
- **NEW** `lib/agent-prompt.ts` — `DEFAULT_NICHES`, `DEFAULT_GENRES`, `buildDefaultAgentPrompt()` helper.
- **NEW** `tests/lib/agent-prompt.test.ts` — 8 tests pinning the interpolation contract and the "no franchise names hardcoded" regression guard.
- `components/SettingsModal.tsx` — local `RECOMMENDED_NICHES`/`RECOMMENDED_GENRES` arrays replaced with re-exports of the lib defaults; Reset button now calls `buildDefaultAgentPrompt(...)` against the user's current tag selections.

## Acceptance criteria
- [x] SettingsModal/DesktopSettingsPanel shows current niches and genres with add/remove controls — already met for SettingsModal (829-915). DesktopSettingsPanel deliberately untouched (see "What I deliberately did NOT change").
- [x] Changes in Settings persist and affect AI generation — `updateSettings` writes through to IDB; runtime call sites read live `settings.agentNiches`/`settings.agentGenres` on every pi.dev request.
- [x] AI system prompt dynamically includes user's niches, genres, and custom tags — runtime injection at three call sites was already correct; the Reset baseline now also interpolates instead of hardcoding.
- [x] AI output is visibly influenced by the configured focus areas — same as above; tested by reading `lib/aiClient.ts` and `lib/pi-client.ts:267-270` which compose `${systemPrompt}\n\n---\n\n${message}` before dispatch.
- [x] The system prompt is not a static string — it rebuilds when settings change — runtime injection rebuilds on every request; Reset rebuilds from current selections.

## What I deliberately did NOT change
- **`DesktopSettingsPanel.tsx`.** The story says "may need the same treatment". The panel's actual responsibilities are the Tauri sidecar, port handling, platform credentials, and updater config — agent personality / niche editing belongs in the AI Engine tab of `SettingsModal` (where it already lives) and would feel out of place under "Desktop". The story file's "may" is doing the load-bearing work here.
- **The runtime call sites** in `useImageGeneration.ts`, `usePipelineDaemon.ts`, `useIdeaProcessor.ts`. They were already correct — `Active Niches: ${...}\nActive Genres: ${...}` appended to `agentPrompt` per request.
- **The local `RECOMMENDED_*` constant names in SettingsModal.** Re-exported the lib defaults under the existing local identifiers so the rest of the modal (lines 861, 912 etc.) continues to compile without churn.

## Trace: how does the dynamic prompt reach pi.dev?
1. User edits niches/genres in SettingsModal → `updateSettings({ agentNiches, agentGenres })` writes to IDB via `MashupContext`.
2. A pipeline cycle fires: `usePipelineDaemon.autoGenerateIdeas()` reads `s.agentPrompt`, `s.agentNiches`, `s.agentGenres` from `latestPropsRef.current.settings` and composes `${agentPrompt}\nActive Niches: ${niches.join(', ')}\nActive Genres: ${genres.join(', ')}` into `systemContext`.
3. `streamAIToString(systemContext, ...)` → `aiClient.streamChat({ systemPrompt: systemContext, ... })` → POSTs to `app/api/pi/prompt/route.ts`.
4. The route composes a final system block (`[directive, systemPrompt, focusBlock, ...].filter(Boolean).join('\n\n')`) and hands it to `piPrompt(message, { systemPrompt: composedSystem })`.
5. `lib/pi-client.ts:269` prepends the system block to the user message before the actual GLM call: `${options.systemPrompt}\n\n---\n\n${message}`.

Same path for image generation (via `useImageGeneration`) and idea expansion (via `useIdeaProcessor`). Confirmed dynamic end-to-end.
