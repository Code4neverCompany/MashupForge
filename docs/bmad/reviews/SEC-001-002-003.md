## SEC-001 — proxy-image SSRF allowlist

### Problem

`app/api/proxy-image/route.ts` previously took a `?url=` query parameter
and proxied any URL the caller supplied. No host check, no protocol
check. Classic SSRF: an attacker could ask the server to fetch
`http://169.254.169.254/latest/meta-data/` (AWS metadata), `file:///etc/passwd`,
or any internal service the deploy box can reach.

### Fix

Allowlist the three hosts we actually use, parse the URL safely, and
require https. 14-test vitest suite covers the boundary:

```ts
const ALLOWED_HOSTS = new Set<string>(['cdn.leonardo.ai', 'i.uguu.se']);
const ALLOWED_HOST_SUFFIXES = ['.storage.googleapis.com'];

export function isAllowedUrl(raw: string): boolean {
  let parsed: URL;
  try { parsed = new URL(raw); } catch { return false; }
  if (parsed.protocol !== 'https:') return false;
  const host = parsed.hostname.toLowerCase();
  if (ALLOWED_HOSTS.has(host)) return true;
  return ALLOWED_HOST_SUFFIXES.some((s) => host.endsWith(s));
}
```

Tests cover all the obvious bypasses:
- http downgrade
- file://, gopher:// protocols
- arbitrary external hosts
- localhost / 127.0.0.1
- **suffix lookalikes** like `cdn.leonardo.ai.evil.com` (caught by
  the exact-host check, not the suffix list)
- empty / malformed URL → false
- case-insensitive hostname match
- bare `storage.googleapis.com` (no subdomain) → false (suffix
  list requires a leading `.` so the bucket subdomain is required)

### Decision: 403 not 400 on miss

A blocked URL is "I refuse to fetch this", not "your request is
malformed". 403 is the correct semantics and lets clients
distinguish "missing param" (400) from "host not allowed" (403).

## SEC-002 — defense-in-depth serverless guards

### Problem

`pi/install`, `pi/start`, `pi/stop` all spawn child processes
(spawnSync, sidecar process). On a serverless runtime
(Vercel/Lambda/Netlify/CF Pages) these would either fail with cryptic
errors or, worse, succeed in unexpected ways. `pi/setup` already had
an `isServerless()` check; the other three did not.

### Fix

Extracted the inline `isServerless()` from `pi/setup` into a shared
`lib/runtime-env.ts` and imported it across all four pi routes.
Each route returns 503 with a clear message if it detects a
serverless platform.

### Why a shared file (not copy-paste)

Four copies of the same 5 lines guarantees drift. One copy in
`lib/runtime-env.ts` means future serverless platforms (e.g.
Deno Deploy) only need a single edit.

## SEC-003 — POSIX shell injection in tmux command

### Problem

`pi/setup` POSIX path used `execSync` with a single-quoted shell
fence:

```ts
execSync(`tmux new-session -d -s pi-setup -x 120 -y 30 '${piPath} /login'`);
```

If `piPath` ever contained a single quote — e.g. resolved from
`$HOME` like `/home/user's stuff/.npm/lib/node_modules/.bin/pi` —
the fence breaks and the tail becomes shell code. Today `piPath`
comes from `getPiPath()` which reads from a known npm prefix, so
exploit surface is low — but it's a pure footgun waiting for a
filesystem path with a quote/space/semicolon to land somewhere
upstream.

### Fix

Switched to `spawnSync` with the array form, which passes args
through `execve` directly without shell parsing:

```ts
spawnSync('tmux', ['kill-session', '-t', 'pi-setup'], { stdio: 'ignore' });
const result = spawnSync(
  'tmux',
  ['new-session', '-d', '-s', 'pi-setup', '-x', '120', '-y', '30', piPath, '/login'],
  { encoding: 'utf8' },
);
if (result.status !== 0) {
  throw new Error(`tmux new-session failed (exit ${result.status}): ${result.stderr?.trim() || 'unknown'}`);
}
```

`piPath` is now a single argv element — no escaping concerns, no
shell parsing, can contain any character valid in a filesystem path.

### Windows path left alone (intentional)

The Windows branch uses `cmd /c start cmd /k <pi> /login` to pop a
visible terminal window. The current implementation uses backtick
double-quotes around `piPath` which has the same class of issue
with embedded double quotes. **But:** Windows filesystem paths
can't contain double quotes (Win32 reserved character), so the
exploit shape is structurally impossible on the Windows surface.
SEC-003 explicitly called out the POSIX fence; leaving Windows
alone keeps the change scoped.
