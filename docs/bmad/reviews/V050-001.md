# V050-001 — Mid-step pipeline resume (credit-preserving)

**Status:** done
**Date:** 2026-04-19
**Branch:** main
**Commits:** (pending)

## Why

All three brainstorm rounds ranked mid-step pipeline resume as the #1 UX
priority. The pre-V050 resume flow was infrastructure-only — checkpoints
were saved (FEAT-006), the resume prompt was shown, but accepting it
restarted the interrupted idea **from scratch**. The prompt itself
warned the user: *"Leonardo will regenerate its images and re-spend the
credits."* That's the failure mode we eliminated.

A typical interrupted run had ≥1 image already saved to the gallery
(checkpoint's `imageIds` field has tracked them since FEAT-006). All
that was missing was the wiring to consume those ids on the resume
path: look them up, hand them to the processor, skip generation.

## What changed

### `lib/pipeline-processor.ts`
- New `ResumeContext { images: GeneratedImage[] }` type.
- `processIdea` accepts an optional 8th arg `resumeFrom?: ResumeContext`.
- When `resumeFrom.images.length > 0`, the processor **skips** the four
  pre-image steps (mark in-work label/log, trending, expand, generate,
  waitForImages) and jumps straight to the per-image captioning loop
  (carousel or per-model) with `expandedPrompt = idea.concept` as the
  caption-failure fallback.
- An explicit `addLog('resume', …)` entry signals to the user that no
  Leonardo credits were spent.

### `hooks/usePipelineDaemon.ts`
- New exported `PipelineResumeHint { ideaId, images }` type.
- `ProcessIdeaFn` signature gains optional `resumeFrom`.
- `runOuterLoop(processIdea, resumeHint?)` accepts the hint, scoped to a
  local `pendingResumeHint` that is consumed by the **first** matching
  idea (by id) and cleared so subsequent ideas — and subsequent cycles
  in continuous mode — run normally.

### `hooks/useIdeaProcessor.ts`
- Forwards `resumeFrom` to `processIdeaFn`.
- Pre-seeds `perIdeaImageIds` with the resume payload so a second crash
  *during* the resumed run still records the same image set in its
  checkpoint (otherwise we'd lose them and force a full re-gen on the
  next launch).

### `hooks/usePipeline.ts`
- `startPipeline` accepts an optional `PipelineResumeHint` and forwards
  it to `runOuterLoop`.
- `acceptResume` provides `getSavedImages: () => savedImages` to the
  helper so the resume path can hydrate the hint from gallery state.

### `lib/resume-checkpoint.ts`
- New optional `getSavedImages` dep on `ApplyResumeDeps`.
- New `ResumeStartHint` type mirroring the daemon's `PipelineResumeHint`.
- `startPipeline` dep signature changed: `(resumeHint?: ResumeStartHint) => void | Promise<void>`.
- After flipping idea status, `applyResumeCheckpoint` looks up
  `cp.imageIds` against `getSavedImages()`. If ≥1 match → forwards a
  hint to `startPipeline`. If zero match (deleted/cleared) → falls back
  to the pre-V050 full-restart behavior. Pre-V050 checkpoints with
  empty `imageIds` are also handled (no hint passed).

### `components/PipelineResumePrompt.tsx`
- Branches on whether the checkpoint's images can still be located in
  the saved gallery:
  - **Credit-preserving** (recoverable images present): green `Sparkles`
    callout, `"Yes, resume (no credits)"` button, copy explains the
    captioning pickup.
  - **Full restart** (no recoverable images): retains the original
    amber `AlertTriangle` warning and `"Yes, restart (uses credits)"`
    button.

## Failure modes

- **All checkpoint images deleted before resume**: helper returns no
  hint; full restart with credits spent. UI warns the user up front.
- **Some checkpoint images deleted**: surviving images are forwarded;
  the resumed run captions/schedules just those. Acceptable degradation.
- **Crash mid-resume**: useIdeaProcessor pre-seeds `perIdeaImageIds`
  with the resume payload, so the next checkpoint write keeps tracking
  the same image set. Credit-preserving resume is then re-applicable.
- **idea.concept used as caption fallback (not the original
  expandedPrompt)**: minor degradation in the rare caption-failure
  path. Pi.dev re-expansion would be ~$0.001 but we skip it to keep
  resume instant. If caption succeeds (the common case), this fallback
  never matters.

## Out of scope (explicit non-goals)

- Re-running pi.dev expand on resume — cheap but adds latency.
- Skipping captioning for images that already have a `postCaption`.
  Future optimization; current behavior re-captions for consistency.
- Per-stage granular resume (e.g. resume *between* captioning and
  scheduling on the same image). The current granularity is per-idea,
  which is the right grain for the Leonardo-credit value proposition.
