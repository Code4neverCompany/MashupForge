## Outcome

### Log persistence — memory → IDB

New `lib/pipeline-log-store.ts` wraps idb-keyval with the same
best-effort swallowing boundary as `pipeline-checkpoint.ts`:

- `savePipelineLog(entries)` — trims to last 50, ISO-stringifies
  `timestamp`, calls `set`. Swallows errors; caller can't be broken
  by a bad storage layer.
- `loadPipelineLog()` — reads, validates `Array.isArray`, re-hydrates
  `timestamp` back to `Date`. Returns `[]` on missing / malformed /
  thrown.
- `clearPipelineLog()` — `del`, swallows errors.

`hooks/usePipelineDaemon.ts` changes:

- Dropped `log` from the old `PersistedPipelineState`. Renamed the
  helper pair `loadPersistedState`/`persistState` → `loadPersistedConfig`/
  `persistConfig` — localStorage now only holds config (small, sync).
- Added a mount useEffect that `loadPipelineLog()`s and hydrates
  `pipelineLog` state (guarded by `cancelled` so unmount-races don't
  clobber fresh state).
- Added a `[pipelineLog]` useEffect that fires `savePipelineLog(…)`
  per change — fire-and-forget; persistence is best-effort.
- Clean end of `runOuterLoop` calls `void clearPipelineLogStore()`
  alongside `void clearCheckpoint()`. Rationale: the log is only
  valuable across a crash; clean completion means both the UI
  state and the persisted copy can be dropped.
- User-facing `clearPipelineLog` control now clears IDB defensively
  in addition to resetting component state.

## Notes

- **Why IDB not localStorage for the log?** localStorage is sync and
  shared with config + other small state. The log is the largest
  single item (50 entries × ~5 fields each) and the only one that
  benefits from async persistence. Splitting storage layers keeps
  config reads fast (sync) and log writes non-blocking.
- **Date serialization:** `JSON.stringify` would have turned `Date`
  into a string silently and `loaded[i].timestamp` would be a string
  at runtime while typed as `Date` — a latent bug. Explicit
  ISO-string round-trip avoids it.
- **Unmount race:** the mount-hydration effect uses `let cancelled`
  so if the user dispatches a new log entry before the IDB read
  resolves, the stale hydrated value doesn't overwrite it. Standard
  React pattern.
