## Why

QA + developer consensus from the v0.4.x retro: a pre-commit hook
running `tsc --noEmit` + `vitest run` would have caught roughly half
the bugs that slipped this session — the v0.4.2 Bug 1 monolith wiring
issue, the v0.4.1 desktop creds detection regression, and a couple of
silent type drifts during the V030-001 split. Two lines in
`package.json`, ~3s on every commit. Net positive.

## What changed

### `package.json`

- `simple-git-hooks` added as devDep (already installed in this
  branch; v2.13.1, ~5KB, zero runtime deps).
- New scripts:
  - `precommit`: `tsc --noEmit && vitest run` — the gate.
  - `prepare`: `simple-git-hooks` — re-installs the hook after
    every `npm install` so a fresh clone gets it for free.
- New top-level `simple-git-hooks` config:
  ```json
  "simple-git-hooks": { "pre-commit": "npm run precommit" }
  ```

### `.git/hooks/pre-commit` (generated by `npx simple-git-hooks`)

Runs `npm run precommit` before every commit. Installed in this
working tree after the package.json change; future contributors get
it via `npm install` (the `prepare` script).

## Escape hatch

`SKIP_SIMPLE_GIT_HOOKS=1 git commit -m '…'` bypasses the hook for the
rare case (e.g. landing a known-broken WIP on a feature branch). Use
sparingly — the whole point is to catch regressions before they land.

## Why simple-git-hooks (not husky)

- Zero runtime deps (husky pulls in Node + a setup script).
- One-line install (`npx simple-git-hooks` after package.json change).
- No commit on `node_modules/`-shaped state — the hook is just a
  shell script in `.git/hooks/`.
- Husky's stronger features (lint-staged integration, multi-hook
  configs) aren't needed here — we just want the gate.
