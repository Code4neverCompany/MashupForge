# V060-003 — Auto-update "check unavailable" warning fix

**Status:** done
**Branch:** main
**Tests:** 483 → 487 (+4)

## Why

Maurice reported the Settings → Desktop → Updates section showing
"Last checked: never" and a red warning, despite the updater being
configured correctly in `tauri.conf.json` and capabilities. Two
distinct bugs were tangled together:

1. **Stale "Last checked: never"** — `localStorage[LAST_CHECKED_AT_KEY]`
   was only written *after* a successful `updaterMod.check()`. On
   systems hitting the BUG-ACL-005 plugin bug (`plugin:updater|check
   not allowed by ACL` from `tauri-plugin-updater@2.10.1` on Windows),
   every check threw, so the timestamp never advanced. The panel
   stayed stuck reading "never" even though the system had tried
   dozens of times.
2. **Misleading red warning** — the catch-all friendly message
   ("Auto-update check unavailable — please check manually") rendered
   in the `error` discriminator, which paints a red `AlertCircle`.
   The user has nothing to action beyond visiting GitHub Releases,
   so framing it as a system fault was wrong.

## What changed

- **`components/UpdateChecker.tsx`** — moved
  `localStorage.setItem(LAST_CHECKED_AT_KEY, …)` to fire **before**
  `updaterMod.check()`. Recording the attempt is what the panel cares
  about, not the outcome. Comment block in source pins the rationale.
- **`components/DesktopSettingsPanel.tsx`** —
  - `handleCheckNow` stamps the timestamp immediately on click,
    matching the launch-time path.
  - Added a new `unavailable` variant to the `CheckResult`
    discriminated union with a `releasesUrl` field. Renders a calm
    informational note + "Visit GitHub Releases" link in cyan, no red
    icon. Aria-label `"Auto-update unavailable on this system"` for
    accessibility and test addressability.
  - The ACL-detection branch in `handleCheckNow` now resolves to the
    new `unavailable` state instead of `error`. Non-ACL errors still
    flow into `error` with the raw detail (network failures, etc.,
    are legitimately actionable).

- **`tests/integration/desktop-update-check-acl.test.tsx`** (new, 4
  cases) — pins the V060-003 contract:
  - LAST_CHECKED_AT_KEY is stamped even when the check throws an ACL
    error.
  - The calm `unavailable` note + Releases link render on ACL denial,
    not the red error path.
  - Non-ACL errors keep the red error path so they remain visible.
  - The success-no-update path also stamps the timestamp.

## Notes

- The underlying `tauri-plugin-updater@2.10.1` ACL bug is **not**
  fixed here — that requires a Rust crate bump + release rebuild +
  cross-platform validation, which is not routine scope. The fix
  here makes the user-visible behavior honest (timestamp updates,
  no false alarm) while the upstream bug is investigated separately.
- The `unavailable` state is a sibling of `error`, not a replacement.
  Future failure modes that *are* actionable (corrupt manifest,
  endpoint changed) keep flowing into the red `error` path with the
  raw detail so the user sees something diagnosable.
- The Releases URL is hard-coded once at the top of the file. The
  same URL appears in `tauri.conf.json` `bundle.homepage` —
  intentionally not extracted to a shared constant per the no-
  scope-creep rule (one consumer, one literal).
