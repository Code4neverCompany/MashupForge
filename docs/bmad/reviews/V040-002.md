# V040-002 — Branch strategy: main / web / desktop

## Why

The repo ships two product surfaces (Vercel-hosted web app and Tauri
Windows desktop installer) from a single trunk. Until now, both relied
on `main` — Vercel auto-deploys from `main`, and Tauri NSIS releases
fire when a `v*.*.*` tag is pushed (regardless of branch). That coupling
makes "ship the web hotfix without re-cutting a desktop installer"
awkward and "hold the web release while the desktop bundle is signed"
impossible without freezing trunk.

Maurice asked for separate branches so each surface has its own release
gate.

## What changed

### `docs/branch-strategy.md` (NEW — durable team-facing reference)
- Documents the three-branch model (`main` / `web` / `desktop`).
- Spells out the promotion flow (feature → main → fast-forward to
  `web` and/or `desktop`).
- Hotfix flow per surface, with mandatory cherry-pick-back to `main`
  to prevent drift.
- CI table mapping branch → workflow → purpose.
- Decisions section justifying why we kept the existing tag-triggered
  Tauri release flow instead of switching to push-triggered.
- Open-questions section flagging the Vercel dashboard change Maurice
  needs to make manually (Production Branch → `web`), branch
  protection deferred, and the stale `cleanup-branch` on origin.

### `.github/workflows/ci.yml` (NEW)
- Runs on `push` and `pull_request` to `main`, `web`, `desktop`.
- Steps: bun install (with cache + npm fallback), `tsc --noEmit`,
  `vitest run`, `next build`. Mirrors the safety net the existing
  `tauri-windows.yml` already uses (Bun primary → npm fallback).
- Ubuntu-latest runner. Fast and cheap; the Windows-only Tauri build
  stays on the existing tag-triggered workflow.

### Branches
- Created `web` and `desktop` from current `main` and pushed both to
  origin. Both branches start at the same commit as `main` so no
  drift exists yet.

### `.github/workflows/tauri-windows.yml`
- **Unchanged.** Still tag-triggered (`v*.*.*`) so a tag pushed on
  `desktop` produces the NSIS installer + signed updater manifest.
  Switching to branch-triggered would publish on every fast-forward,
  defeating the "release gate" purpose of the split.

## Out of scope (deliberate)

- **Vercel "Production Branch" switch.** That's a dashboard action,
  not in code. Flagged in `docs/branch-strategy.md` open questions.
  Until Maurice flips it, `web` is just a label and Vercel keeps
  deploying from `main`.
- **Branch protection rules** for `web` / `desktop`. Recommended once
  the strategy has run for a sprint or two. Adding them now would
  block the maintainer's first fast-forwards without a clear payoff.
- **Per-branch divergence in CI.** `ci.yml` is one workflow that runs
  the same steps on all three branches. If `web` later needs a
  Lighthouse audit and `desktop` needs `cargo check`, they can be
  added as conditional steps (`if: github.ref == 'refs/heads/web'`)
  or as additional workflows. Pre-empting that now would be
  speculative.
- **Cleanup of the stale `cleanup-branch` on origin.** Flagged in the
  doc but not deleted — it predates this task and Maurice should
  decide.

## Risks + caveats flagged for Hermes

This is a "complex" task per the routing rubric (cross-branch + CI +
shared infra). Per CLAUDE.md the autonomic loop would normally lift
this to proposals, but Maurice assigned it directly. Calling out the
moving parts so Hermes can audit:

1. **Vercel deploys still come from `main`** until the dashboard
   switch is made. The `web` branch is currently a no-op for users.
2. **No branch protection** — anyone with push access can force-push
   `web` or `desktop`. Strategy assumes maintainer discipline until
   protection is added.
3. **Drift between `main` and release branches** is the most likely
   failure mode. The hotfix-and-cherry-pick rule in the doc is the
   only mitigation; consider a CI check that rejects PRs on `web`/
   `desktop` if their parent commit isn't reachable from `main`.
