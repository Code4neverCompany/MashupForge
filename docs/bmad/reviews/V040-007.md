# V040-007 — Persistent platform health signal in the sidebar footer

Implements Part A of `docs/bmad/stories/V040-DES-002.md`. Part B
(daily digest card) shipped under V040-005.

## Why

Without this, the only way to see whether Instagram / Pinterest /
Twitter / Discord credentials are working is to open Settings and
click through each platform section. So the user discovers broken
creds *at post time*, when a scheduled post silently fails. Not
acceptable for an automation tool.

The spec calls for a persistent, always-visible signal anchored to
the bottom of the sidebar — always on screen while the sidebar is
open, which for this app is the default.

## What changed

### `components/platform/HealthDot.tsx` (NEW)

A single-button health indicator. Props: `code` (IG/PN/TW/DC),
`platformLabel`, `state` ∈ {healthy, stale, broken, unused},
`detail` (the one-liner shown in the native browser tooltip via
`title`), and an optional `compact` flag for the mini-rail variant.

Colors follow the calendar palette language the rest of the app
uses:

| State    | Dot                                                          |
|----------|--------------------------------------------------------------|
| healthy  | emerald-500 + soft shadow                                    |
| stale    | amber-400                                                    |
| broken   | red-500 + `motion-safe:animate-pulse` + static ⚠ for reduce  |
| unused   | zinc-600/70                                                  |

Only `broken` pulses. Healthy/stale/unused stay still — pulsing is
an attention demand and should be rare.

Click dispatches `window.dispatchEvent(new CustomEvent('mashup:open-settings'))`.
MainContent *already* has a listener at L1013-1017 for that exact
event (added in an earlier story), so the dot plugs in with no
wiring change on the MainContent side.

A11y: computed `aria-label` combines platform name + state + detail,
plus the native `title` as a tooltip fallback. The pulsing dot
respects `prefers-reduced-motion` via `motion-safe:` + a static
`AlertTriangle` overlay exposed only when motion is reduced.

### `components/platform/HealthStrip.tsx` (NEW)

Derivation + layout. `usePlatformHealth` is an inline hook that
maps `settings.apiKeys`, `settings.pipelinePlatforms`, and
`settings.scheduledPosts` into per-platform `{ state, detail }` via
a pure `derive(platform, credsPresent, inPipeline, posts)` function.

**State classification rules (all from existing data, no new schema):**

- **broken** — creds missing but the platform is enabled in the
  pipeline *or* at least one post exists that uses it; OR the most
  recent `failed` post is newer than the most recent `posted` one.
- **healthy** — creds present *and* last successful post was within
  24h.
- **stale** — creds present but no post within the 24h window (or
  no post activity yet).
- **unused** — creds missing, platform not in the pipeline, and no
  scheduled posts reference it.

The detail line is shaped by state:
- healthy: `"Last post: 2h ago · 3 in queue"`
- stale: `"Last post: 4d ago · credentials may have expired"` or
  `"Configured but unused · 0 in queue"` when there's no post history.
- broken: `"Missing credentials — platform enabled in pipeline"` or
  `"Last post failed 30m ago · 1 in queue"`.
- unused: `"Not configured · not in pipeline"`.

Two exported variants:
- **`HealthStrip`** — horizontal row mounted in the sidebar's
  bottom border strip (`border-t bg-zinc-950/60`). Renders when at
  least one platform is not in the `unused` bucket; when all four
  are unused the strip hides entirely so it doesn't waste pixels.
- **`HealthMiniRail`** — a dot-only vertical rail variant for a
  future collapsed-sidebar state. **Not currently mounted** —
  `Sidebar.tsx` doesn't have a collapsed-icon-column mode yet (it's
  an off-canvas mobile drawer, not a desktop width toggle). The
  component is complete and typed; when the design adds a
  collapsed state, it drops in. See the "deliberate scope" note
  below.

### `components/Sidebar.tsx`

Two edits:
- Import `HealthStrip`.
- Render `<HealthStrip />` as the final child of the `<aside>`,
  below the input form and above the closing tag. Inherits the
  sidebar's existing persistence — always visible while the sidebar
  is open.

### No MainContent change needed

The `mashup:open-settings` CustomEvent listener already exists at
`components/MainContent.tsx:1013-1017`. HealthDot dispatches into
it for free; zero new wiring on the receiver side.

## Spec compliance

| Acceptance criterion | Status |
|---|---|
| HealthStrip in sidebar | ✅ Mounted at the bottom of the open `<aside>` |
| 4 states per platform | ✅ healthy/stale/broken/unused, derived from apiKeys + scheduledPosts + pipelinePlatforms |
| Tooltip | ✅ Native `title` attribute carries `platform · state` + one-liner detail; aria-label duplicates the information for screen readers |
| Mini-rail variant | ⚠ `HealthMiniRail` built, exported, typed, and documented — but **not mounted**. Sidebar has no collapsed-icon-column state today; mounting it requires a design decision that's out of scope for this task. See below. |
| Write inbox | ✅ (after commit) |

## Out of scope (deliberate)

- **Mounting the mini-rail.** Sidebar is an off-canvas mobile drawer,
  not a desktop width toggle — there is no collapsed-icon column to
  render the mini-rail into. Adding one is a layout change that
  belongs in its own task. The component ships ready so a future
  collapsed-state task is a one-line mount.
- **Rich positioned tooltip (multi-line, `[Open settings]` link
  inside).** The spec's L3 link is covered by the click-opens-
  settings behavior on the dot itself, so pulling in a custom
  Tooltip component for a three-line panel would be noise. Native
  `title` + `aria-label` cover L1-L2 of the spec's tooltip; the
  click handles L3.
- **Active credential validation ping** (`/api/{platform}/health`
  worker). Spec itself flags this as a separate future PROP. The
  dot gracefully degrades to `stale` when it can't confirm, which
  is honest.
- **"Open settings scrolled to platform section" deep link.** The
  CustomEvent opens the global settings modal; landing on the
  correct platform section would need a modal-side prop or hash
  route. Follow-up work.
- **Auto-live announcement of state changes.** The strip container
  carries `aria-live="polite"` so SR users will hear changes on
  the next idle — but we don't emit explicit `aria-relevant`
  tuning. If state churn on broken→healthy recovery becomes an
  accessibility issue we can revisit.
