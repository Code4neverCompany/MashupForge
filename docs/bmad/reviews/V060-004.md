# V060-004 — Pipeline: fill week first, pause between weeks

**Severity:** STORY (behavior change)
**Status:** shipped
**Date:** 2026-04-20

## Summary

Reshapes continuous-mode pipeline behavior to:

1. Fill the **current week** with engagement-best slots first.
2. Pre-schedule **2 weeks** ahead.
3. **Pause** for the configurable `interval` timer between weeks.
4. **Repeat** rolling forward into the next 2-week horizon.
5. **Removes** the `pipelineAutoPost` toggle entirely — all pipeline
   posts now land as `pending_approval` (already enforced by
   BUG-CRIT-001) and publish through the approval flow.

## Why

Two failure modes in the previous behavior:

- `findBestSlot` searched a 14-day window from the first call. The
  highest-scoring slot (often Saturday 20:00, week 2) outranked
  lower-scoring slots in the current week. Result: holes in this
  week while next week filled up first.
- Continuous mode always slept `intervalMin` between cycles even
  when the schedule was nowhere near full. Filling a 2-week window
  could take a full day instead of running back-to-back until full
  and only pausing once nothing more was needed.

The auto-post toggle is also removed because:

- BUG-CRIT-001 already forces every pipeline post into
  `pending_approval` for safety + watermark enforcement. The toggle
  did nothing useful — it just confused the UI by suggesting
  pipeline could publish directly.
- The new fill-week-first behavior is the user-visible "post
  priority" knob, so the old toggle's slot in the UI is reclaimed
  by the implicit "fill weeks in order" priority.

## Changes

### `lib/smartScheduler.ts`

- `findBestSlots` and `findBestSlot` accept `options.horizonDays`
  (default 14, back-compat). The candidate window iterates
  `horizonDays` days instead of a hardcoded 14.

### `lib/fill-week-scheduler.ts` (new)

- `pickFillWeekSlot({ posts, engagement, postsPerDay, platforms,
  caps, now? })`:
  - Computes `computeWeekFillStatus` for the next 7 days.
  - If week 1 is **not** filled → `findBestSlot` with
    `horizonDays: 7` (engagement-best slot stays inside week 1).
  - If week 1 **is** filled → `findBestSlot` with `horizonDays: 14`
    (slot can extend into week 2).
- Returns `{ date, time, week: 1 | 2 }` so callers can log which
  week the slot landed in.
- Pure: pass `now` for deterministic tests.

### `hooks/useIdeaProcessor.ts`

- `findNextAvailableSlot` now routes through `pickFillWeekSlot` and
  takes `postsPerDay`. Reason string includes `(week N, …)`.
- The processor closure passes `getSettings().pipelinePostsPerDay
  ?? 2` on every call so the live setting is read each time.

### `lib/pipeline-processor.ts`

- Removed the `autoPost` settings read.
- Removed the `if (autoPost && pipelinePlatforms.length > 0) { fetch
  '/api/social/post' … }` block (~40 lines). Pipeline posts now
  always land as scheduled `pending_approval` and never call the
  direct-publish HTTP route.
- The unused `scheduledPostId` local was deleted with that block.

### `hooks/usePipelineDaemon.ts`

- Continuous-mode sleep block rewritten:
  - Computes `horizonDays = max(settings.targetDays, 14)` so the
    daemon's pre-schedule window is at least 2 weeks even when the
    user configured a smaller UI target.
  - If `futurePosts < targetTotal` → log "running next cycle
    immediately" and `continue` the outer loop (no sleep).
  - If filled → log the one-shot week-filled prompt, log the sleep
    intent, sleep `intervalMin`, then **reset** the prompt latch so
    the next 2-week cycle can announce its own fill.
- The progress message changes from "Next cycle in N min" to "Next
  week in N min" to match the new semantics.

### UI surface

- `components/pipeline/stages.ts`:
  - Dropped `Send` icon import.
  - Dropped `'pipelineAutoPost'` from the `PipelineToggleKey` union.
  - Dropped the `'post'` stage from `STAGES`.
- `components/PipelinePanel.tsx`:
  - Removed the `autoPost` settings read, the `pipelineAutoPost`
    branch in `toggleStage`, the toggle row in the Stages grid,
    and the `(autoPost || autoSchedule)` gate on the Platforms
    picker (now `autoSchedule` only).

### Types

- `types/mashup.ts`: removed `pipelineAutoPost?: boolean;` and
  updated the surrounding doc comment to note the removal +
  reference V060-004.

### Tests

- New `tests/lib/fill-week-scheduler.test.ts` (3 cases):
  - Empty schedule → slot lands in week 1.
  - Week 1 fully booked → slot lands in week 2.
  - Week 1 partially booked → slot still lands in week 1.
  - Uses `vi.useFakeTimers` to pin `findBestSlots`'s internal `new
    Date()` to the same anchor as `computeWeekFillStatus`.
- Removed `pipelineAutoPost: false` from 4 fixtures:
  `tests/lib/pipeline-processor.test.ts`,
  `tests/hooks/useIdeaProcessor.test.ts`,
  `tests/integration/pipeline-platform-detection.test.ts`,
  `tests/integration/approval-gate-watermark.test.ts`.
