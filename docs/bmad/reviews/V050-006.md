# V050-006 — Integration tests: catch wiring bugs

**Status:** done
**Classification:** complex
**Why:** QA flagged that 319 pure-function tests didn't catch the
v0.4.2 Bug 1 family (Push to Studio bypassing param-suggest, IG creds
silently inferring `[]`, etc.). Those bugs are *wiring* bugs —
individual functions are correct in isolation, but the orchestrator
calls the wrong one or threads the wrong dep. Pure tests by design
can't see them. We needed a higher-level test layer.

## What shipped

1. **jsdom + RTL infrastructure** — minimal, additive, doesn't
   touch the existing 367 node-env tests.
2. **3 integration test files** covering the 3 wiring bugs in scope:
   - `tests/integration/push-idea-to-studio.test.ts` (6 cases)
   - `tests/integration/carousel-approval-flow.test.tsx` (4 cases)
   - `tests/integration/pipeline-platform-detection.test.ts` (4 cases)
3. **One extraction** — `lib/push-idea-to-studio.ts` — to make the
   "Push to Studio" handler unit-testable. Original handler body was
   inline in `MainContent.tsx:1064-1106`; the slim wrapper now just
   forwards setters to the helper.

### Vitest configuration

The default `environment` stays `node` so the 367 existing pure-function
tests under `tests/lib/` run unchanged with zero per-file env-switching
cost. Integration tests opt into jsdom via the per-file directive
`// @vitest-environment jsdom` at the top of each `.tsx` file. The
glob expanded from `tests/**/*.test.ts` to `tests/**/*.test.{ts,tsx}`.

`tests/setup/jest-dom.ts` registers the jest-dom matcher pack
(`toBeInTheDocument`, `toHaveTextContent`, etc.) on vitest's `expect`.
The setup file is imported in node tests too — it only registers
matchers, no DOM access happens at import time.

### Why per-file `@vitest-environment` over a workspace split

Vitest supports two patterns:

1. **Per-file directive** — `// @vitest-environment jsdom` at the top
   of each integration `.tsx`. Adds ~600 ms env startup per file but
   only on files that need DOM.
2. **Workspace split** — separate `vitest.config.node.ts` /
   `vitest.config.jsdom.ts`, two Vitest invocations.

(1) keeps the default `npm test` invocation single-pass, lets a
single watch session cover both layers, and avoids the directory
bifurcation cost. The integration suite is small enough that the
per-file env startup is negligible — total run time went from 0.92 s
(367 tests) to 1.64 s (381 tests), with ~750 ms attributable to one
jsdom spin-up.

## Test 1 — Push to Studio → param-suggest (`push-idea-to-studio.test.ts`)

**Bug it catches:** V041-HOTFIX-3BUGS Bug 1 — handler called a
hand-rolled prompt-enhance instead of the per-model `suggestParametersAI`
engine. The two surfaces have different argument shapes, so a regression
of that exact bug would now fail the `toHaveBeenCalledWith` arg-shape
assertion immediately.

**Approach:** extracted `pushIdeaToStudio(prompt, deps)` to
`lib/push-idea-to-studio.ts`. Deps interface includes a function-shaped
`suggest` member; the production call site passes `suggestParametersAI`
directly, the test passes `vi.fn()`. Six cases cover: arg shape,
view/carousel-watcher arming, suggestion fan-out into setters,
`negativePrompt` fallback to prev, pushing-flag clearing on success,
and pushing-flag clearing + state-write suppression on rejection.

**MainContent change:** `handlePushIdeaToCompare` shrank from ~42
lines to ~16 (one call to the helper, all setters threaded as deps).

## Test 2 — Carousel approval flow (`carousel-approval-flow.test.tsx`)

**Bug it catches:** the V040-DES-003 carousel grouping is React-state
+ optimistic UI + ghost memory + degrade guard, all wired through
several callback layers. A wiring break (e.g. "Approve carousel" only
fires `onApprovePost` once instead of N times, or fires on the wrong
ids) would be silently broken — the UI would still update visually
because of the local optimistic state, but the queue would never
actually clear.

**Approach:** RTL-renders `CarouselApprovalCard` in jsdom with three
mock posts, clicks the "Approve carousel" button, and asserts
`onApprovePost` was called exactly 3 times with the three sibling
post ids in order. A second case does the same for "Reject carousel".

Two more cases test the `ApprovalQueue` side: that
`groupApprovalPosts` collapses carousel-tagged posts into one card
(rendered with the "Carousel · 3 images" header), that singles render
as separate cards, and — bridging V050-005 — that editing the caption
on a carousel card fires `onUpdateCaption` with the **full sibling
postIds array**, not just the first sibling. This last case is the
integration check that the V050-005 carousel-sync rule reaches the
caption-edit helper.

## Test 3 — Pipeline platform detection (`pipeline-platform-detection.test.ts`)

**Bug it catches:** V041-HOTFIX-3BUGS Bug 3 — `processIdea` derived
`inferredPlatforms` from `settings.apiKeys` only, ignoring desktop
config.json creds, and treated `{ accessToken: '', igAccountId: '' }`
as configured because of naive object truthiness.

**Approach:** the existing `tests/lib/pipeline-processor.test.ts`
already has 2 cases for the inference call site (asserting what's
passed to `findNextAvailableSlot`). This test goes one layer further
and asserts the **output ScheduledPost shape** — the array
`accumulatedPosts` mutated by `processIdea` must contain a post with
`platforms: ['instagram']` when only `desktopCreds.hasInstagramToken`
is set. If a future refactor moved the inference correctly but
stopped wiring it into the `newPost.platforms` field, the unit tests
would still pass and this one would catch it. Four cases cover:
desktop-only IG inference, explicit `pipelinePlatforms` override
beating inference, no-creds → no post + "No platforms" log, and
empty-string apiKeys regression → no post.

## Out of scope (deliberate)

- **Full MainContent render test.** Rendering `MainContent` directly
  in jsdom requires mocking `useMashup` (huge surface), `useDesktopConfig`,
  and a dozen child components. The extracted `pushIdeaToStudio` helper
  proves the same wiring with one assertion. Worth revisiting only if
  another bug shows up at the `MainContent` boundary specifically.
- **`useIdeaProcessor` hook test for desktopCreds threading.** The
  V041-HOTFIX-3BUGS fix added `useDesktopConfig()` to the hook and
  threads `desktopCreds` into `ProcessIdeaDeps`. A hook test would
  catch a regression where someone removed the threading. Skipped for
  this round because: (a) the existing hook is small and grep-able,
  (b) the integration test in `pipeline-platform-detection.test.ts`
  catches the user-visible symptom of the regression (no post
  produced), and (c) hook testing requires `renderHook` from RTL
  which adds another layer of mocking. Add when something breaks.
- **Visual regression / snapshot.** Out of scope for "wiring bug"
  defense. Visuals are designer territory; the test layer here is
  about *behavior* on click/state-change.
- **Watch-mode pre-commit guard.** The pre-commit hook still runs
  the full suite. With 381 tests in 1.6 s that's still cheap; we'll
  revisit if it crosses ~5 s.
