## Outcome

### 1. `pipelinePostsPerDay` — user-configurable daily target

Previously the daemon's "schedule target met" check used a hardcoded
`targetPerDay = 2`. New `UserSettings.pipelinePostsPerDay` (optional,
defaults to 2 for back-compat) drives both the daemon's continuous-mode
decision and the Week Progress meter, so the two always agree on what
"filled" means.

Settings UI wiring is left for a follow-up — the field is read through
`settings.pipelinePostsPerDay ?? 2` everywhere it's consumed, so the
default-of-2 path preserves today's behavior until a slider lands.

### 2. `lib/weekly-fill.ts` — pure helper + `WeekFillStatus`

New module with `computeWeekFillStatus(posts, targetDays, postsPerDay, now?)`
returning:

```ts
interface WeekFillStatus {
  targetDays: number;
  postsPerDay: number;
  scheduledTotal: number;
  targetTotal: number;
  filled: boolean;
  percent: number;           // 0..100, clamped
  days: DayFill[];           // length === targetDays
}

interface DayFill {
  date: string;              // YYYY-MM-DD local
  dayLabel: 'Mon' | ... | 'Sun';
  scheduledCount: number;
  target: number;
  gap: number;               // max(0, target - count)
}
```

- Filters out `posted` / `failed` (matches `countFutureScheduledPosts`).
- Filters out past-time entries even on today.
- `now` is an argument (defaulting to `new Date()`) so tests pin time.
- Malformed date strings on posts are skipped, not thrown — settings are
  user-editable and may be half-filled.
- Percent caps at 100 when `scheduledTotal > targetTotal`.

Tests cover: empty, status filtering, past-time exclusion, out-of-window
exclusion, exact fill, over-fill clamp, non-default postsPerDay,
malformed date, and the `targetDays=0` edge case.

### 3. Daemon: reactive status + one-shot "week filled" log

`hooks/usePipelineDaemon.ts`:

- Added `useMemo` import + a `weekFillStatus` memo keyed on
  `settings.scheduledPosts`, `settings.pipelinePostsPerDay`, and
  `pipelineTargetDays`. Exposed on the hook return.
- Replaced `const targetPerDay = 2` with
  `const targetPerDay = latestPropsRef.current.settings.pipelinePostsPerDay ?? 2`.
- Inside the continuous-mode "target met" branch, added a
  `weekFilledPromptedThisRun` latch so the new `pipeline-week-filled` log
  event fires at most **once per run**, not on every sleep cycle. The
  daemon still continues to sleep and re-check — whether to generate for
  the next week is the user's call, not an auto-action.
- Threaded `weekFillStatus` through `hooks/usePipeline.ts` (composer),
  `components/MashupContext.tsx` (provider), and `types/mashup.ts`
  (`MashupContextType`).

### 4. `WeekProgressMeter` — the visible gap meter

`components/pipeline/WeekProgressMeter.tsx` reads `weekFillStatus` from
the context and renders a 7-cell bar grid with:

- per-day fill bar, colored by ratio
  (emerald when ≥100%, gold when ≥50%, muted gold >0, zinc for empty)
- per-day `count/target` label
- today-highlighted `dayLabel` in the brand gold
- a one-line summary: either "Week filled — …" or "N more posts to fill …"

Tailwind conventions match `BestTimesWidget` (`pt-2 border-t
border-[#c5a062]/15`, `label-overline`, mono numerals). Mounted in
`components/PipelinePanel.tsx` immediately above the Best Posting Times
widget per Designer's ordering guidance ("approval queue → week-ahead
progress → engagement heatmap").

### 5. Step label

Added `'pipeline-week-filled': 'Week Filled'` (plus the missing
`'pipeline-timeout': 'Timeout'`) to `STEP_LABELS` in `PipelinePanel.tsx`
so the log renders a friendly name for the new event.

## Scope calls I made

- **No UI control yet for `pipelinePostsPerDay`** — the field is read
  throughout but no slider was added. Reason: the ACs don't mention one,
  and the follow-up story exists implicitly (the meter now shows the
  number — adding a slider is the natural next step and can land with
  DES-003 or a settings polish task).
- **No auto-advance to "next week"** — interpretation of "prompt for
  next week" is deliberately passive. The daemon logs the event and the
  UI surfaces it; the user decides whether to keep pushing content. An
  automatic re-target slide would be a behavior change, not a prompt.
- **Heatmap is out of scope** — that's V030-DES-002, which ships
  separately.
