# CI-001 — Bun primary, npm fallback in `tauri-windows.yml`

## TL;DR

`.github/workflows/tauri-windows.yml` now installs deps and builds Next via
**Bun 1.3.12** by default. If the Bun step fails for any reason, a
`continue-on-error` + conditional fallback step runs the original
`npm ci && npm run build` path with a `::warning::` so the regression
surfaces in the Actions UI. Rust / Tauri / signing / artifact upload steps
are byte-identical to the previous workflow.

`bun.lock` is now committed (BETA-001 left it untracked pending this
decision).

## Why Bun primary

BETA-001 numbers (cold runs):

| Step | npm | Bun | Speedup |
|---|---|---|---|
| Install (cold) | ~30 s | 0.7 s | ~40× |
| `next build` | 2.8 s | 2.8 s | parity (Bun is a runner here) |
| `vitest run` | 540 ms | 540 ms | parity |

Install is the long pole on cold CI runs (no cache hit). Build/test are
runtime-bound on Next/vitest, not on the launcher. So the win is almost
entirely the install: cutting `npm ci` from ~30 s to ~1 s saves on every
fresh runner / cache-miss day.

## Why keep npm as fallback

Bun's failure modes:

- **Lockfile drift** — if `package.json` and `bun.lock` disagree under
  `--frozen-lockfile`, Bun aborts. npm will at least *try* with `npm ci`.
- **Native module quirks** — Bun handles optional deps (sharp, swc) well
  in our testing, but the bundled-app step still uses `npm install --force`
  (line 85 of the workflow) because we want platform-specific resolution
  semantics there. Keeping npm in the toolchain mirror means we can swap
  in/out without yanking another action.
- **Bun version ratchets** — pinning `1.3.12` means a future Bun breaking
  change doesn't strand CI, but a npm-side regression is also covered.
- **Genuinely opaque Bun bugs** — if Bun 1.3.x ships a bad release, we
  want CI to keep producing installers without an emergency PR.

The fallback is cheap: it adds ~5 lines of YAML and ~0 runtime when Bun
succeeds (the `if:` short-circuits).

## Diff (workflow only)

The relevant block, before:

```yaml
- name: Setup Node.js
  uses: actions/setup-node@v4
  with: { node-version: '22', cache: 'npm' }

- name: Install dependencies
  run: npm ci

- name: Build Next.js (standalone)
  run: npm run build
```

After:

```yaml
- name: Setup Node.js
  uses: actions/setup-node@v4
  with: { node-version: '22', cache: 'npm' }

- name: Setup Bun
  uses: oven-sh/setup-bun@v2
  with:
    bun-version: '1.3.12'

- name: Cache Bun install
  uses: actions/cache@v4
  with:
    path: ~/.bun/install/cache
    key: ${{ runner.os }}-bun-${{ hashFiles('bun.lock') }}
    restore-keys: |
      ${{ runner.os }}-bun-

- name: Build (Bun primary)
  id: bun_build
  continue-on-error: true
  shell: bash
  run: |
    bun install --frozen-lockfile
    bun run build

- name: Build (npm fallback)
  if: steps.bun_build.outcome == 'failure'
  shell: bash
  run: |
    echo "::warning::Bun build failed; falling back to npm. Investigate Bun step output."
    npm ci
    npm run build
```

## Why Node setup stays

Even with Bun primary, `actions/setup-node@v4` is still required because:

1. **`npm run build` chains `node scripts/check-bundle-size.mjs`.** The
   `&&` is interpreted by the shell, and the second command literally
   says `node ...`. Bun won't substitute itself there.
2. **The bundled-app native-deps step** (line ~85) runs
   `npm install --force --os=win32 --cpu=x64 @img/sharp-win32-x64
   @next/swc-win32-x64-msvc`. That's a targeted npm command we don't want
   to convert to Bun until we've tested its `--os`/`--cpu` semantics on
   bundled apps.
3. **The npm fallback path** needs it.

Net effect: Node is set up but its install cache only fills when the
fallback runs. Acceptable.

## Why `--frozen-lockfile`

Three reasons:

1. **Determinism.** CI must build from exactly the lock; no implicit
   resolution drift between local dev and runner.
2. **Drift detection.** If a developer bumps a dep in `package.json` but
   forgets `bun install`, CI fails fast with a clear error rather than
   silently picking up a different transitive tree.
3. **Cache stability.** `--frozen-lockfile` makes the cache key
   (`hashFiles('bun.lock')`) the source of truth. Without it, Bun could
   write to `bun.lock` mid-run and invalidate the assumption.

## Why we commit `bun.lock` now

BETA-001 listed this as a Hermes/team decision deferred from that ticket.
With Bun on the critical CI path, the lockfile MUST be committed:

- CI needs a deterministic resolution. Without `bun.lock` checked in,
  every CI run would re-resolve from `package.json` and write a different
  lock — defeating reproducibility.
- The 229 KB cost is one-shot at commit; subsequent diffs are small.
- Local devs running `bun install` will get the same tree as CI.

`package-lock.json` stays alongside it so the npm fallback also has a
deterministic path. Two lockfiles, two tools, one truth (because Bun
seeds its lock from `package-lock.json` semantics, the trees agree).

## Local pre-flight (Linux)

Before committing, exercised the exact commands the new Bun primary step
will run on the Windows runner — but on WSL Linux, against the freshly
regenerated `bun.lock`:

```
$ bun install --frozen-lockfile
bun install v1.3.12 (700fc117)
Checked 801 installs across 873 packages (no changes) [27.00ms]
```

```
$ bun run build
$ next build && node scripts/check-bundle-size.mjs
▲ Next.js 16.2.3 (Turbopack)
✓ Compiled successfully
... (TS, page gen)
PASS: all routes within 300 KB budget
```

Both pass. **No Bun bugs surfaced.** Things this rules out:

- Lockfile drift after the BUG-002 dep addition (`@tauri-apps/plugin-process`).
- Bun choking on `next build` for this codebase (BETA-001 already covered
  this, but re-verified post-dep-bump).
- The `&&` chain to `node scripts/check-bundle-size.mjs` working under
  Bun's process spawner (it does — Bun shells out to Node literally).

Things this does NOT rule out — Windows-runner-specific failure modes:

- `oven-sh/setup-bun@v2` install on `windows-latest` (Server 2022). Action
  documents support; trust but verify on first real run.
- Path resolution differences between WSL bash and Git-Bash on Windows
  for the `~/.bun/install/cache` cache key. The cache action handles `~`
  expansion per-platform; should be fine.
- Bun's behavior under Windows file locking when sharp is being patched
  in the bundled-app step (which runs AFTER the Bun build, separately,
  so they shouldn't interact, but flagging it).

If any of these surface on the next real tag push, the npm fallback
catches the build. Investigation steps if Bun fails on Windows-only:

1. Read the Bun step's stderr from the failed Actions run.
2. Diff against this local Linux output to localize: lockfile parse
   error vs install resolution vs build vs Node interop.
3. Try the same commands on a Windows machine (or WSL with Bun's Windows
   binary) to reproduce.
4. **If reproduced and root-caused as a Bun bug:** open
   https://github.com/oven-sh/bun/issues with: Bun version, Windows
   build (Server 2022), reproduction command, expected vs actual,
   minimal repro repo if possible. Title prefix `[bug]` per their template.
5. **If patch is feasible:** fork `oven-sh/bun`, add a regression test,
   submit PR. Not generally feasible (Bun is Zig); only attempt if the
   bug is in `@oven-sh/bun-types` or the GitHub Action wrapper.

Until the first failure, this section is forward-looking — no upstream
issue to file.

## Verification path

The workflow only runs on `tags: ['v*.*.*']` push or `workflow_dispatch`.
Two ways to verify the Bun path live:

1. **Wait for the next release.** The next bump (e.g. v0.2.2) tags-and-
   pushes through this new workflow naturally. Acceptable; no code
   changes to ship right now.
2. **Manual `workflow_dispatch` against v0.2.1.** Risky because the
   workflow's `upload` job overwrites release assets via
   `gh release upload --clobber`. If Bun produces bit-different bundles
   than npm, the v0.2.1 release assets would mutate under users.
   Recommend NOT doing this — the asset signature would also change,
   confusing anyone who already grabbed the v0.2.1 installer.

Defer to (1).

## Rollback plan

If the Bun path explodes on the first real run AND the fallback also
fails (most likely scenario: Bun's `--frozen-lockfile` succeeds but
`bun run build` does something unexpected), the cleanest revert is:

```bash
git revert <CI-001 commit>
```

Restores the npm-only workflow. The fallback already auto-recovers if
only Bun fails, so the manual revert is only needed if BOTH paths
break — which would mean an upstream npm registry / Node 22 issue, not
something this PR introduced.

## What I did NOT do (out of scope)

- **Bun-ify the bundled-app native-deps step** (`npm install --force --os
  --cpu`). Bun supports `--os` and `--cpu`-style targeting, but the exact
  behavior on optional/native deps in a NESTED node_modules with stripped
  Linux bindings hasn't been tested. Leave for a future ticket.
- **`bun test` in CI.** BETA-001 documented `vi.runAllTimersAsync` gaps.
  vitest still runs via `npm test` (or local dev). No CI test job exists
  in `tauri-windows.yml` to switch.
- **Cross-platform CI workflows.** Only `tauri-windows.yml` exists;
  there's no Linux/macOS workflow to update.
