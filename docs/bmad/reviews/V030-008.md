# V030-008 — AI-driven parameter pre-fill (pi.dev model selection)

## Summary
Replaced the deterministic rule-only suggestion path in the Studio Compare
tab's "Smart Suggestions" card with an **AI-driven model-selection reasoner**
backed by pi.dev (GLM via zai). The rule engine stays as the offline
fallback: any failure on the pi side (network, throw, empty, unparseable
JSON, missing fields) silently degrades back to the deterministic suggestion
so the user never sees an empty card.

The task is reframed in the pi prompt as **model SELECTION from a model
database**, not parameter FILLING from a form matrix: pi reads the full
Leonardo.AI v2 parameter catalog for all 6 production models and chooses
the model(s) whose capabilities fit the idea, then picks parameters each
chosen model actually accepts.

## Prompt shape fed to pi
```
You are the Leonardo.AI model-selection reasoner for MashupForge.
USER IDEA: <prompt>

MODEL DATABASE — authoritative catalog of what each model supports.
--- BEGIN MODEL DATABASE ---
<full parameter specs for 6 models, sample curls stripped>
--- END MODEL DATABASE ---

IN-APP ELIGIBILITY (only these ids may appear in modelIds):
<compact per-model capability JSON from LEONARDO_MODEL_PARAMS>

AVAILABLE STYLE NAMES: [...]

PRIOR WINNERS ON SIMILAR IDEAS: [...]

HOW TO REASON:
1. Identify what the idea needs (subject, medium, motion, aspect...)
2. Scan MODEL DATABASE — pick up to N best-fit ids from ELIGIBILITY
3. Choose parameters each chosen model accepts
4. In reasoning.overall, explain WHY this model fits this idea

HARD CONSTRAINTS: <enum-bounded>

Return ONLY a JSON object: { modelIds, aspectRatio, style, imageSize,
  quality?, promptEnhance, negativePrompt?, reasoning: {...} }
```

## Failure modes handled
| Failure | Behavior |
| --- | --- |
| `aiCall` throws (network / timeout / abort) | return fallback, `source: 'rules'` |
| pi returns empty string | return fallback, `source: 'rules'` |
| pi returns non-JSON prose | `extractJsonObjectFromLLM` returns `{}`, fallback |
| pi returns JSON with unknown model id | id dropped, `ai+rules` |
| pi returns JSON with unknown style | style dropped, `ai+rules` |
| pi returns JSON with invalid enum (imageSize, quality, promptEnhance) | field back-filled, `ai+rules` |
| pi returns JSON missing `reasoning` | reasons copied from fallback, `ai+rules` |

## Token cost note
The MODEL DATABASE string is ~4 KB (sample curls stripped). Per pi call
cost is non-trivial but acceptable — correctness > tokens for this story.
Pi only sees parameter options, never example values, so it cannot be
primed to copy specific prompts / style_ids / image IDs from the upstream
docs.
