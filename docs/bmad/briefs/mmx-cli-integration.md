## Context

MiniMax's Token Plan CLI (`mmx-cli`) is installed at `/home/maurice/.local/bin/mmx` (v1.0.12) and authenticated. It provides:

Maurice wants MMX CLI to become the **primary creative engine** — replacing Leonardo for images, adding music/video/speech, and using MiniMax M2.7 for coding. pi.dev stays as secondary/fallback.

## Architecture

### New module: `lib/mmx-client.ts`

Wrap `mmx` CLI as a typed Node.js client. Each method calls `mmx <resource> <command>` via `child_process.spawn` or `exec` and returns parsed output.

```typescript
export async function generateImage(prompt: string, opts?: ImageOptions): Promise<string[]>
// -> mmx image generate --prompt <prompt> [--aspect-ratio 16:9] [--out-dir ./public/gen/]
// returns URLs

export async function generateMusic(prompt: string, opts?: MusicOptions): Promise<string>
// -> mmx music generate --prompt <prompt> [--out ./public/gen/music.mp3]
// returns file path

export async function generateVideo(prompt: string): Promise<{taskId: string}>
// -> mmx video generate --prompt <prompt>
// returns task ID for polling

export async function synthesizeSpeech(text: string, opts?: SpeechOptions): Promise<string>
// -> mmx speech synthesize --text <text> [--out ./public/gen/speech.mp3]
// returns file path

export async function describeImage(imagePath: string): Promise<string>
// -> mmx vision describe <imagePath>
// returns description

export async function webSearch(query: string): Promise<SearchResult[]>
// -> mmx search query <query>
// returns structured results
```

### Image generation — both sources, user picks

Leonardo stays. MMX is an **additional source**. User chooses in settings or per-generation which provider to use. No automatic fallback — let the user decide.

```
generateImage(prompt, provider: 'mmx' | 'leonardo')
```

Both sources coexist in `lib/image-generator.ts` as named providers.

### Parameter sheets — both providers

The model spec files (`lib/model-specs/*.json`) are used to enhance prompts for **both** image providers:

- **For MMX:** Inject aspect ratio, style keywords, quality settings into the mmx prompt
- **For Leonardo:** Same specs are already Leonardo-native, but the enhanced prompt flow ensures consistency

Both providers get the same prompt enhancement treatment — the user picks which provider to use, both get the benefit of the parameter sheets.

```typescript
// For both providers:
const spec = loadModelSpec(styleKey); // e.g. "cyberpunk-landscape"
const enhancedPrompt = buildEnhancedPrompt(prompt, spec);
// MMX: mmx image generate --prompt "cyberpunk city at night, 16:9, cinematic lighting"
// Leonardo: same enhanced prompt -> leonardo API
```

### Weekly Sunday video recap

Every Sunday, a cron job triggers:
1. Query all posts published in the last 7 days
2. Extract topics/themes from captions + hashtags
3. Compose a video prompt: "This week's AI art highlights: [topics]..."
4. Generate video via `mmx video generate`
5. Generate music via `mmx music generate`
6. Generate voiceover via `mmx speech synthesize` — style matching AI art 4never avatar (warm, enthusiastic, slightly dramatic)
7. Combine into a recap clip and post to social

Cron: `0 10 * * 0` (Sunday 10:00)

## Bug Fixes (Post-Ready / Calendar)

### Fix 1: Rescheduled posts — stale schedule in Post Ready tab

When a post is rescheduled, the Post Ready tab still shows the old schedule time. The tab doesn't re-fetch or react to schedule changes. Fix: after any schedule update, refresh the Post Ready tab's data or invalidate the relevant cache key so the UI shows the new time.

### Fix 2: Deleted / deactivated posts in calendar grid

The calendar grid currently shows deleted or deactivated posts/schedules. Only failed schedules should appear. Fix: filter the calendar grid data source to exclude posts where `status === 'deleted'` or `status === 'deactivated'`.

### Fix 3: Calendar DnD — drag to bin to delete

Currently, dragging scheduled content in the calendar grid can reschedule it but there's no way to delete via DnD. Add a trash/drop zone at the bottom of the calendar view. User drags a card onto the bin and drops it → confirmation → post/schedule is deleted. Visual feedback: bin highlights on drag-over.

### Fix 4: Calendar view scheduled content is too hidden

Currently the user has to scroll back to the top and press a "view" image button to see scheduled content in the calendar grid. This is too many steps. Fix: make scheduled content more visible — either always visible by default, a persistent toggle button, or a subtle indicator that the content is there without requiring the extra click.

### Fix 5: Posting fails when scheduling (old content)

Posting fails when scheduling posts, at least for older content. New generated content hasn't been tested yet. This is a regression. Investigate: check the posting logic for scheduled posts — is the API call malformed? Auth token expired? Missing fields? Log the actual error and fix the root cause. Test with both old and new content after fix.
