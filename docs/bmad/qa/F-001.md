## Problem

`lib/pipeline-finalize.ts` `finalizePipelineImage()` built the `isPostReady`
property via a conditional spread:

```ts
...(markPostReady ? { isPostReady: true } : {})
```

On the reject path (`markPostReady=false`) the spread collapsed to `{}` — a
no-op. If the target image already had `isPostReady: true` from a prior
approval, rejecting it left the flag intact and the image stayed in the
Post Ready tab. This violated the documented contract in the function's
header: *"rejected images land in Gallery only, never Post Ready."*

## Fix

Replaced the conditional spread with an unconditional assignment at
`lib/pipeline-finalize.ts:87`:

```ts
isPostReady: markPostReady,
```

Reject path now always writes `false` and clears any stale `true` left
over from a prior approval. Approve path is unchanged (still writes
`true`). Header comment updated to reference F-001 and spell out the
previously-approved → rejected invariant.

## Tests

`tests/lib/pipeline-finalize.test.ts` — 12/12 pass (vitest).

Added the critical regression case:

```ts
it('F-001: clears isPostReady=true when rejecting a previously-approved image', async () => {
  const img = mkImg({ id: 'a', pipelinePending: true, isPostReady: true });

```

Also tightened two pre-existing reject-path tests from `expect(undefined)`
to `expect(false)` to match the new explicit-write contract.
