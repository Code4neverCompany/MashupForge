## What was done

Tested the module boundary split introduced by V030-001: `usePipeline` →
`usePipelineDaemon` + `useIdeaProcessor` composer. Since both hooks use React
state internally (untestable in `environment: 'node'` without jsdom), tests
target the pure functions and wiring patterns they implement.

**New files:**
- `lib/pipeline-daemon-utils.ts` — extracted `countFutureScheduledPosts` and
  `IdeaTimeoutError` from daemon for direct testability
- `tests/lib/pipeline-daemon.test.ts` — 17 tests for daemon pure functions
- `tests/hooks/useIdeaProcessor.test.ts` — 24 tests for processor deps-bag wiring

**Hook change:**
- `hooks/usePipelineDaemon.ts` — removed inline `countFutureScheduledPosts` and
  `IdeaTimeoutError`; imports them from `lib/pipeline-daemon-utils`

## Test cases (41 new)

### `tests/lib/pipeline-daemon.test.ts` — 17 tests

#### countFutureScheduledPosts [10 tests]
- Returns 0 for empty array
- Returns 0 for undefined
- Counts pending post (undefined status) within horizon
- Counts "scheduled" post within horizon
- Counts "pending_approval" post within horizon
- Excludes "posted" posts
- Excludes "failed" posts
- Excludes past posts
- Excludes posts beyond the horizon
- Correctly counts only in-window active posts from a mixed array

#### IdeaTimeoutError [3 tests]
- Is instanceof Error
- Has kind === "timeout"
- Has expected name and message

#### Per-idea timeout race [4 tests]
- Resolves with fast task before hard cap
- Rejects with IdeaTimeoutError when task exceeds hard cap
- Aborts the skip controller when timeout fires
- IdeaTimeoutError is distinguishable from SkipIdeaSignal by kind

### `tests/hooks/useIdeaProcessor.test.ts` — 24 tests

- Pre-aborted signal → SkipIdeaSignal thrown immediately

- Abort during waitForImages → SkipIdeaSignal propagates out
- Fresh AbortController per idea: spent ctrl from idea N does not affect idea N+1

#### writeCheckpoint delegation [2 tests]

- writeCheckpoint is called at multiple pipeline steps (not just once)

#### perIdeaImageIds accumulation [1 test]

## Architecture notes

- `countFutureScheduledPosts` and `IdeaTimeoutError` are now in
  `lib/pipeline-daemon-utils.ts` — importable from tests without pulling in
  React/IDB dependencies from the hook module.
- Hook-level `useIdeaProcessor.processIdea` tests are behavioural (via
  `processIdeaFn` from `lib/pipeline-processor`) because React hooks require
  jsdom. Adding jsdom/`@testing-library/react` would unlock `renderHook` tests
  for the full hook — consider for a future QA gate if signal wiring needs
  coverage at the hook call boundary.
- tsc clean: `npx tsc --noEmit` → no errors.
