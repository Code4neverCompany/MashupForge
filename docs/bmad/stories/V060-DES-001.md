# V060-DES-001 — Inline schedule calendar for Post Ready cards

**Status:** spec ready (Designer-mandate spec only — implementation is complex, dispatch separately)
**Classification:** complex
**Replaces:** the raw `<input type="date">` + `<TimePicker24>` + Schedule button cluster on each Post Ready card (`components/MainContent.tsx:3977-4009` for single image, `:3705-3741` for carousel)
**Reuses:** `lib/smartScheduler.ts` (engagement + slot scoring), `components/WeekHeatmap.tsx` (tint + tooltip primitives), `hooks/useSmartScheduler.ts` (lazy IG fetch), `lib/ui-tokens.ts` (button + pill recipes)

---

## Why

Today, scheduling a single Post Ready card requires the user to:

1. Type a date into a browser-native date input.
2. Spin a 24-hour time picker.
3. Click Schedule.

There's no signal about *when* is a good time. The Smart Schedule modal exists for batch scheduling (`SmartScheduleModal`, opened from the Post Ready header) but a single-card user has to either guess or open the full modal for a one-off. Maurice approved replacing the per-card date+time inputs with an inline calendar that:

- Tints each day cell by engagement strength (heatmap).
- Lets the user click a day → pick a time, with the best hours pre-highlighted.
- Offers an Auto-Schedule button that picks the next best slot in one click.
- Stays compact: opens *under* the card without forcing the page to scroll.

## A — UX flow

### A.1 Trigger

The Schedule button on each card stays. Clicking it now toggles the inline calendar (`<InlineScheduleCalendar>`) open *below the card body*, replacing the current date/time/schedule row. The calendar is collapsible — clicking Schedule again, or pressing Escape, closes it.

State lifts to the card scope. The cluster lives inside the existing right-column footer (`MainContent.tsx:3976` for single, `:3717` for carousel).

**Per-card open state:** one card open at a time. Opening a second card's calendar collapses the first — keeps the page short and prevents two open calendars from competing for engagement-fetch races.

**Total height (closed → fully open with time picker):** target ≤ 360px. Month grid 6 rows × 38px = 228 + 56px header + 76px time strip + padding ≈ 360. The card itself is ~420–520px tall in the existing 2-col grid; opening the calendar pushes the *next* card down but does not force the calendar's own contents to scroll. The calendar is `overflow-visible` — never wraps in its own scrollbar.

### A.3 Heatmap tints (month grid)

For each rendered day cell, score = `findBestSlots` over that single day's 18 picker hours (06:00–23:00) with `count: 1` from `lib/smartScheduler.ts:329`. Take the top score; map to a tint band:

| Score range | Tint | Reuse |
|---|---|---|
| ≥ 75 | `bg-emerald-500/15 border-emerald-500/30` | matches existing "best" band in `WeekHeatmap` |
| 60–74 | `bg-emerald-500/8 border-emerald-500/15` | "good" |
| 40–59 | `bg-[#c5a062]/8 border-[#c5a062]/15` | "ok" — gold tint to keep brand presence |
| 20–39 | `bg-zinc-800/30 border-zinc-800/60` | "weak" |
| < 20 | `bg-zinc-900/40 border-zinc-900` | "avoid" |

**Override states** (composited on top of tint):

- Past day: `opacity-40 cursor-not-allowed`, no click handler.
- Today: gold ring `ring-1 ring-[#c5a062]/60` over whatever tint applies.
- Selected day: gold solid ring `ring-2 ring-[#c5a062]` + slightly raised tint.
- Day with existing scheduled posts for *this image* anywhere: amber `border-amber-500/50` + amber dot top-right.
- Day with existing scheduled posts for *any* image: small zinc-400 dot, no border change.

**Tooltip on hover** (300ms delay, reuse `WeekHeatmap` tooltip pattern): "Thu Apr 16 · best at 19:00 (score 78) · 1 other post scheduled." Positioned above the cell with `pointer-events-none`.

### A.4 Day click → time picker

Clicking a non-past day reveals the time strip **below** the grid (animates height + opacity 150ms; existing grid does not reflow). The strip:

- 18 cells, hours 06–23, horizontal scroll only if viewport < 480px (it shouldn't on desktop).
- Each cell tinted using the same emerald/gold/zinc bands above, but scored per `(day, hour)` from `scoreSlotDetailed(date, hour, engagement)` (`lib/smartScheduler.ts:195`).
- Top 3 hours for the selected day get a gold star icon `★` above the hour number.
- Hours that already have a post scheduled (any image) for the selected day are crossed out with a thin strikethrough and `opacity-50`. Click is allowed but Confirm shows a "Conflict at HH:00 — schedule anyway?" pill above the button.
- Hours that would put us *over the daily cap* for the selected card's platforms (read `settings.pipelineDailyCaps`) get the same crossed-out treatment with tooltip "Daily cap for {platform} reached for this day."

**Default selection** when a day is clicked: highest-scoring non-conflicting hour. If all are conflicting, no preselection — Confirm stays disabled.

**Confirm button** (gold solid, right-aligned): writes one `ScheduledPost` (or N for a carousel — reuse `scheduleCarousel` flow at `MainContent.tsx:640`) and closes the calendar. Status: `'scheduled'`. Updates `settings.scheduledPosts` via `updateSettings`.

### A.5 Auto-Schedule button

Top-right of calendar header, beside the source label. Single button, gold solid, label `⚡ Auto-Schedule`.

On click:

1. Call `findBestSlot(existingPosts, engagement, { platforms: card.platforms, caps: settings.pipelineDailyCaps })` (`smartScheduler.ts:403`).
2. If a slot is found within the next 14 days:
   - Animate the picked day cell to "selected" state (gold ring + scale 1.05 → 1.0 over 200ms).
   - Open the time strip with the picked hour preselected.
   - Wait 250ms for the user to see what was chosen.
   - Fire the same Confirm path automatically.
   - Toast under the card: "Scheduled for {DayName} {Date} at {HH:MM} (best slot: score {score}, {reason})."
3. If no slot is found (all caps full for 14 days): toast amber "No open slot in next 14 days — daily caps reached. Schedule manually or raise caps in Settings."

**Loading state:** if `useSmartScheduler.loading === true` (Instagram engagement still fetching), the Auto-Schedule button shows a spinner and reads "Analyzing…" — same pattern as the existing Smart Schedule button (`MainContent.tsx:2776-2786`).

### A.6 Engagement source line

Right-aligned in the calendar header, small (10px uppercase tracking-wide):

- `Source: IG insights · {samples} posts` (when `engagement.source === 'instagram'`)
- `Source: Research defaults` (fallback)
- `Source: loading…` (during `useSmartScheduler.loading`)

The line clarifies *why* the heatmap shades look the way they do. Cached engagement (24h TTL) means most opens render instantly with no network spinner.

### A.7 Empty / error states

- **No engagement at all** (first run, no IG creds, fetch failed): use `DEFAULT_HOUR_WEIGHTS` + `DEFAULT_DAY_MULTIPLIERS` from `smartScheduler.ts`. Heatmap renders but shading is uniform-ish; source line says "Research defaults." No banner — silent fallback.
- **All future days fully booked** by daily caps: month grid renders all-zinc; Confirm is disabled with tooltip "All upcoming slots full for {platforms.join(', ')}."
- **No platforms selected on the card**: calendar still opens but Confirm is disabled with helper text under the time strip "Select a platform above to schedule."

## B — Component shape

### B.1 New file

`components/InlineScheduleCalendar.tsx` (~280 LOC target)

```tsx
interface InlineScheduleCalendarProps {
  // Card identity
  imageId: string;                    // for single-image cards
  carouselGroupId?: string;           // alt: schedules whole carousel
  carouselImages?: GeneratedImage[];  // required if carouselGroupId set

  // Scheduling inputs
  selectedPlatforms: string[];        // from card's platform pills
  defaultCaption: string;             // for single-image schedule

  // Lifecycle
  isOpen: boolean;
  onClose: () => void;
  onScheduled: (posts: ScheduledPost[]) => void; // called after settings update
}
```

Internally:

- Mounts `useSmartScheduler` with `count: 1` and the card's platforms.
- Memoizes a `Map<dateStr, dayBestScore>` over the visible month + `Map<"date:hour", hourScore>` for the selected day.
- Reads `settings.scheduledPosts` from `useMashup()` to render conflict markers.
- On Confirm, calls `updateSettings(prev => ({ scheduledPosts: [...prev.scheduledPosts, ...newPosts] }))` and `onScheduled(newPosts)`.

### B.2 Edits to MainContent.tsx

- Single-image card (`:3976-4009`): replace the date+time grid + Schedule button with a single `<button>Schedule</button>` that toggles the calendar; mount `<InlineScheduleCalendar>` immediately below the right column when open.
- Carousel card (`:3717-3741`): same pattern, passing `carouselGroupId` + `carouselImages`.
- Add `openCalendarFor: string | null` state in `MainContent` so only one card's calendar is open at a time. Setter checks the previous value and closes it before opening the new one.

### B.3 Reused without modification

- `lib/smartScheduler.ts` — `findBestSlot`, `findBestSlots`, `scoreSlotDetailed`, `computeWeekScores`, `DEFAULT_*` constants.
- `hooks/useSmartScheduler.ts` — initialized once per calendar instance.
- `components/WeekHeatmap.tsx` — extract `<HeatmapCell>` and `<HeatmapTooltip>` into a small shared module if not already exported (verify before implementing — a small refactor may be needed; classify as routine).
- `lib/ui-tokens.ts` — `btn-gold-sm`, `btn-blue-sm`, pill recipes.

## D — Out of scope (PROP separately if requested)

- **Drag-drop from Post Ready into the calendar.** The existing Calendar view (in `view === 'post-ready'` Calendar mode, `MainContent.tsx:3120+`) supports drag-drop. The inline calendar is click-only — adding drag would conflict with the page-level scroll on touch devices and balloon scope.
- **Recurrence / "schedule weekly" UI.** Out of scope; would change `ScheduledPost` shape.
- **Multi-day range select.** Out of scope; the Auto-Schedule button covers the "many slots" use case for single posts.
- **Time zone picker.** All scheduling stays in user-local time per existing convention. Adding TZ awareness is its own initiative.
- **Editing an already-scheduled post via this calendar.** This calendar creates new schedules only. The existing edit flow (Calendar view popover, `MainContent.tsx:2911+`) handles edits.

## E — Implementation hints (for the next dev)

1. Build `InlineScheduleCalendar` as a leaf component first with mocked engagement, then wire `useSmartScheduler`.
2. The month grid is the trickiest piece; consider building from `date-fns` `eachDayOfInterval` over `startOfMonth → endOfMonth` padded to whole-week boundaries. The repo already uses `date-fns` (verify in package.json before assuming).
3. For animation, prefer a simple `max-height` + `opacity` CSS transition over a library — the existing codebase keeps animations minimal and bundle-free.
4. The two cards (single + carousel) share enough scheduling logic that lifting `openCalendarFor` to `MainContent` and passing it down both rendering branches is cleaner than duplicating state.
5. Verify `WeekHeatmap` exports the cell + tooltip primitives before reaching for them. If not, a small extraction PR is routine and worth doing first.
