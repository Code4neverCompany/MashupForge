## 0. What already exists (important)

An `ApprovalQueue` component **already ships** at `components/pipeline/ApprovalQueue.tsx` and is mounted inside `components/PipelinePanel.tsx:625`. It is a *section* inside the Pipeline tab, not a top-level view. What it already does:

- Grid of cards, one per `pending_approval` ScheduledPost
- Checkbox selection + bulk approve / reject
- Filter pills by idea topic / model / platform
- Per-card Approve / Reject buttons
- Thumbnail + caption + platforms + date/time + model name

What it does **not** do (the gap this task fills):

- Not a top-level view — buried in Pipeline tab
- No Focus (linear keyboard) mode
- No traffic-light or confidence-star quality signals
- No reject-with-reason dropdown
- No keyboard navigation
- Uses amber chrome (`bg-amber-500/10`) — off-brand for status-indigo (`pending_approval`'s color in the calendar palette at `MainContent.tsx:455-460`)
- Uses plain `<img>`, not `LazyImg` (perf regression)

**Developer direction:** extend + relocate the existing file, don't rebuild from scratch. Keep its filters, its image-indexing memos, its bulk-select plumbing — graft the new affordances on top.

## 7. `TrafficLightStrip` — 5-dot quality row

### 7.1 Checks

```ts
export type CheckKey = 'captionLength' | 'hashtagCount' | 'scheduleConflict' | 'imageQuality' | 'brandSafety';

export interface TrafficLightResult {
  checks: {
    key: CheckKey;
    label: string;
    status: 'pass' | 'fail' | 'skip';
    reason?: string;           // only set when status === 'fail'
  }[];
  score: number;               // count of pass divided by non-skip, 0..1
}
```

Pure function `computeChecks(post, image, allScheduled): TrafficLightResult` in `lib/approval-checks.ts`. Called once per card render, memoized on post.id.

Per-check rules:

| Key | Pass condition | Fail reason example |
|---|---|---|
| captionLength | `caption.length` within platform max (IG 2200, Pinterest 500, Twitter 280, Discord 2000) — use the MIN of the post's selected platforms | "Caption is 2,341 chars; Twitter limit 280" |
| hashtagCount | `hashtags.length >= 8 && <= 30` (IG sweet-spot) | "Only 4 hashtags — add ≥4 more" |
| scheduleConflict | no other `scheduled` / `pending_approval` post within ±30 min on same platform | "IG post at 18:45 conflicts" |
| imageQuality | `image.status !== 'error'` AND `image.url` present | "Image failed to generate" |
| brandSafety | skip for now (`status: 'skip'`) until a safety model is wired | — |

`computeChecks` is pure and fast — fine to recompute on every render; no async.

### 7.2 Visual

```tsx
<div className="flex items-center gap-1.5">
  {checks.map((c) => (
    <span
      key={c.key}
      title={c.status === 'fail' ? `${c.label}: ${c.reason}` : c.label}
      aria-label={`${c.label}: ${c.status}`}
      className={`w-1.5 h-1.5 rounded-full ${
        c.status === 'pass' ? 'bg-emerald-500'
        : c.status === 'fail' ? 'bg-red-500'
        : 'bg-zinc-700'
      }`}
    />
  ))}
  {hasFailures && (
    <button
      onClick={toggleExpandChecks}
      className="ml-1 text-[10px] text-red-400 hover:text-red-300"
    >
      {failCount} issue{failCount === 1 ? '' : 's'}
    </button>
  )}
</div>
```

Clicking the issue pill expands a small inline list below:
```
• Caption too long (280/280 for Twitter)
• Only 4 hashtags
```
Expansion lives in the card's DOM, not a modal — keeps the flow.

### 7.3 Confidence stars

Top-right of the card. Derived from `checks.score`:

```ts
const stars = Math.round(checks.score * 5);
```

Rendered as 5 lucide `<Star>` icons, filled `[#c5a062]` up to `stars`, empty `text-zinc-700` after. Used as a visual sort signal in grid mode — scan for the ★★★★★ rows first.

## 10. `RejectWithReasonMenu` — inline dropdown

### 10.1 Reason shape

```ts
export type RejectionReason = 'off-brand' | 'wrong-platform' | 'bad-caption' | 'image-quality' | 'other';

interface RejectionReasonEntry {
  reason: RejectionReason;
  label: string;
  freeText?: string;         // only for 'other'
}
```

Extension to `ScheduledPost` (flagged in §13):
```ts

```

### 10.2 Placement + trigger

Rendered as an inline dropdown anchored to the Reject button — **not a modal, no page jump**.

Reusing `KebabMenu` from DESIGN-003 with a custom items array:

```tsx
<KebabMenu
  ariaLabel="Reject reason"
  items={[
    { kind: 'label', text: 'Reject because…' },
    { kind: 'item', id: 'off-brand', label: 'Off-brand', onSelect: () => reject('off-brand') },
    { kind: 'item', id: 'wrong-platform', label: 'Wrong platform', onSelect: () => reject('wrong-platform') },
    { kind: 'item', id: 'bad-caption', label: 'Bad caption', onSelect: () => reject('bad-caption') },
    { kind: 'item', id: 'image-quality', label: 'Image quality', onSelect: () => reject('image-quality') },
    { kind: 'separator' },
    { kind: 'item', id: 'other', label: 'Other…', onSelect: () => setFreeTextOpen(true) },
    { kind: 'item', id: 'skip', label: 'Reject without reason', destructive: true, onSelect: () => reject(undefined) },
  ]}
  triggerClassName="... red reject button ..."
/>
```

### 10.3 "Other…" free-text inline

When user picks Other, the dropdown content swaps to:

```
┌──────────────────────────────────────┐
│ Reject because…                      │
│ ┌──────────────────────────────────┐ │
│ │ Tell us why…                     │ │  ← textarea, 2 rows, max 240 chars
│ │                                  │ │
│ └──────────────────────────────────┘ │
│                  [Cancel]  [Reject]  │
└──────────────────────────────────────┘
```

`Cancel` → returns to the reason list. `Reject` → submits with `{ reason: 'other', freeText: '...' }` and closes.

### 10.4 Keyboard

- `Esc` → skip reason, reject without one (skip = explicitly choosing no reason)
- Wait — actually, `Esc` inside a menu closes the menu without acting. For "reject without reason" use the explicit destructive item instead. This keeps Esc meaning "nevermind" across the whole app, which is what users expect.
- `Enter` on a reason item → reject with that reason + advance
