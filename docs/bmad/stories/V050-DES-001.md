# V050-DES-001 — Onboarding wizard + error recovery UI

**Status:** spec
**Author:** Designer
**Classification:** complex (new context flag, cross-cutting failure flows, schema additions)
**Owns:** first-run wizard, failed-post recovery affordances across calendar/post-ready/pipeline log, failure dashboard hooks
**Depends on:** V040-DES-002 Part A (health dot — referenced, not duplicated)
**Out of scope:** auto-poster retry policy at the network layer (separate PROP), credential validation worker

---

## 0 · Why this exists

Two trust holes ranked #1 and #2 in the v0.4 brainstorm:

1. **Onboarding is invisible.** First-run user lands on Studio with empty state. To get a real post live they must: open Settings → enter API keys → configure niches → use sidebar to brainstorm → add ideas → switch to Pipeline → enable + start. Six tabs, zero hand-holding. Most users won't get past step 2.
2. **Errors are silent.** Failed posts surface as a red square in the calendar with no explanation, no retry, no recovery flow. The user discovers a posting failure days later when they notice a missing post on Instagram. This kills trust in the automation.

This spec ships both as one cohesive "first 5 minutes + every minute after" UX layer.

# Part A — Onboarding wizard

## A.1 · First-run trigger

**Storage flag:** `localStorage` key `mashup.onboarded` — boolean. Set to `true` when the user completes step 3 OR explicitly skips with intent. Persists per-browser.

**Schema flag (preferred over localStorage):** `UserSettings.onboardingCompletedAt?: number` — UTC ms timestamp. Survives browser change, lets us A/B onboarding revisions, and lets the daily digest acknowledge new users for their first week. **Schema change → flagged complex (PROP).** Until then, localStorage is acceptable.

**Trigger logic** (root layout level):
```
if (!settings.onboardingCompletedAt && !localStorage['mashup.onboarded']) {
  show <OnboardingWizard />
}
```

The wizard mounts as a focus-trapped modal over the rest of the app. The user sees the Studio behind it dimmed at 60% opacity — not hidden. They get to glimpse where they're going.

**Container:** `max-w-[640px] w-[calc(100vw-2rem)] bg-zinc-950/95 backdrop-blur-xl border border-[#c5a062]/30 rounded-2xl shadow-2xl`. Vertical auto-height to step content; min-height 540px so the body doesn't jump between steps.

**Backdrop:** `bg-black/70 backdrop-blur-sm`. Click-outside does NOT close — only [Skip for now] dismisses. Escape key only fires the skip confirmation (§A.6), not direct dismiss.

**Header:**
- Logo wordmark left (`AETHER SANS`, gold `#c5a062`)
- Skip button right — text only, `text-zinc-500 hover:text-zinc-300`

**Stepper:** 3 dots horizontal, separated by short connecting lines. Dots:
- Past steps: `bg-emerald-500` filled (no hover)

Connectors: `bg-emerald-500` for completed, `bg-zinc-800` for pending.

**Footer:**

- On step 3: Next becomes "Run pipeline" (loading state during execution)

**A11y:**

- Initial focus on the first interactive element of the body
- Focus trap inside the modal until dismissed
- Stepper has `role="progressbar" aria-valuenow="2" aria-valuemax="3"`

# Part B — Error recovery UI

## B.1 · Three layers

Failure visibility runs at three altitudes; each shows the user the same fact at a different level of detail:

| Layer | Where | What you see |
|---|---|---|
| 1. Glance | Sidebar health dot · Daily Digest tile · sidebar Approval badge | "Something is broken" |
| 2. Card | Failed slot in calendar · failed Post Ready card · failed pipeline log entry | "Which post, which platform, which error" |
| 3. Recover | Inline action row on the card · popover on the slot | "Retry · Reschedule · Edit · Discard" |

A user who isn't actively looking learns about the failure via Layer 1. A user reviewing posts learns via Layer 2. A user actively recovering uses Layer 3. None of the three duplicate the others — they nest.

## B.2 · Layer 1 — Glance

### B.2.1 · Health dot (already spec'd)

V040-DES-002 Part A defines the four-state dot at the bottom of the sidebar. **This spec adds two behaviors:**

1. **Click-to-recover deep link.** When dot state is `Broken`, clicking the dot opens the Settings modal AND immediately scrolls to that platform's section AND highlights the first invalid field. Direct path from "I see red" to "I'm fixing it" in one click.
2. **Hover preview of recent failures.** Tooltip extends with last 1-line failure: "Last fail: Token expired · 4h ago". Pulls from the same `scheduledPosts.filter(status === 'failed')` derivation V040-DES-002 used.

### B.2.2 · Daily Digest "Needs attention" tile

V040-DES-002 Part B's 4-tile grid: when `failedCount > 0`, the **Pending** tile transforms into a **Needs attention** tile:

```
┌──────────────┐
│   Needs      │
│  attention   │
│              │
│  3 failed    │
│  posts       │
│  [Review ▸]  │
└──────────────┘
```

- Background: `bg-red-500/10 border-red-500/30`
- Title: `text-red-300`
- Number: large, `text-red-200`
- Button: opens a new "Failures" view (or filters Approval to failed only — see §B.5)
- Pending count moves to a smaller subtitle line if it's also nonzero: "+ 2 awaiting approval"

If `failedCount === 0`, tile reverts to Pending (V040-DES-002 default).

### B.2.3 · Sidebar / nav badge

If user has 1+ failed posts: small red dot badge on whichever tab hosts the failures view. Until top-level Approval ViewType ships, the dot lives on the **Pipeline** tab (since Failures live in PipelinePanel today).

## B.3 · Layer 2 + 3 — Calendar slot recovery

### B.3.1 · Failed slot visual (existing + reinforcement)

`MainContent.tsx:539` already returns `bg-red-500/80 border-red-400/60` for failed posts. **Add:**
- Subtle `animate-pulse` on the border only (NOT the fill — keep it readable)
- Red `AlertTriangle` icon top-right corner of the slot
- Reduced-motion: skip the pulse, show a static red border-dashed instead

### B.3.2 · Recovery popover (NEW)

Click a failed slot → popover anchored to the slot (auto-flip per DESIGN-003 KebabMenu logic):

```
┌──────────────────────────────────────────────────┐
│ ⚠ Post failed                                    │
│                                                  │
│ Instagram returned 401: token expired            │
│ at Fri 18:00 (4 hours ago)                       │
│                                                  │
│ Caption: "Marvel meets Cyberpunk in Night City…" │
│                                                  │
│ ┌──────────┬─────────────┬───────────┬─────────┐ │
│ │ ↻ Retry  │ ⏰ Reschedule│ ✎ Edit    │ ✕ Drop  │ │
│ └──────────┴─────────────┴───────────┴─────────┘ │
│                                                  │
│ [Open in Post Ready →]   [Fix credentials →]     │
└──────────────────────────────────────────────────┘
```

Container: `w-80 bg-zinc-950/95 border border-red-500/30 rounded-xl shadow-2xl p-4 space-y-3`.

**Header row:** `AlertTriangle` red + bold "Post failed" in red-300.

**Error line:** Plain English. Maps the raw error string through a translator:
- 401/403 → "Token expired" + suggest credential refresh
- 429 → "Rate-limited — Instagram is throttling us" + suggest "Try in 30 min"
- 5xx → "Platform unreachable — service may be down" + suggest auto-retry
- Timeout → "Request took too long" + suggest retry
- Unknown → raw error text + "Report this if it keeps happening"

**Caption preview:** 1-line truncated so the user can ID the post without leaving the popover.

**Action row** — 4 buttons:
- **Retry** (gold primary): re-queue the post for immediate posting; popover shows spinner; on success closes + slot turns emerald
- **Reschedule** (zinc secondary): closes popover, opens schedule picker pre-loaded with this post (existing UI pattern)
- **Edit** (zinc secondary): closes popover, opens caption editor inline OR routes to Post Ready focused on this card
- **Drop** (red destructive): removes the ScheduledPost entirely; 6s undo toast at bottom of viewport

**Footer links:**
- "Open in Post Ready →" — routes to Post Ready view scrolled to the failed card (full context)
- "Fix credentials →" — opens Settings to that platform (only shown if error is auth-class)

**Dismiss:** Click outside, Escape, or any action button closes the popover.

**A11y:**
- `role="dialog" aria-labelledby="failed-popover-title"` (since it's a focused decision interface, not a tooltip)
- Initial focus on Retry button (most common action)
- Focus trap inside while open

Card already has the red status badge (DESIGN-001 spec). **Add a persistent error banner** across the bottom of the card:

```
┌──────────────────────────────────────────────────┐
│              [ image ]                           │
│                                                  │
│ Caption…                                         │
│ Fri 18:00 · Instagram                            │
│                                                  │
│ ─────────────────────────────────────────────    │
│ ⚠ Token expired · 4h ago                         │
│ [Retry] [Reschedule] [Discard]                   │
└──────────────────────────────────────────────────┘
```

- Banner: `bg-red-950/40 border-l-2 border-red-500 px-3 py-2 text-xs`
- Error line: same translator as §B.3.2
- 3 actions: Retry · Reschedule · Discard (Edit isn't here because the user is already in Post Ready and the caption is editable in the card body)
- Reused error translator lives in `lib/error-translate.ts` (NEW pure file, routine)

The banner is **always visible** on failed cards — not hover-only. Failures demand attention.

## B.5 · Pipeline log error entry

Pipeline log currently shows generic timestamp + step + message. Failed entries get richer treatment:

```
14:32  pipeline-error  ⚠ Image generation failed
                       Leonardo returned 503 — service unavailable
                       Idea: "Marvel × Cyberpunk: Stark…"
                       [Retry idea] [Skip idea]
```

- Container: `bg-red-950/30 border-l-2 border-red-500 pl-3 py-2`
- Title row: red `AlertTriangle` + step label + bold one-line summary
- Detail line: full error
- Idea reference if applicable (lets user trace back)
- Inline retry actions

Sticky behavior: failed entries in the log do NOT auto-clear with `clearPipelineLog` until the failure is resolved (retried, dismissed, or the underlying post discarded). Visible debt.

## B.6 · Failures view (or filter)

Two implementation paths — pick based on PROP outcome:

**Option A (preferred):** Add `'failures'` to the Failures-only filter on Approval/Post Ready (no new ViewType, just a filter pill: `[Failed only · 3]`)

**Option B:** New top-level `'failures'` ViewType — only appears in the sidebar when `failedCount > 0`. Disappears when 0. Self-cleaning navigation.

Designer recommendation: **A** — fewer schema changes, hooks into the existing approval/post-ready surfaces. Activate via a query param like `?filter=failed` so deep links from the digest tile work.

## B.7 · Auto-recovery rules (for the auto-poster, not designer-implemented)

The UI assumes the auto-poster does the following — flagging here so the design and the system stay aligned:

- **1st failure:** silent retry once after 30s (no UI noise)
- **2nd failure:** mark `status: failed`, surface in all three layers
- **3rd failure on same platform within 24h:** auto-pause that platform (set health dot to `Broken` even if last attempt was 401), notify user via digest tile

These rules are flagged complex (auto-poster logic, PROP). The UI is designed assuming they exist — graceful behavior even if they don't.

## Cross-cutting decisions

### Tone

**Onboarding:** confident and brief. The user is competent; we're a tool, not a guide. No emoji-heavy copy. No marketing-speak. Action-oriented headings ("Where do you post?" not "Welcome to MashupForge!").

**Errors:** plain English, no blame. "Token expired" not "Authentication error 401 invalid_grant". Suggest the next action; don't dump stack traces. Reserve technical detail for an expandable "Show details" link.

### Color (consistent with prior specs)

- **Onboarding success:** emerald (`bg-emerald-500`)
- **Onboarding interactive:** gold (`#c5a062`)
- **Setup unfinished pill:** amber (`bg-amber-500/15`) — between go and stop
- **Failure surfaces:** red (`bg-red-500`, `bg-red-950/40` for banners)
- **Health dot states:** unchanged from V040-DES-002 (emerald/amber/red/zinc)

### When to interrupt vs. when to wait

- **Onboarding:** interrupts the user once on first run. Skippable. Never re-interrupts unless the user explicitly opts back in via the pill.
- **Failures:** never interrupt with modals. All failure surfaces are passive (color, icon, badge, banner) and require user-initiated action to enter recovery.

A failed post should not push a toast across the screen — that breaks flow. The dot pulses, the badge appears, the digest reflects it. The user comes to the failure on their own time.
