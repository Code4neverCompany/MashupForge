# V030-001 — Extract usePipelineDaemon + useIdeaProcessor from god hook

**Status:** in_progress
**Classification:** complex
**Owner:** Developer (autonomous design+execute, authorized by Hermes 2026-04-18)

## 1. Motivation

`hooks/usePipeline.ts` is 933 lines with 8 `useEffect → useRef` sync pairs. Every stale-closure bug we've shipped in v0.2.x (skip-idea bleed-through, slot-collision on mid-run edits, continuous-interval not picking up changes) traces back to the same root cause: a long-running async function closes over props/state at call time, and we paper over it with ref mirrors that get out of sync at their own pace.

This refactor splits the god hook into two domain-focused hooks — **usePipelineDaemon** (run-state + queue + checkpoint + outer loop) and **useIdeaProcessor** (per-idea steps) — plus a thin `usePipeline` composer that keeps the public `MashupContextType` surface unchanged.

## 2. Constraints (user directives, 2026-04-18)

1. `usePipelineDaemon` exposes state setters + observable values to `useIdeaProcessor`.
2. `useIdeaProcessor` accepts a deps bag from the daemon — same pattern as `lib/resume-checkpoint.ts::applyResumeCheckpoint`.
3. Functional `setState` everywhere — no refs, no experimental APIs. **One pragmatic exception** documented in §4.3.
4. Checkpoint ownership: daemon-only (single source of truth).
5. `MashupContextType` shape stays the same — `usePipeline` keeps returning the same 20-field surface, internal wiring changes only.

## 3. Module boundaries

### 3.1 `lib/pipeline-processor.ts` (already extracted, untracked)
Pure per-idea processor. No React. Accepts `ProcessIdeaDeps`. Unit-testable without jsdom.

### 3.2 `lib/pipeline-runner.ts` (already extracted, untracked)
`withPipelineRunning<T>(setRunning, fn)` helper that guarantees `setRunning(false)` on every exit path — the try/finally gap from v0.2.3 QA.

### 3.3 `hooks/useIdeaProcessor.ts` (new)
Hook-layer wrapper around `lib/pipeline-processor.ts`. Builds the `ProcessIdeaDeps` bag from daemon-owned state + caller-provided primitives (generateComparison, saveImage, etc.) and returns a bound `processIdea`. Holds **no state of its own**.

### 3.4 `hooks/usePipelineDaemon.ts` (new)
Owns:
- `pipelineEnabled`, `pipelineRunning`, `pipelineQueue`, `pipelineProgress`, `pipelineLog`
- `pipelineDelay`, `pipelineContinuous`, `pipelineInterval`, `pipelineTargetDays`
- `pendingResume` + checkpoint save/load/clear
- `startPipeline` / `stopPipeline` / `skipCurrentIdea` / `acceptResume` / `dismissResume`
- Outer-cycle loop (pi pre-check, engagement fetch, per-cycle idea queue, continuous-mode sleep)
- localStorage persistence of pipeline config + log

### 3.5 `hooks/usePipeline.ts` (refactored — ~60 LOC)
Composer that wires daemon → processor. Returns the same 20-field surface that `MashupContext` consumes. No behavior changes visible to callers.

## 4. The refs question

The 8 ref-useEffect pairs fall into three groups, each handled differently:

### 4.1 Control signals (`stopRequestedRef`, `skipCurrentIdeaRef`) → AbortController
Daemon holds `runCtrl: AbortController | null` and `skipCtrl: AbortController | null` in `useState`. `startPipeline` instantiates `runCtrl`; per-idea iteration instantiates a fresh `skipCtrl`. `signal.aborted` is a **live read on the same object reference** — no closure staleness. `stopPipeline()` / `skipCurrentIdea()` use functional setState peek to reach the current controller:

```ts
const stopPipeline = () => setRunCtrl(c => { c?.abort(); return c; });
```

### 4.2 Daemon-owned config (`pipelineContinuous`, `pipelineInterval`, `pipelineTargetDays`, `pipelineDelay`) → functional setState peek
Long-running loop reads live values via the "peek" idiom:

```ts
let currentInterval = 120;
setPipelineInterval(v => { currentInterval = v; return v; });
// currentInterval now holds the latest committed value
```

No refs. The updater runs synchronously under React 18's batching model; `currentInterval` is assigned before the next line executes.

### 4.3 External props (`ideas`, `settings`, `images`, `savedImages`) → ONE consolidated snapshot ref
This is the documented pragmatic exception. These values come from parent (`MashupProvider`) and the daemon genuinely needs live reads inside long-running loops (e.g. `settings.scheduledPosts` for slot collision avoidance across a 10-minute run). There is no ref-free, closure-free way to read live prop values without restructuring the parent's contract or introducing `useSyncExternalStore`-backed stores — both out of scope for V030-001.

The compromise: **one** ref + **one** `useEffect` that mirrors all four props into a single snapshot object. Reduces 4 sync pairs to 1 (75% reduction). The remaining pair is isolated, named `latestPropsRef`, and limited to values the daemon cannot own.

### 4.4 Checkpoint snapshots (`checkpointImagesRef`, `checkpointSettingsRef`)
These were workarounds for carrying per-idea mutable state. The `lib/pipeline-processor.ts` deps-bag pattern makes them unnecessary:
- `checkpointImagesRef` → local variable inside `processIdea` closure, passed into `writeCheckpoint` via the deps bag.
- `checkpointSettingsRef` → snapshot taken at `startPipeline` time, captured in daemon's closure for the whole run.

Net result: **from 8 ref-useEffect pairs down to 1**, plus two AbortControllers held in useState.
