# V040-DES-001 — Pipeline Approval section with carousel-grouped approval

**Status:** spec
**Author:** Designer (visual/interaction only — no implementation)
**Depends on:** V030-DES-001 §A (approval-queue ViewType), V030-DES-003 (single-post approval refinement), DESIGN-003 (KebabMenu)
**Owns:** carousel card anatomy, per-image approve/reject interaction, pending/approved/rejected visual states, separation from Captioning tab
**Out of scope:** single-image approval card (owned by V030-DES-003), Focus-mode keyboard layer (owned by V030-DES-003), heatmap (owned by V030-DES-002), schema PRs

---

## 0 · Why this exists beyond V030-DES-003

V030-DES-003 designed the approval queue for single-image posts. The pipeline also emits **carousels** — N images sharing one caption, one schedule slot, one `carouselGroupId`, shipped as a single multi-image post. Under V030-DES-003 those N posts render as N loose cards, which:

1. Hides the grouping — the user sees 4 unrelated cards instead of one carousel they can judge as a whole
2. Makes "reject image #2 but keep the other 3" impossible without opening each card
3. Forces 4 clicks + 4 cognitive evaluations for a single creative unit

v0.4.0 needs carousels to render as **one card with internal image review**. This spec covers that card and nothing else — it plugs into the queue layout V030-DES-003 already defined.

Pipeline Approval must also **look and feel unambiguously distinct from Captioning Studio**. They are different workflows:

| | Captioning Studio | Pipeline Approval |
|---|---|---|
| Trigger | User picks images | Pipeline emits completed work |
| User intent | Craft / edit | Review / gate |
| Default action | Write caption | Ship or drop |
| Success metric | Better caption | Throughput |
| Time pressure | None | Schedule slot is ticking |

If a user can't tell the two apart at a glance, the design has failed — see §3.

Same Tailwind primitives, different silhouette. A user glancing at a cropped screenshot should know which view they're in within 300ms.

## 4 · Card types in the approval queue

The queue renders a **mixed feed** of two card shapes. Layout is still the responsive grid V030-DES-003 defined.

| Card kind | Source | Component |
|---|---|---|
| Single post | `ScheduledPost` with no `carouselGroupId` | `ApprovalCard` (V030-DES-003) |
| Carousel post | Set of `ScheduledPost`s sharing a `carouselGroupId` | `CarouselApprovalCard` ← **this spec** |

**Grouping rule** (deterministic, pure — mirrors `lib/carouselView.ts` logic):

```
groups = groupBy(posts.filter(p => p.status === 'pending_approval'), p => p.carouselGroupId)
for each groupId:
  if groupId is undefined or group.size === 1 → render as N single cards
  else → render as one CarouselApprovalCard
```

Sort order: newest-first by `max(post.createdAt)` inside each group, matching the Post Ready tab so the user sees the same ordering everywhere.

## 5 · CarouselApprovalCard anatomy

### 5.1 Condensed (default) layout

```
┌─────────────────────────────────────────────────────────────┐
│ ▢   ◈ Carousel · 4 images        [indigo pill: "Pending"]   │  ← header row
│                                                             │
│ ┌────────┬────────┬────────┬────────┐                       │
│ │ [img1] │ [img2] │ [img3] │ [img4] │    ← thumbnail strip  │
│ │   ●    │   ●    │   ●    │   ●    │    ← per-image dot    │
│ └────────┴────────┴────────┴────────┘                       │
│                                                             │
│ 💡  Spaghetti Westerns meets cyberpunk                      │  ← idea chip
│ "Neon-lit duel at high noon, six-shooters..."               │  ← caption 2-line clamp
│                                                             │
│ Fri Apr 18 · 18:00          [Instagram] [Pinterest]         │
│                                                             │
│ [ Review images ▸ ]  [ ✓ Approve carousel ]  [ ✕ Reject ▾ ] │  ← action row
└─────────────────────────────────────────────────────────────┘
```

**Header row** (always visible):
- Checkbox for bulk selection (aligns with single-card checkbox column)
- Stacked-squares icon (`Layers`) + "Carousel · N images" label
- Status pill — pending / partial / approved / rejected (see §6 for palette)

**Thumbnail strip:**
- Flex-row, equal-width, `grid-cols-{images.length}` up to 4; >4 images wraps or clips with `...+2` overflow (see §5.3)
- Each thumb: `aspect-square rounded-lg object-cover`
- Border: matches per-image state (§6)
- Per-image status dot at bottom-center of each thumb: 6px, indigo (pending) / emerald (approved) / red (rejected)
- Click a thumb → opens Review mode (§5.2)

**Caption + metadata rows:** identical to single-card for consistency (idea chip → caption → date + platforms).

**Action row** (bottom):
- `[ Review images ▸ ]` — tertiary, opens Review mode
- `[ ✓ Approve carousel ]` — primary, approves ALL non-rejected images in one sweep
- `[ ✕ Reject ▾ ]` — KebabMenu trigger; options: **Reject carousel** / **Reject with reason…** (reuses DESIGN-003 submenu)

### 5.2 Review (expanded) mode

Triggered by `[Review images ▸]` or clicking any thumbnail. Renders **inline, not modal** — expands the card into full-width with a vertical stack so the user sees each image at reviewable size without losing the list context.

```
┌─────────────────────────────────────────────────────────────┐
│ ◈ Carousel · 4 images          [— Collapse ▴]               │
│ "Neon-lit duel at high noon, six-shooters..."               │
│                                                             │
│ ┌─────────────────────────────────────────────────┐         │
│ │                    [ image 1 full ]              │  ← 16:9│
│ │                                                  │         │
│ │            [ ✓ Approve ]   [ ✕ Reject ]          │         │
│ └─────────────────────────────────────────────────┘         │
│                                                             │
│ ┌─────────────────────────────────────────────────┐         │
│ │                    [ image 2 full ]              │         │
│ │                                                  │         │
│ │          [ ✓ Approved (2s ago · undo) ]          │ ←acted │
│ └─────────────────────────────────────────────────┘         │
│                                                             │
│ ... continues for each image                                │
│                                                             │
│ ────────────────────────────────────────────────────────    │
│ [ ✓ Approve remaining (2) ]  [ ✕ Reject carousel ]  [Done]  │
└─────────────────────────────────────────────────────────────┘
```

**Inline over modal rationale:**
- Matches the Gallery lightbox pattern (lives inside the grid, not over it)
- Lets the user keep scrolling through other carousels while one is expanded
- Avoids focus-trap complexity when the user wants to jump to another carousel mid-review

**Per-image action zone:**
- `[ ✓ Approve ]` → sets that image's post to `scheduled`, button transforms to `[ ✓ Approved (Ns ago · undo) ]` in emerald/15 for 6 seconds, then fades to a static `Approved` chip
- `[ ✕ Reject ]` → destructive; 6-second undo toast at card bottom ("Image rejected — Undo"), then permanent
- Undo window reuses `UserSettings.approvalUndoSeconds` proposed in V030-DES-003 §15

**Bottom action bar** (sticky to card bottom while expanded):
- `[ ✓ Approve remaining (N) ]` — where N = count of images still `pending` (not yet acted on individually)
- `[ ✕ Reject carousel ]` — nukes all images at once (with confirmation if any were already approved)
- `[Done]` — collapses back to condensed view; does NOT commit or cancel anything

### 5.3 Overflow behavior for carousels > 4 images

Instagram carousels cap at 10 images; most pipeline carousels will be 2–4. For 5+:

- **Condensed view:** show first 3 thumbs + overflow tile `+N` on the 4th position. Overflow tile is `bg-zinc-800/80` with "+3 more" in `text-zinc-400 text-xs`. Clicking it opens Review mode (same as clicking any thumb).
- **Review view:** renders all images vertically. No pagination. If the list is long, the browser scrolls — that's fine, carousels are rarely >6.

## 6 · Visual states (per-image + carousel-level)

### 6.1 Per-image states (on thumbnails and in review)

| State | Thumbnail border | Dot | Opacity | Extra chrome |
|---|---|---|---|---|
| Pending | `border-indigo-500/30` | indigo-500 | 1.0 | — |
| Approved | `border-emerald-500/60` | emerald-500 | 1.0 | `CheckCircle2` badge top-right, `bg-emerald-500/10` overlay |
| Rejected | `border-red-500/40` | red-500 | 0.45 (grayscale via `saturate-50`) | `X` badge top-right, red diagonal stripe overlay at 15% opacity |
| Acting (optimistic) | `border-[#c5a062]` + `animate-pulse` | gold | 1.0 | — |

Same Tailwind utilities the calendar uses (`MainContent.tsx:455-460` palette), so the color language is already in the user's head.

### 6.2 Carousel-level aggregate states

Derived from per-image states (no new persisted field needed at the group level — the status pill is a pure computation):

| Aggregate | Condition | Status pill | Card border |
|---|---|---|---|
| Pending | all images pending | indigo: "Pending" | `border-indigo-500/25` |
| Partial | any image approved, any pending | gold: "Partial · 2 of 4" | `border-[#c5a062]/40` |
| Approved (queued) | all images approved (no rejections) | emerald: "Queued" | `border-emerald-500/40`, 600ms fade, then remove from queue |
| Approved w/ drops | some approved, rest rejected, none pending | emerald: "Queued · dropped N" | `border-emerald-500/40`, 600ms fade, then remove |
| Fully rejected | all images rejected | red: "Rejected" | `border-red-500/40`, 600ms fade, then remove |

**Card removal animation:** opacity 1 → 0 over 600ms + `translateY(-8px)` so the queue visibly shrinks after an action. Reduced-motion users get an instant snap.

### 6.3 Selected state (bulk operations)

When the carousel card's header checkbox is ticked:
- Border overrides state-border with `border-[#c5a062]/60`
- `shadow-[0_0_12px_rgba(197,160,98,0.18)]`
- Per-image states and dots remain visible — selection is orthogonal to approval

Bulk actions operate at the **carousel level**: "Approve selected" approves all pending images in selected carousels; "Reject selected" rejects all images in selected carousels.

## 7 · Interaction model — what happens on each click

Designer's note: these describe the **desired UX contract**, not the implementation path. Schema/logic PRs are flagged complex in §10.

### 7.1 Approve one image in a carousel
- Optimistic: image flips to `approved` state instantly
- Effect: that `ScheduledPost.status` → `'scheduled'`; the auto-poster will still publish it when the carousel ships, provided at least one sibling also ships
- Queue: card stays until all images are resolved

### 7.2 Reject one image in a carousel
- Optimistic: image flips to `rejected` state; 6-second undo toast appears
- Effect after undo window: that `ScheduledPost` is removed; `CarouselGroup.imageIds` loses that id
- **Degrade rule** (see §8): if remaining approved+pending < 2, the carousel falls back to a single-image post or gets fully dropped

### 7.3 Approve carousel (condensed action)
- Approves **all pending** images; already-rejected images stay rejected
- If any images were already rejected, the card transitions to "Queued · dropped N" (§6.2)
- Queue: card fades out and is removed after 600ms

### 7.4 Reject carousel (condensed action)
- If no images were previously approved → single confirmation toast "Reject all 4 images?" Yes/No
- If some were already approved → stronger confirmation: "Reject carousel and drop 2 already-approved images?" Yes/No
- On confirm: all sibling ScheduledPosts removed, `CarouselGroup` deleted
- 6-second undo toast ("Carousel rejected — Undo") identical to single-post reject

### 7.5 Reject with reason (submenu)
Reuses `RejectWithReasonMenu` from V030-DES-003 §9. Reasons inherit the list:
- Off-brand
- Wrong platform
- Bad caption
- Image quality
- Other (free-text, 80-char max)
- Reject without reason

Reason attaches to all rejected posts (bulk-write).

## 8 · Degrade rules (when a carousel can no longer ship)

Instagram/Pinterest carousels require ≥2 images. If rejections drop the carousel below threshold:

| Remaining (approved + pending) | Behavior |
|---|---|
| 2+ | Carousel ships normally with remaining images |
| 1 | **Degrade to single:** the one surviving post keeps its schedule slot, `carouselGroupId` cleared, `CarouselGroup` deleted. Card shows a small notice: "Degrading to single-image post" for 2s before transition. |
| 0 | Carousel fully rejected. Same path as §7.4 with undo. |

**Why not auto-regenerate the missing image?** Out of scope for v0.4.0 approval UX. The design only needs to not break; regen is a future pipeline feature.

The degrade notice matters — without it, the user who rejected image #3 has no idea the carousel is now a single post. This preserves trust in the system.

## 11 · Edge cases

- **Race: pipeline finishes a second carousel mid-review.** Append to queue bottom with a 400ms slide-in; don't disturb the user's current expanded card.
- **Same carousel expanded when last image resolved.** Smoothly collapse the expanded panel into the "Queued" state, then 600ms fade-out. Prevents the card from vanishing from under the user's cursor.
- **User rejects every image individually, one by one.** Each rejection shows its own 6s undo toast; on the 4th rejection the card transitions to "Fully rejected" state. Don't show a carousel-level confirmation — the user has been making explicit choices.
- **Image load fails.** Thumbnail shows `ImageOff` icon (already used at `ApprovalQueue.tsx:364`) with `bg-zinc-800`. Approve/reject buttons remain functional — the user can still make a decision from the caption alone.
- **Carousel has exactly 2 images and user rejects one.** Degrade notice fires, card transitions into a single `ApprovalCard` in place (keeps same DOM slot if feasible, otherwise animate swap). The surviving ScheduledPost keeps its slot.
- **Expanded while another user action marks it approved** (ex. auto-approve worker): collapse gracefully and fade out; don't orphan the expanded panel.
- **Keyboard nav into a carousel card** (Focus mode from V030-DES-003): → expands; ←/↓/↑ navigate between images inside; `A` approves current image, `R` rejects; `Esc` collapses back. Reuses V030-DES-003's key table; no new bindings.

## 13 · Open questions (for Hermes / product)

1. **Should approval history be visible?** Once a carousel ships, do we show "4 images shipped, 1 dropped" in the Post Ready view? Current spec assumes yes (small "dropped 1 via approval" badge on the carousel card in Post Ready) but it's not designed here.
2. **Approve-without-review in bulk.** Is there a "trust this idea, auto-approve all its carousels" shortcut for proven topics? Would plug into the Focus-mode keyboard layer. Not in v0.4.0 scope, but the hook point is `lib/approval-grouping.ts`'s groupBy — easy to add an "auto-approve for idea id X" filter.
3. **Per-image rejection reason.** The current spec applies one reason to all rejected images in a carousel reject. If users want per-image reasons, the submenu needs a different placement (per-image kebab instead of carousel-level kebab). Flag for user research.
