# SECTION A — Approval Queue surface

## A.1 Goals

Today, `pending_approval` posts only show up via `bulkApproveScheduledPosts` / `bulkRejectScheduledPosts` — there's no dedicated review surface. v0.3.0 makes the pipeline more autonomous, which means the approval queue becomes the user's primary interaction with their content. It must scale from 3 posts to 200.

Three goals:
1. **One place** to find every post awaiting human approval, regardless of which idea spawned it
2. **Fast** for confident batches — keyboard-driven, no extra modal taps
3. **Editable in place** — approving usually means "this is fine but tweak the caption"

## A.2 Surface placement

Add a new top-level view: `'approval'` joins the existing `ViewType` (`types/mashup.ts:214`).

- Sidebar entry: `Approval Queue` with a `<ListChecks />` icon and a numeric badge showing `pending_approval` count
- Badge color: `bg-indigo-500 text-white` (matches the existing pending status palette)
- Badge animates in/out — appears when count > 0, disappears when empty (idle = no badge clutter)
- Pulse the badge gently when count increases (new pipeline-generated posts arrived)

## A.3 Two review modes

User toggles at the top of the queue. Default = grid for >10, focus for ≤10.

### A.3.1 Focus mode (linear stack)

One post fills most of the viewport. Big preview, side-by-side caption editor.

│ │       image         │ │ Space Marine in cinematic… │ │

│ │                     │ │ HASHTAGS (12/30)            │ │

│ │                     │ │                             │ │

│                         │ ◉ Instagram  ◉ Pinterest    │ │
│ Pipeline confidence:    │                             │ │
│ ★★★☆☆  3 of 5 checks    │ SCHEDULED                   │ │
│                         │ Sat Apr 20 · 19:00          │ │
│ Source idea:            │ [edit time] [find better]   │ │

│                                                          │

**Keyboard:**
- `←` reject, `→` approve, `↑/↓` move between posts in stack
- `E` enter edit mode (focus moves into caption textarea)
- `Esc` exit edit mode back to navigation
- `⌘Enter` (or `Ctrl+Enter`) approve and move to next
- `⌘K` opens shortcut cheat-sheet overlay

**Animations:**
- Approve → card slides right, emerald flash on action button (180ms)
- Reject → card slides left, red flash (180ms)
- Next card fades in (120ms, prefers-reduced-motion: opacity only)

## A.4 Per-post traffic-light (V030-UX item 1d)

A small inline strip on each card (both modes) showing 5 checks. Each check is one dot.

| Check | Pass condition | Visual |
|---|---|---|
| Caption length | ≤ platform's max char count | dot color |
| Hashtag count | between 8 and 30 | dot color |
| Schedule conflict | no other same-platform post within 30min | dot color |
| Image quality | image not flagged as failed/error | dot color |
| Brand-safety | optional — caption passes the captioning model's safety filter | dot color |

Visual: 5 small `w-1.5 h-1.5 rounded-full` dots in a row.
- Pass = `bg-emerald-500`
- Fail = `bg-red-500`
- Skipped/N-A = `bg-zinc-700`

Hover the strip → tooltip listing each check with text + reason if failed.

**Confidence stars** above the strip (5-star rating): `count(passing) / 5` rounded down. Drives the visual sort signal — user can scan grid mode and process the easy 5-star ones first.

## A.5 Reject-with-reason

Reject button opens a small inline dropdown (NOT a modal — modals kill flow):

```
✕ Reject because…
  · Off-brand
  · Wrong platform fit
  · Bad caption
  · Image quality
  · Other (free text)
```

Reasons feed into a `rejectScheduledPost(postId, reason?: string)` extension. If the captioning loop ever wants the signal it has it; otherwise it is durable telemetry. Skip the reason entirely with `Esc` or by clicking outside.

## A.6 Empty state

When zero `pending_approval` posts:

```
        [LayoutGrid icon, gold/40, 64px]
       
            All caught up.
   
   The pipeline will queue posts here for review.
   
   [⚙ Pipeline settings]    [▶ Run pipeline now]
```

Centered, gold icon, calls to action go to the Pipeline tab (existing) — turns the empty state into a productive moment instead of dead space.

# SECTION B — Week-Ahead Progress meter

## B.1 Goals

`pipelineTargetDays` already exists (`types/mashup.ts:602`) — the daemon tries to keep the schedule filled N days ahead. Today the user has no visualization of how full each week actually is. v0.3.0 leans on weekly fill strategy: the meter is the receipt.

## B.2 Placement

Inside the existing Post Ready calendar week view (`MainContent.tsx` around L3504+ for the week grid), a sticky header strip above the week grid.

## B.3 Visual

```
┌──────────────────────────────────────────────────────────────────────┐
│ ← Apr 20 — Apr 26       Week of Apr 20      Apr 27 — May 3 →         │
│                                                                      │
│ Filled  ████████████████████░░░░░░░░░  12 of 21 slots  · 9 gaps     │
│         ↑                                                            │
│         gold fill                                                    │
│                                                                      │
│ [+ Auto-fill gaps]    [Settings: 3 posts/day per platform ⚙]         │
└──────────────────────────────────────────────────────────────────────┘
```

- Bar bg: `bg-zinc-800/60`
- Bar fill: gradient `from-[#c5a062] to-[#c5a062]/70` — gold says "this is the goal"
- "21 slots" comes from `pipelineDailyCaps` × 7 days (or a sane default of 3/day if no caps configured)
- Gap count: `max(0, target - currentCount)`
- Below 50% fill: bar fill stays gold; above 80%: bar end gets a subtle pulse (the goal is in sight)
- Above 100%: bar caps at 100% width, append " · over target" in `text-amber-400` text — over-scheduling is not failure but it is worth flagging

## B.4 Per-day mini meter (under each day cell)

Inside each calendar day cell, a thin progress bar (`h-1`) at the bottom edge.

| State | Style |
|---|---|
| 0–60% of cap | `bg-emerald-500/70` (plenty of room) |
| 60–90% | `bg-amber-500/70` (filling up) |
| 90–100% | `bg-orange-500/70` (close to cap) |
| at cap | `bg-zinc-600` (locked) — also dim the cell to `opacity-70` |

Cells are no longer drop targets when at cap. Drag-and-drop hover state turns red border to communicate the rejection.

## B.5 Empty-slot affordances

Today calendar cells with no posts just appear blank. Replace with a dashed outline + inline action:

```
┌────────────────────┐
│ Sat 20             │
│  ┌──────────────┐  │
│  │   + Fill     │  │  ← dashed border-[#c5a062]/30 with ghost text
│  │              │  │
│  └──────────────┘  │
│  ████░░░░ 1/3      │
└────────────────────┘
```

- Empty slots only render in the **current** week (or `pipelineTargetDays` ahead, whichever is shorter) — beyond that, blank cells stay blank to avoid visual noise
- Click `+ Fill` → opens a "Generate post for this slot" mini dialog: pick an idea (existing) or auto-generate, scheduled time auto-set to the slot's optimal hour from §C
- This wires into the existing pipeline machinery — the dialog just pre-fills the slot's date/time and triggers a single-idea pipeline run

## B.6 "Auto-fill gaps" button

Top-right of the week header. Triggers a focused pipeline run targeting only the empty slots in the current week (NOT the full daemon loop). Shows a progress toast `Filling 9 gaps in week of Apr 20…` while running.

Behavior:
- Honors `pipelineDailyCaps`
- Honors §C engagement scores when picking which slot to fill first (best slots get filled first, not chronologically)
- Posts land in `pending_approval` if `pipelineAutoPost` is off — flowing back into Section A's queue

## B.7 Loading + error states

- Loading: progress bar + per-day bars render at zero with a slow shimmer
- Error (e.g. pipeline run failed mid auto-fill): toast in red with retry button; per-day bars don't lie about partial progress

# SECTION C — Engagement heatmap on week view

## C.1 Goals

`smartScheduler.ts` already produces `EngagementHour` (0-1 weight per hour) and `EngagementDay` (multiplier per weekday). The data exists; it's invisible in the UI. v0.3.0 surfaces it directly inside the week grid so users can see WHEN to post without reading numbers.

## C.2 Placement + toggle

Toggle button in the week-view header: `[🌡 Heatmap]`. Off by default for new users (avoids overwhelming first impression); persists per user via `userSettings`.

When ON:
- Each hour cell in the week grid gets a background tint based on its slot score
- A tiny legend appears bottom-right of the calendar: gradient bar from low to high

## C.3 Cell tinting

Score formula reuses `scoreSlot` (smartScheduler.ts:180) — combined `dayMult × hourWeight + weekendBonus`. Range roughly 0.0–1.1.

Tint via background opacity on `#c5a062` (gold) — staying on-brand:

| Score | Tailwind |
|---|---|
| < 0.3 | `bg-[#c5a062]/0` (no tint) |
| 0.3–0.5 | `bg-[#c5a062]/5` |
| 0.5–0.7 | `bg-[#c5a062]/10` |
| 0.7–0.85 | `bg-[#c5a062]/18` |
| ≥ 0.85 | `bg-[#c5a062]/28` |

Cells with already-scheduled posts in them keep their existing pill chrome on top — heatmap is decoration, not an overlay that fights for attention.

**Why gold tinting (not green/red):** the existing palette already uses emerald + amber + red for *post status*. Reusing them for *time quality* would be ambiguous. Gold is reserved for "where to look" — perfect for "the best times to post."

## C.4 Top-3 slot star markers

When heatmap is on, also surface the 3 highest-scoring **available** (not already-scheduled, not over-cap) slots as a small ★ in the top-right of those cells:

- Color: `text-[#c5a062]`
- Size: `w-3 h-3`
- Tooltip on hover: see §C.5

Three stars max — more than that is noise.

## C.5 Hover tooltip per cell

```
┌──────────────────────────────┐
│ Sat 20 · 20:00               │
│                              │
│ Engagement score: 0.92       │
│ ★★★★☆  4 of 5                │
│                              │
│ • Day weight: 1.0× (Saturday)│
│ • Hour weight: 0.95          │
│ • Source: Instagram Insights │
│   (37 past posts, updated 4h │
│   ago)                       │
│                              │
│ [Schedule a post here →]     │
└──────────────────────────────┘
```

- Tooltip uses the same panel chrome as DESIGN-003 KebabMenu (`bg-zinc-900/95 border-[#c5a062]/30`) — visual coherence
- Always tells the user **why** (day, hour, source, sample size) — not just a green dot. Trust = legibility (V030-UX §3 core principle)
- "Source" line is critical: differentiates `instagram` (high confidence, real data) from `default` (research baseline, low confidence)
- Star count uses confidence (sample-size-adjusted), not raw score — see §C.6

## C.6 Confidence indicator

Sample size matters. A score of 0.9 backed by 50 past posts is very different from 0.9 backed by 3.

Star rating (`★★★★☆`) modifies raw score by sample-size confidence:

```
displayedStars = round(rawScore × 5 × confidenceMultiplier)

confidenceMultiplier =
  source == 'default'        → 0.6   (max 3 stars — research baseline only)
  source == 'instagram' && samples < 10 → 0.7  (max ~3.5 stars — early signal)
  source == 'instagram' && samples < 30 → 0.85
  source == 'instagram' && samples >= 30 → 1.0  (full confidence)
```

This requires `loadEngagementData` to also return the sample count (currently it only returns `source`). Small extension — flagged as a Developer task in §D.2.

## C.7 Inline hint on the existing scheduling form

Two inputs at `MainContent.tsx:3618-3632` already exist (Date + Time pickers in the Post Ready card scheduling block). When user picks a time:

- Compute the slot's score with `findBestSlots` for that single date
- Show a one-line hint below the time picker:

```
✨ 18:00 is a top slot for Instagram (0.85 score)
⚠ 03:00 is below average — try 19:00 or 20:00 instead
```

Color: emerald for ≥0.7, amber for 0.4–0.7, zinc-500 for < 0.4 (don't shame the user, just inform).

## C.8 Off state

When heatmap toggle is OFF:
- All tinting disappears
- Star markers disappear
- Tooltips on cells revert to default browser title (or are removed entirely — they shouldn't render heavy tooltips when no heatmap data is being communicated)
- Inline hint on scheduling form (§C.7) **stays on** — that's a focused, opt-in interaction not a passive overlay

All animations respect `prefers-reduced-motion: reduce` (opacity only).
