### 2.2 `<HeatmapTint>`

Single absolutely-positioned `<div>` inside each slot cell. Pure visual, pointer-events: none.

```tsx
interface HeatmapTintProps {
  score: number;        // 0..1.1 from scoreSlot()
  enabled: boolean;     // global toggle state
}

function HeatmapTint({ score, enabled }: HeatmapTintProps) {
  if (!enabled || score < 0.3) return null;
  const tint = classifyTint(score);  // returns a Tailwind class
  return (
    <div
      aria-hidden="true"
      className={`absolute inset-0 pointer-events-none transition-opacity duration-200 ${tint}`}
    />
  );
}
```

Tailwind tint table (`classifyTint`):

| Raw score from `scoreSlot()` | Tailwind class | Rendered effect |
|---|---|---|
| `< 0.30` | `null` (no element rendered) | base cell only |
| `0.30 – 0.49` | `bg-[#c5a062]/5` | barely-there warmth |
| `0.50 – 0.69` | `bg-[#c5a062]/10` | visible warm haze |
| `0.70 – 0.84` | `bg-[#c5a062]/18` | clear hot zone |
| `≥ 0.85` | `bg-[#c5a062]/28` | unmistakable peak |

**Why these exact breakpoints:** `smartScheduler.ts` `scoreSlot` formula is `dayMult × hourWeight + weekendBonus(0 or 0.15)`. With default weights, most hours land 0.2–0.85; the ≥0.85 tier fires on weekend evenings specifically — exactly the "prime time" the scheduler is already calling out.

**Overnight cells (hours 0–5):** smartScheduler skips `hour < 6` entirely (L281). These cells get `score = 0` and therefore no tint — correct behavior, but worth a unit test to confirm `<HeatmapTint score={0} />` renders nothing.

### 2.3 `<TopSlotStar>`

Only the 3 highest-scoring **available** slots (not taken, not over-cap) render a star.

```tsx
interface TopSlotStarProps {
  rank: 1 | 2 | 3 | null;   // null = not in top 3
}
```

Visual:
- `<Star className="absolute top-1 right-1 w-3 h-3 text-[#c5a062] fill-[#c5a062]/30 z-10" aria-label={`Top slot #${rank}`} />`
- Rank 1 gets a subtle glow: add `drop-shadow-[0_0_4px_rgba(197,160,98,0.5)]`
- Ranks 2 and 3 are plain gold, no glow
- All three pulse gently in sync (2s ease-in-out infinite, 0.6→1.0 opacity) so the eye finds them — respects `prefers-reduced-motion: reduce` by falling back to static full opacity

Top-3 is computed by:
```
findBestSlots(existingPosts, 3, engagement, { platforms, caps })
  → SlotScore[]
  → map date+time to cellKey = `${date}:${hour}`
```

The computation happens once per week-view render (memoize on `scheduled`, `engagement`, and the week's date range).

`sourceLabel` / `sourceColor` mapping:

| `engagement.source` | Label | Color |
|---|---|---|
| `'instagram'` + `samples >= 30` | `Instagram Insights (high confidence)` | `text-emerald-400` |
| `'instagram'` + `samples >= 10` | `Instagram Insights (moderate)` | `text-emerald-400/80` |
| `'instagram'` + `samples < 10` | `Instagram Insights (early signal)` | `text-amber-400` |
| `'default'` | `Research baseline (no IG data yet)` | `text-zinc-500` |

`updatedAgo` — human-formatted delta from `engagement.fetchedAt` (e.g. "4h ago", "yesterday", "6 days ago"). Use a small local helper; don't pull in date-fns for this.

**The CTA button** is the one interactive element inside the tooltip. It shouldn't suppress the tooltip's own `pointer-events: none` — so the button itself opts back in with `pointer-events-auto`. Clicking triggers the same flow as clicking an empty cell (`setCalendarSlotClick`).

### 2.7 Confidence-star formula (authoritative)

```ts
function computeDisplayedStars(rawScore: number, source: 'instagram' | 'default', samples: number): number {
  const confidence =
    source === 'default' ? 0.6 :
    samples >= 30 ? 1.0 :
    samples >= 10 ? 0.85 :
    0.7;

  const stars = Math.round(rawScore * 5 * confidence);
  return Math.max(0, Math.min(5, stars));
}
```

Examples with default hour/day weights (source='default'):
- Saturday 20:00 — raw ≈ 0.95 × 1.0 + 0.15 = 1.10, stars = round(1.10 × 5 × 0.6) = **3**
- Tuesday 12:00 — raw ≈ 0.75 × 0.75 = 0.56, stars = round(0.56 × 5 × 0.6) = **2**
- Monday 03:00 — raw ≈ 0, stars = **0**

With `source='instagram', samples=50` the same Saturday 20:00 becomes **6 → capped to 5**. That's correct: real data + peak hour = max confidence.

## 3. State + data flow

### 3.1 State added to MainContent.tsx

```ts
const [heatmapEnabled, setHeatmapEnabled] = useState<boolean>(
  () => settings.heatmapEnabled ?? false
);
const engagement = useMemo(() => loadEngagementData(), []);
const weekScores = useMemo(
  () => computeWeekScores(days, engagement),
  [days, engagement]
);
const topSlots = useMemo(
  () => findBestSlots(existingPosts, 3, engagement, { platforms: allActivePlatforms, caps: settings.pipelineDailyCaps }),
  [existingPosts, engagement, allActivePlatforms, settings.pipelineDailyCaps]
);
```

`computeWeekScores` — one-pass builder mapping `${dateStr}:${hour}` → `{ score, dayMult, hourWeight, weekendBonus }` for the 7 × 24 grid. Pure function, lives alongside `scoreSlot` in `smartScheduler.ts`. Developer work; trivial.

### 3.2 Cell render diff

Drop-in replacement for L3096-3103:

```tsx
className={`relative border-l border-zinc-800/60 min-h-[40px] p-1 space-y-1 transition-colors ${
  isDragOver
    ? 'ring-2 ring-emerald-500/50 bg-emerald-500/5'
    : isEmpty
      ? heatmapEnabled
        ? 'cursor-pointer hover:ring-1 hover:ring-[#00e6ff]/40'         // blue hover when heatmap is on
        : 'cursor-pointer hover:bg-emerald-500/5'                        // emerald hover (legacy)
      : ''
}`}
```

Plus **inside** the cell, before the posts map, insert:
```tsx
<HeatmapTint score={weekScores.get(cellKey)?.score ?? 0} enabled={heatmapEnabled} />
{heatmapEnabled && topSlots.some(s => `${s.date}:${parseInt(s.time)}` === cellKey) && (
  <TopSlotStar rank={rankOf(cellKey, topSlots)} />
)}
```

The key change is `hover:bg-emerald-500/5` → `hover:ring-1 hover:ring-[#00e6ff]/40` **only when heatmap is on**. Hover switches from a color wash (which would compound with the gold tint and look muddy) to a clean blue outline (readable over any tint).

### 3.3 Tooltip lifecycle

- `onMouseEnter` on cell → `setTimeout(..., 120)` to show tooltip (debounces flicker)
- `onMouseLeave` → clear timeout + hide tooltip immediately
- Scroll anywhere → hide tooltip immediately (prevents detachment)
- Escape key → hide tooltip
- Only one tooltip exists at a time (portal singleton, anchored by cell ref)

## 4. Color mapping — final

One table to rule all cell states:

| Cell state | Background | Border / Ring | Why |
|---|---|---|---|
| Idle, heatmap OFF | none | `border-l border-zinc-800/60` | Today's baseline, unchanged |
| Idle, heatmap ON, score 0–0.29 | none | `border-l border-zinc-800/60` | Overnight + low-engagement hours stay quiet |
| Idle, heatmap ON, score 0.30–0.49 | `bg-[#c5a062]/5` | unchanged | Warm hint |
| Idle, heatmap ON, score 0.50–0.69 | `bg-[#c5a062]/10` | unchanged | Mid warm |
| Idle, heatmap ON, score 0.70–0.84 | `bg-[#c5a062]/18` | unchanged | Hot |
| Idle, heatmap ON, score ≥ 0.85 | `bg-[#c5a062]/28` | unchanged | Peak |
| Empty + hovered (heatmap OFF) | `bg-emerald-500/5` | unchanged | Legacy |
| Empty + hovered (heatmap ON) | heatmap tint persists | `ring-1 ring-[#00e6ff]/40` | Blue = active interaction |
| Dragover (always wins) | `bg-emerald-500/5` | `ring-2 ring-emerald-500/50` | Drop target — overrides heatmap |
| Has posts | whatever the pills render | unchanged | Heatmap tint still renders underneath (pills sit at z ≥ 10) |

The drag-over state intentionally wins over everything else, even when heatmap is on — drop-target clarity is critical for drag UX.

## 5. The hover-tint collision (resolved)

**Problem:** existing cell uses `hover:bg-emerald-500/5`. Stacking that on top of `bg-[#c5a062]/18` (gold heatmap) produces a muddy brown-green tint. Bad.

**Fix (already encoded in §3.2):** when heatmap is ON, swap hover from a background tint to a border ring (`ring-1 ring-[#00e6ff]/40`). The ring sits on top of any background tint without muddying it and brands hover as "you are interacting" (blue) distinct from the static "this slot is prime" (gold).

When heatmap is OFF, legacy behavior is preserved — emerald hover stays.

## 6. Accessibility

- Toggle button has `aria-pressed={heatmapEnabled}`
- Heatmap tint `<div>`s are `aria-hidden="true"` — they carry no information for screen readers (the legend + tooltip do)
- Star markers have `aria-label={`Top slot #${rank}`}`
- Tooltip has `role="tooltip"` and is associated with its anchor cell via `aria-describedby={tooltipId}` when open
- Legend has `aria-label="Engagement heatmap legend"`
- `prefers-reduced-motion: reduce`:
  - star pulse animation disabled
  - tooltip fade-in replaced with instant show
  - tint transitions (200ms) still run — they're subtle enough

## 9. Open questions

1. **Hour granularity** — smartScheduler scores whole hours. Fine for the overlay. But if the scheduler ever moves to 30-min granularity, the cell grid doesn't support it. Flag, not block.
2. **Multi-week view** — if v0.3.0 adds a 2-week view later, the heatmap scales trivially; no design change needed. Just flagging for Developer.
3. **Timezone** — `smartScheduler.ts` uses local time implicitly. Users traveling / posting to a different audience TZ will see a heatmap that doesn't match their audience. Out of scope for this spec; worth a follow-up.
4. **Heatmap for the month view** — month cells are too small for hourly tinting. Proposal: show per-day tinting (max score of any hour that day) when heatmap is on in month mode. Flagged as a follow-up, not part of this task.
5. **Mobile** — week grid is horizontally scrollable on mobile (`overflow-x-auto` at L3026). Tooltip should anchor to the cell and follow scroll, or close on scroll. Current spec says close on scroll. Confirm that's desired for mobile, or do we want it to follow?
