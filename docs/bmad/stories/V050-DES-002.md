### Wiring

`components/MashupStudio.tsx` — added `useOnboardingState` hook that reads three
localStorage flags and resolves to one of `loading | show-wizard | show-pill | hidden`:

- `mashup.onboarded` → `'1'` once Step 3 completes (or success state's "Review" click)
- `mashup.onboardingDismissedAt` → user explicitly hid the pill forever
- `mashup.onboardingSkippedAt` + `mashup.onboardingProgress` → resume context

The wizard mounts inside `MashupApp` after the auth/load gate so it never
renders behind the loading screen. The pill mounts at fixed `top-4 right-4`.

### Two intentional deviations from spec

1. **Test connection → Save credentials.** The spec described a
   `POST /api/{platform}/test` endpoint that doesn't exist (verified
   `app/api/`). Rather than ship a fake spinner, the button reads
   "Save credentials" and writes straight to `settings.apiKeys` after a
   presence-check. The PROP for real test endpoints is unchanged.
2. **Step 3 progress is simulated.** Real `runPipelineOnce()` invocation
   was flagged complex in the spec (touches pipeline orchestration). The
   Step 3 stub (`runSimulation` in `Step3Pipeline.tsx`) seeds the user's
   chosen idea via `addIdea` and walks through 4 progress lines with a
   3-second total stagger. The actual pipeline picks up the queued idea
   on next tick, exactly as if the user had typed it manually. The
   visual progression is honest: ideaSelected → imageGenerated →
   captioned → scheduled all flip `done` independent of real status,
   because the wizard isn't subscribed to the pipeline progress channel.

The wizard ships with these exactly because the spec said it should —
"build the routine UI; PROP the orchestration wiring." Both are flagged
in code comments at the deviation site so the next dev knows where to
swap in the real call.
